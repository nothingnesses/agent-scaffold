## [Unreleased]

### Added

- A `type:"waiver"` instrumentation record and a W5 `validate --workflow` check (`src/metrics.rs`, `src/workflow.rs`, `pack/instrument.md`), one unified concept for an authorised exemption from the convergence bar. A waiver carries a `unit` (`step` or `increment`), the `step` slug (plus an `increment` token present exactly for an increment-unit waiver), a `reason` (`predates-logging`, `review-skipped`, or `accepted-at-escalation`), an `evidence_tier` (`self-declared` or `record-backed`), and an `evidence` pointer present exactly for the record-backed tier. W3 becomes convergence-OR-waiver: a `complete` step with no round records is exempt only when a step-unit waiver covers it, and an increment whose peak `consecutive_clean` is short of its risk class's required streak is exempt only when an increment-unit waiver names it, while a step with neither converging rounds nor a covering waiver still fails and a `risk_class`-inconsistency is never waived. The new W5 check keeps waivers honest, orthogonally to W3: every waiver names a real Roadmap step, an increment-unit waiver's `step` owns its `increment`, every record-backed waiver's `evidence` joins to a real `type:"escalation"` record with `human_decision:"decision"` that is scoped to the waived unit (so a self-declaration cannot cite an unrelated escalation to earn the strong tier), and the `reason` <-> `evidence_tier` pairing is enforced so a self-declaration cannot claim the strong tier. Best-effort `parse_waivers` and `parse_escalations` projections feed the checks, and a malformed waiver is reported by the strict schema check and dropped by the projection, so it can never silently grant an exemption. This is the second Q-44 structured-data pilot.
- A `type:"decision"` instrumentation record (the decision receipt), a `type:"baseline"` record declaring the historical-exemption cutoff, and a W4 `validate --workflow` check (`src/metrics.rs`, `src/workflow.rs`, `pack/instrument.md`). The decision record captures a human decision as `q_id`, the presented `options` (a non-empty string array), the `recommendation`, and the human's `chosen` option, with the validator enforcing that `chosen` is one of the presented `options` so the record is auditable evidence the options were shown rather than a self-certified flag. The baseline record declares a `questions_through` cutoff (a `Q-<n>` id) that names the last Open-Questions item predating the mechanism. Best-effort `parse_decisions` and `parse_baseline` projections feed W4, which requires a matching receipt for every `decided -> folded into <slug>` Open-Questions item whose id index is strictly after the declared cutoff, so the historical decided items at or below the cutoff are exempt while later ones are checked. The cutoff is an independent declared value, not derived from the receipt set, so a forgotten receipt cannot move its own exemption boundary; when no baseline is declared, every decided item requires a receipt.
- `validate --workflow` cross-references a plan's Roadmap against the round log (`src/workflow.rs`): every Roadmap step marked `complete` must have converging round records, so a step marked done without its review loop, or one that never reached the clean-round streak its risk class requires, is reported. It requires `--plan` and reuses the same metrics log as the rest of `validate`.
- Two Roadmap statuses, `trivial` and `grandfathered`, terminal complete-like statuses that declared a step's review history so the workflow cross-reference did not flag it. RETIRED in the same unreleased cycle: their review-exemption is now carried by the unified `type:"waiver"` record above, so a step that predates round-logging or whose review was skipped is marked `complete` with a covering waiver rather than given a special-case status. `skipped` stays a distinct status.
- An optional `harness` field on each `reviewers[]` entry in the instrumentation `round` record, naming the CLI a reviewer ran on (for example `claude-code`, `codex`, or `gemini-cli`), so per-reviewer calibration can distinguish harnesses, not just models. It is optional and validated as a string only when present, so existing round records stay valid.
- Optional module `isolation` (`--module isolation`): a guidance-only module that wires a project to the external agent-box and agent-images projects for container/worktree writer-agent isolation. It renders setup pointers into `AGENTS.md`'s `{{modules}}` block (install an agent-images image, create the global `~/.agent-box.toml`, spawn writers via `ab new`/`ab spawn --git`) as the concrete form of the container tier the Writer isolation rule names. It drops no files into the project tree, takes no runtime dependency on those projects, and reimplements no isolation.
- Optional modules: a `--module <name>` selection scaffolds an opt-in bundle on top of the core pack. A `{{modules}}` render slot concatenates each enabled module's guidance into `AGENTS.md` (empty and byte-identical when no module is selected), and a module may declare `requires` to auto-enable its dependencies. Asset manifests gain a `module` tag and an `executable` field (drop a file with its executable bit set). The first module is `checks` (`--module checks`): a deterministic verification gate that scaffolds a user-owned `.agents/checks.toml` (flat `[[check]]` entries with a `command`, a `kind` of `lint`/`format`/`test`/`mutation`, an optional non-mutating `check` command, and optional `paths` globs), a seeded `ast-grep` config plus example rule, a read-only `checks-reviewer` role prompt, and a pre-commit hook. The `agent-scaffold checks [--staged]` subcommand reads `checks.toml` and runs the lint and format checks inside a throwaway git worktree, so an in-place formatter can never mutate the live tree (`--staged` isolates the index content for hook use). `scaffold --module checks --with-precommit-hook` installs a git pre-commit hook create-if-absent (only when the resolved hooks dir is inside the scaffolded project, never clobbering an existing hook).

### Changed

- Hardened the scaffolded workflow guidance (`AGENTS.md` and the role prompts) after a design review. The review ledger and orchestrator round state move to a file committed beside its plan, so the workflow survives the orchestrator losing context and resumes across machines and sessions; the orchestrator follows an explicit per-round procedure to append the ledger and increment or reset the clean-round count. Convergence now requires consecutive clean rounds scaled to the stakes, and a review loop escalates to a human when the total rounds on an artifact reach the total-round cap (default five). A dismissed high-severity finding is re-checked by a second independent triager. The acceptance review gains a triager, matching the other review phases. A human interrupt now goes through a bounded intake assessment that advises on impact and gives feedback on the request, routing trivial requests directly and non-trivial ones through the plan.
- Split the review and triage steps into separate nodes in the README workflow diagram, so the triager owns the convergence decision.
- Generalised the reviewer and explorer diversity guidance in `AGENTS.md` from "different models where available" to "different models or harnesses where available", so a user running more than one CLI is guided to spread reviewers across harnesses too, and extended the review-request prompt's depth cue to match.

## [0.0.1] - 2026-07-10

### Added

- Initial release. Scaffolds a repeatable agent workflow into a project: a canonical, harness-agnostic `AGENTS.md`, a planning-document template under `docs/plans/`, reusable per-role prompts, and a selectable set of engineering principles.
- Principle selection. `--principles` accepts `default`, `all`, `none`, a comma-separated list of ids, and `tag:<t>` tokens; `--principle-detail` (`name`, `summary`, or `full`) sets the rendered detail, and `--list-principles` prints the selection.
- Interactive two-pane selector (ratatui), the default on a terminal: toggle inclusion, reorder, filter by name/id/tag, undo and redo, and a save-confirmation modal.
- Write safety. A run previews a plan and writes only with `--write` or the selector's Save confirmation; `--dry-run` previews explicitly. Two-tier ownership refreshes tool-owned reference assets under `.agents/` and creates user working files only if absent, unless `--force`.
- Bring-your-own packs. `--template <dir>` scaffolds from a user-supplied pack via a `pack.toml` manifest (`[[asset]]` and `[[var]]` entries), with `--var key=value` substitution and pack-owned principles.
- Version control. Initialises a git repository by default (`--vcs git|none`), skipped when the target is already inside a repository.
- Runs without Nix as a standalone binary (Rust 1.88 or newer).
