# Q-59 design notes: where the session-lifecycle state lives (the data-model lens)

This is one of several parallel explorer documents on Q-59. It covers ONLY the data-model axis: where and how the session-lifecycle state (`active` / `checkpointed`, plus the checkpoint kind and commit) is represented. The command-surface axis (a `checkpoint` / `resume` subcommand pair versus validation-only reads) and the enforcement axis (what `validate --workflow` should assert about the transition sequence) are the subjects of the sibling documents; this one touches them only where the data model constrains them. The recommendation here should be read together with those.

## 1. The question

Today nothing records that a session is checkpointed. `next`'s `LoopState` (`src/next.rs:169`) models only the review loop (`ReadyToPlan` through `Done`); it has no `Active` / `Checkpointed` / `Paused` state. `.agents/user-prompts/pause.md`, `compaction-prep.md`, and `resume.md` are prose checklists that point at the "Checkpoint and resuming after context loss" procedure in `AGENTS.md` (lines 97 to 102). Nothing records "this session is checkpointed", nothing validates that a resume follows a checkpoint, and nothing stops "do work" advice while paused.

Q-59 asks whether the lifecycle should become explicit, tool-validated state. This document answers the sub-question: WHERE should that state live, and HOW should it be shaped, and crucially how does it compose with Q-58 (the tool-side structuring of the transient resume-state), so the two are designed as one thing rather than bolted together.

## 2. What already exists (the substrate facts the options are judged against)

Three durable substrates already carry workflow state, at three lifetimes (`AGENTS.md`, "Tracking progress", line 61):

- The plan source `<task>.plan.toml`. Durable, committed, NOT deleted at task close. It is the single source of truth for step status (`[[step]].status`) and for decisions (the Open-Questions queue). Its `[meta]` block (`src/plan/source.rs:103`) is a structured, `deny_unknown_fields` table that `validate --source` already parses and checks (`src/plan/source.rs:527`). It is a single-cell current-value store, not a history. The orchestrator already edits it directly on main as an integration act (flip a step status, re-render).
- The review ledger `<task>.ledger.md`. Committed and versioned beside the plan, but DELETED at task close. It is the declared home of "the non-plan-derivable transient in-flight state" and its `## RESUME STATE` anchor (`.agents/LEDGER.template.md`) is described as "a pointer plus transient state, not a restatement of the Roadmap". It is Markdown prose today; `next` extracts the `## RESUME STATE` section VERBATIM (`src/next.rs:936`, `extract_resume_state`) rather than parsing it, and the Stage-1 A-b decision deliberately chose verbatim-echo to avoid heuristic prose parsing.
- The round log `docs/metrics/workflow.jsonl`. Append-only, schema-validated (`src/metrics.rs`), never rewritten, committed, accumulating across tasks. It is the natural home for a full lifecycle TRAIL and for offline sequence checks (`validate --workflow` W3 / W4 / W5 in `src/workflow.rs`). One decisive constraint: the log exists ONLY when the project was scaffolded with `--instrument`. The `AGENTS.md` Instrumentation section (line 151) states it is "present only because instrumentation was enabled; a scaffold without `--instrument` omits it entirely." The pause / resume / compaction-prep lifecycle, by contrast, is core: it ships in every scaffold and is not instrumentation-gated.

A second fact matters for what belongs in the transient. `next` already RECOMPUTES the active loop, the streak, and the round count from the plan plus the log (`src/next.rs:629`, `build_in_progress_loop`); the streak is derivable, not stored state. So the genuinely non-derivable transient is small: the locked in-round context (a pending dismissal re-check, a running debate) and, once Q-59 lands, the session-lifecycle mode. The session-lifecycle mode is a NEW piece of exactly the same category the `## RESUME STATE` anchor already exists to hold: non-plan-derivable, non-log-derivable, would-be-lost-on-compaction in-flight state.

## 3. The design space

The state has two parts that should be named separately because they behave differently:

- The lifecycle cell: `state = active | checkpointed`, `checkpoint_kind = pause | compaction`, `checkpoint_commit = <hash>` (the last two present exactly when checkpointed). This is small and changes rarely (only at a pause or a resume). The question "am I checkpointed right now?" is answered by reading it.
- The lifecycle trail: the ordered history of checkpoint and resume events. This answers "did every resume follow a checkpoint?" and "were there two checkpoints with no resume between them?" and feeds an offline `validate --workflow` sequence check.

The current state is a fold over the trail (last event wins), so a trail can serve both needs; but a trail alone cannot exist on a non-instrumented project. The options below are for the CELL (the always-needed current-state home); the trail is treated as an optional add-on in the recommendation.

### Option A: a `[meta]` enum in `plan.toml`

Add `[meta.session]` (or scalar `[meta]` keys) carrying `state`, `checkpoint_kind`, `checkpoint_commit`. `validate --source` enforces the shape and the "checkpoint fields present iff checkpointed" invariant; `next` / `status` read it from the plan they already load.

### Option B: append-only `type:"checkpoint"` / `type:"resume"` records in `workflow.jsonl`

Add two record types to the metrics schema (`src/metrics.rs`). Each checkpoint record carries `checkpoint_kind` and `checkpoint_commit`; the current state is the fold over the newest checkpoint/resume record. `validate --workflow` enforces the ordering.

### Option C: a structured field in the ledger

Replace the prose `## RESUME STATE` anchor's non-derivable transient with a machine-parseable structured region (a fenced ` ```toml ` block delimited inside the section, or the section promoted to a small structured companion source the ledger view is projected from). It carries the lifecycle cell alongside the other non-derivable transient (the locked in-round context and the active-step pointer). `next` parses that one region instead of echoing prose.

### Option D (hybrid): Option C as the always-present source of truth, plus Option B as an optional trail

The lifecycle cell lives in the structured ledger transient (C), which is also the carrier Q-58 structures, so Q-58 and Q-59 share ONE mechanism. When instrumentation is on, ALSO append `type:"checkpoint"` / `type:"resume"` records to the log (B) for the audit trail and the offline ordering check. Option A is declined as the primary home.

## 4. Trade-offs, judged against the plan's Project Principles (by name)

### Option A: `[meta]` enum in `plan.toml`

- For, "Structured data first, project for humans" and "Make illegal states unrepresentable": the plan source is the trusted structured substrate; `validate --source` already runs over `[meta]` and can enforce the enum and the present-iff-checkpointed invariant with almost no new machinery, so an illegal combination (checkpointed with no commit, a `checkpoint_kind` while active) is rejected at the boundary.
- For, "Safe on existing projects": the plan is always present, unlike the instrument-gated log, so the lifecycle works on every scaffold.
- For, "Idempotent": a single current-value cell is naturally idempotent; setting `state = checkpointed` twice is the same state.
- Against, "Prefer the cleaner long-term architecture over the smallest diff" and one-source lifetime separation: the plan is the DURABLE cross-session artifact and is not deleted at task close, whereas the lifecycle mode is per-session transient that should reset to `active` when the task closes. Putting it in `[meta]` mixes two lifetimes in one file. More importantly, it SPLITS the lifecycle away from the Q-58 transient it is a sibling of: the locked in-round context (also non-derivable in-flight state) cannot sensibly live in the durable plan because it churns every round and would thrash the committed plan, fighting the "clean tree before a writer" discipline (`AGENTS.md`, line 77). So Option A forces two homes for one category of state.
- Against, "Minimal by default": binding Q-59 to `[meta]` while Q-58 restructures the ledger transient means two schemas, two validators, and two things a resuming agent must read, for one lifecycle story.

### Option B: append-only records in `workflow.jsonl`

- For, "Structured data first" and "Prefer the cleaner long-term architecture": append-only, schema-validated records are the pattern the instrumentation already uses; the ordering enforcement (resume-follows-checkpoint, no double checkpoint) is a sequence check over records exactly like W4 / W5, and the trail is genuine evidence for calibration (how often sessions checkpoint, whether resumes are disciplined).
- For, "Idempotent" and "Reproducible": the log is never rewritten, so a re-run appends rather than mutating, and the current state is a deterministic fold over the file.
- Against, "Safe on existing projects" (decisive): the log exists only under `--instrument`. A non-instrumented scaffold has no `workflow.jsonl` at all, so a lifecycle that lives ONLY here has no home on those projects. The pause / resume lifecycle is core, not calibration, so it cannot depend on an optional module. This rules B out as the SOLE or PRIMARY home; it does not rule B out as an optional trail layered on top of an always-present cell.
- Against, "Make illegal states unrepresentable": deriving the current state by folding a trail is more indirection than reading a cell, and a truncated or partially-written log makes the current state ambiguous in a way a single validated cell is not.

### Option C: structured field in the ledger

- For, "Prefer the cleaner long-term architecture" and one-source lifetime separation: the ledger is already the declared home of non-plan-derivable transient in-flight state and has the exactly-right lifetime (per-task, deleted at close, reset implicitly by starting a fresh ledger). The session-lifecycle mode is a new instance of precisely the category the `## RESUME STATE` anchor exists for, so it lands in its natural home rather than a borrowed one.
- For, "Structured data first, project for humans": structuring the transient is Q-58's own direction (its option 2). A fenced, delimited ` ```toml ` block is deterministic extraction (a hard boundary, parsed like `section_lines` already bounds `## RESUME STATE`) followed by a strict TOML parse; it is NOT the heuristic prose parsing the Stage-1 A-b decision avoided, so it is compatible with that decision while making the transient tool-projectable and drift-free. The surrounding human prose is then a projection of, or a pointer to, the structured block.
- For, "Safe on existing projects": the ledger is always present once a task is in flight (created from `.agents/LEDGER.template.md` at task start), independent of `--instrument`.
- For, "Minimal by default": this is the mechanism Q-58 is building anyway; Q-59 adds a few fields to it rather than a second substrate. One carrier, one parser, one thing to read on resume.
- Against, "Structured data first" (a purity caveat) and "Reproducible": embedding a structured block inside a Markdown file is slightly less clean than the plan's own precedent, which EXTRACTED the skeleton into `<task>.plan.toml` and rendered the `.md` rather than embedding structure in prose. The fully-consistent form is a small structured companion source (a `<task>.resume.toml`) the ledger view is projected from; that is more P8-pure but adds a new artifact and a second file to keep in sync and delete at task close (a "Minimal by default" cost). The fenced-block-in-ledger form avoids the new file at the cost of one parsing surface inside Markdown.
- Against, "Make illegal states unrepresentable": the ledger is not currently machine-parsed for structured fields (only verbatim-extracted), so enforcing the present-iff-checkpointed invariant needs new parsing and validation, which `[meta]` (Option A) would get almost for free. This is real work, but it is work Q-58 already implies.

### Option D: hybrid (C as source of truth, B as optional trail)

- For, "Prefer the cleaner long-term architecture" and "One source of truth" in spirit: this is the SAME dual-write the workflow already uses for round-level state, which the "Tracking progress" rule (`AGENTS.md`, line 61) describes exactly: the counts are read from the ledger narrative, and the workflow ALSO appends a structured `round` record when instrumentation is on. The lifecycle cell (always-present, in the ledger transient) is the source of truth; the log trail (present only under `--instrument`) is the calibration-and-audit projection and the offline enforcement substrate. The current state never depends on the optional log.
- For, "Safe on existing projects": lifecycle works on every scaffold via the cell; the trail is a bonus where instrumentation is on.
- For, "Make illegal states unrepresentable" at two tiers: the always-present cell carries the write-side invariant (present-iff-checkpointed), and the log carries the offline ordering invariant (resume-follows-checkpoint, no double checkpoint) where the records exist, matching W4/W5's split of concerns.
- Against, "Minimal by default": it is two writes, so the checkpoint / resume action must keep the cell and the (optional) trail consistent. This is bounded (the same discipline round-logging already lives with) but it is more than a single write.

## 5. Recommendation

Adopt Option D: ONE structured transient carrier as the always-present source of truth, plus an OPTIONAL append-only trail in the log. Concretely:

1. The session-lifecycle cell lives in the SAME structured transient that Q-58 defines for the resume-state, in the ledger (Option C). This is the load-bearing composition claim: Q-58 and Q-59 are ONE mechanism, not two. The carrier holds the lifecycle cell (`state`, `checkpoint_kind`, `checkpoint_commit`) next to the other genuinely non-derivable in-flight transient (the locked in-round context, a pending re-check or running debate) and a POINTER to the active step. It does NOT restate anything derivable: the streak, the round count, and the active loop stay projected by `next` from the plan and the log (one source of truth, "Make failure and absence explicit" for the pointer). The session-lifecycle mode is simply the newest member of the non-derivable-transient category the `## RESUME STATE` anchor already exists to carry, which is why it shares that anchor's home and lifetime.
2. Structure the carrier as a machine-parseable region (a fenced, delimited ` ```toml ` block inside `## RESUME STATE`), parsed deterministically and validated for the present-iff-checkpointed invariant. This satisfies Q-58's structuring goal and Q-59's illegal-states goal in one move, and it is compatible with the Stage-1 A-b "no heuristic prose parsing" decision because a delimited fenced block is a hard boundary, not prose heuristics. The human-readable RESUME STATE prose is projected from, or points at, the block ("Structured data first, project for humans").
3. When instrumentation is on, ALSO append `type:"checkpoint"` / `type:"resume"` records to `workflow.jsonl` (Option B) for the audit trail and the offline `validate --workflow` ordering check. This mirrors the existing round-level dual-write exactly, so it introduces no new architectural pattern, and the current state never depends on the optional log.

Reasoning against the principles. The decisive one is "Safe on existing projects": the pause / resume lifecycle is core and must work without `--instrument`, which eliminates the log as the primary home (Option B) and forces an always-present cell. "Prefer the cleaner long-term architecture" and the plan's own lifetime separation eliminate `plan.toml` `[meta]` as the primary home (Option A): the lifecycle mode is per-session transient with a sibling (the locked in-round context) that cannot live in the durable, churn-averse plan, so binding Q-59 to `[meta]` would split one category of state across two homes and two schemas. The ledger is the substrate whose declared purpose and per-task lifetime already match. "Structured data first" and "Make illegal states unrepresentable" are satisfied by structuring that carrier (which Q-58 wants anyway) and validating the enum plus the present-iff-checkpointed invariant. "Minimal by default" is satisfied because Q-59 adds fields to the Q-58 mechanism rather than standing up a parallel one. The optional trail earns its second write the same way round-logging already does, under "Ground decisions in evidence": the trail is what later shows whether the discipline holds in practice.

One sub-decision I leave to the human, with a lean. The structured carrier can be realized as (i) a fenced block inside the ledger's `## RESUME STATE` section, or (ii) a small structured companion source `<task>.resume.toml` that the ledger view is projected from. Form (ii) is the more consistent expression of "Structured data first" (it matches the plan's own skeleton-extraction precedent and keeps structure out of prose) but adds a new artifact and a second file to keep in sync and delete at task close. Form (i) is more "Minimal by default" and keeps the one transient file the resume procedure already centers on. I lean to (i) for now because the ledger is already the transient home and Q-58's structuring is the smaller move there; if the ledger later becomes a fully rendered artifact like the plan, promoting the block to (ii) is a clean follow-up. This is a form choice within the recommendation, not a change to it, and it should be decided together with the Q-58 explorer's view since they build the same carrier.

## 6. What NOT to build (the YAGNI boundary)

- Do NOT put the lifecycle state in `plan.toml` `[meta]` as its primary or sole home. It mixes the durable plan's lifetime with per-session transient and splits Q-59 from the Q-58 transient. (If the command-surface explorer finds a strong reason to expose a coarse "is this plan's work paused" flag to consumers that never load the ledger, that is a narrow projection question to raise then, not a reason to make `[meta]` the source of truth now.)
- Do NOT make the lifecycle depend on `workflow.jsonl` as its only home. That breaks on every non-instrumented project. The log is the optional trail, never the source of truth.
- Do NOT store derivable facts in the carrier. The consecutive-clean streak, the round count, and the active loop are recomputed by `next` from the plan and the log; duplicating them into the transient block creates a second source that can drift. Store only the non-derivable: the lifecycle cell, the locked in-round context, and a pointer to the active step.
- Do NOT invent a rich lifecycle state machine (paused / suspended / hibernating / draining and the like). The evidence is a two-state need (`active` vs `checkpointed`) with a two-value kind (`pause` vs `compaction`). Model exactly that; add a state only when a concrete transition needs it.
- Do NOT add a general-purpose key-value "session metadata" bag to the transient. Add named, typed fields for what the lifecycle and the resume actually need, so `validate` can check them; a free-form bag defeats "Make illegal states unrepresentable".
- Do NOT design the checkpoint / resume COMMAND surface or the `validate --workflow` ENFORCEMENT rules here. Which subcommands write the cell and the trail, and which ordering invariants are asserted offline, are the sibling explorers' axes; this document fixes only where the data lives and what shape it takes, so those axes have a settled substrate to build on. The one constraint this data model hands them: the current state is a validated cell (cheap to read and to keep idempotent), and the trail, where it exists, is the append-only evidence for the ordering checks.
