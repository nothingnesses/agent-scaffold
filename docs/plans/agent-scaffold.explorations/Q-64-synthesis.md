# Q-64 design-pass synthesis

Synthesises the four lens explorations in this directory (`Q-64-generation-architecture.md`, `Q-64-checkable-promptable.md`, `Q-64-delivery-fsm.md`, `Q-64-yagni-skeptic.md`) plus the source questions Q-64, Q-65, Q-51 (decided), and Q-58 (open). Written to move Q-64 from `exploring` to `open` with options for the human, and to produce the Q-65 identity/rename output. It does NOT decide; it crystallizes the design and lays out options with a recommendation. Principles are cited BY NAME (plan.toml `[[principle]]` 1-8).

## The question

Does a separate hand-authored AGENTS.md still make sense once the tool matures, and how far can the tool be pushed toward self-sufficiency? Q-65 rides along: is this still a scaffolding tool or a workflow engine/driver, and should it be renamed?

## 1. Where the four lenses converge

1. Direction A holds, but SLOT-WISE, not whole-file. Keep a hand-authored AGENTS.md TEMPLATE and grow the set of GENERATED slots inside it; reject whole-file generation. Explicit in generation-architecture (sec 5.2): whole-file generation would force ~95% authored paragraphs through a structured source for no marginal drift-safety over slots. delivery-fsm reaches the same shape from the delivery side (keep the role prompt a POINTER, deliver only the small control slice inline, sec 3). yagni agrees by exclusion: single-source only the genuinely-duplicated slices, do not generate the un-duplicated floor (sec 5).

2. The irreducible AUTHORED floor is SMALL and stable. All four agree the residue that must stay authored at any maturity is: the judgment-eliciting prompts (which already live in `pack/prompts/*.md`, not AGENTS.md), the unobservable behavioral-compliance rules, and the self-bootstrap stub (checkable-promptable sec 2; generation-architecture sec 5.5; delivery-fsm sec 6 escape hatches). checkable-promptable sizes the irreducible authored floor at ~5-10% of the file.

3. Rung 1 DELIVERY is largely shippable NOW, reusing `next`/`render`/W3. delivery-fsm lays out D1-D5 as small edits to the shipped `build_instruction`/`build_context`/`run_next` seam, no new subsystem (sec 6). yagni concurs: "Rung 1 DELIVERY... LARGELY SHIPPED," justified now (sec 2). generation-architecture and checkable-promptable both treat the rung-1 fragment work as the cheap, proven continuation.

4. The `ISOLATION_POLICY_FRAGMENT` / `control_fragment` single-source pattern is the PROVEN mechanism to generalize. One source of truth (a Rust const or spec-derived render), consumed by BOTH the AGENTS.md slot AND the runtime, byte-guarded so the copies cannot drift (generation-architecture sec 2; checkable-promptable table; delivery-fsm sec 2-3; yagni sec 5, "the isolation fragment is the model"). Q-60 already DECIDED to generalize exactly this pattern to the recommendation-in-options rule.

5. A whole-file byte guard for AGENTS.md is a cheap, high-value gap. Today only per-fragment `.contains()` guards exist; nothing asserts a fresh `build_assets` render byte-equals the committed root AGENTS.md, so a hand-edit of authored prose or a dropped slot is not caught the way `render --check` catches an edit of `<task>.md` (generation-architecture sec 2, 5.3). This is the cheapest item and is independent of any slot-widening.

## 2. The genuine disagreement (the skeptic dissents)

Three lenses (generation, checkable, delivery) lean AFFIRMATIVELY toward the generated-projection direction and a descriptive rename. The yagni-skeptic dissents on four load-bearing points, and the dissent is honest, not noise:

- Drift evidence is THIN. The cited recurring-drift case is two incidents, both low severity, both caught in-process: the driver principle-numbering bug (found in a design pass, cured by DELETING the fragile citation via "de-number reminders" commit 77b9376, not by a generator) and the W4-baseline doc drift (`low_risk`, one increment, step `reconcile-baseline-doc-drift` order 73). By Ground-decisions-in-evidence, two caught-cheaply lags are a thin basis for the largest tooling initiative on the roadmap (yagni sec 1).
- Generating the single-copy floor removes ZERO drift. Drift bites only where TWO copies exist. The judgment prompts, bootstrap stub, and human contract have ONE copy, so generating them adds a generator for no drift removed. Direction A over-reaches if it treats the un-duplicated floor as duplicated (yagni sec 5).
- The rung1->rung2 KNEE is where cost overtakes benefit. `pack/workflow.toml` is deliberately constants-only ("ONLY constants, no logic and no orderings," `:4-5`); widening it to states/transitions/guards/templates plus a generator that synthesizes readable English workflow prose from a state table is unproven work. Neither proven generator does this: `control_fragment` renders one arithmetic line from constants, `render` splices verbatim sidecars; neither SYNTHESIZES control-logic prose (yagni sec 2).
- The meta-stability gate is measurably NOT met. The Q-51 skeptic set "a multi-week stretch with no structural change" as the precondition for moving logic into code; the review protocol itself changed THIS WEEK via Q-63 (triager runs only on rounds with findings, 2026-07-22). Encoding a states/transitions/guards spec now pours concrete around a design still being poured; while the workflow is molten, PROSE is the cleaner substrate under Prefer-the-cleaner-long-term-architecture (yagni sec 4).
- A rename now is PREMATURE and self-fulfilling. Naming the tool a "workflow engine" would justify building the deferred rungs to earn the name, dragging scope against Minimal-by-default (yagni sec identity read).

The four live axes, both sides stated:

- (a) HOW FAR to push generation. Affirmative: build the rung-1 fragment slots now (roles, findings-naming, the Q-60 rule, per-phase rule fragments, per-phase principles by name), reusing the proven pattern (generation sec 5, delivery D1-D5). Skeptic: build ONLY the few proven-duplicated slices; treat widening `workflow.toml` to carry per-phase RULE fragments and generating the workflow SECTION prose as the deferred knee. Note the boundary is subtle: delivery-fsm's D2 relocates EXISTING authored rule text into the spec and renders it into both places (opaque-template relocation, like `render` sidecars), which is milder than the rung-2 "synthesize prose from a state table" the skeptic objects to; the `{{workflow_phases}}` phase-machine generation, which generation-architecture itself calls "the largest and last to land" gated on Q-51's widened spec (sec 5.6), is deferred by EVERYONE.
- (b) WHETHER/WHEN to rename. Affirmative: identity has already crossed; rename now to a descriptive driver name. Skeptic: rename only when rungs 2-3 actually ship and the tool OWNS transitions. See sec 4.
- (c) WHICH rungs are build-now vs evidence-gated. All four agree rungs 2-4 (widen to states/transitions/guards, generate the workflow section, `record-*` write-path, bootstrap stub) are gated. The split is only over the rung-1 MARGIN in (a).
- (d) The UN-GATE signal. yagni makes it concrete and measurable (sec 5): rung 2 un-gates only when ALL hold: (i) doc-vs-mechanism drift RECURS at a material rate AFTER the cheap single-sourcing ships (count such incidents in `docs/metrics/workflow.jsonl` / the ledger, threshold e.g. more than a small N/month that ESCAPED review); (ii) a multi-week stretch with no structural workflow change; (iii) the AGENTS.md-section-from-spec generator PROVEN on one small slice to produce correct readable guidance. The other three do not dispute this gate; they front-load more of rung 1 beneath it.

## 3. Reconciliation with Q-51 and Q-58 (Q-64 composes, not competes)

Q-51 is DECIDED: the human chose the FULL DRIVER (staged) as the TARGET over the four explorers' advisory-MVP-only recommendation, folded into the `workflow-driver` umbrella. So "build the whole ladder eventually" is SETTLED; Q-64's live question is SEQUENCING and GATES, not whether. Q-64's generation work is Stage 0b generalized (generate the AGENTS.md control fragment from the spec, byte-guarded), not a second initiative or a second spec format (generation-architecture sec 4). The projection-family framing is Q-51's: `validate` = backward check, `status` = summary, `next` = forward instruction, `render`/the AGENTS.md fragments = forward prose, all reading one spec.

Q-58 (open, the structured transient) gates the Stage-2 SHARED RECONSTRUCTOR seam but NOT the Stage-1 delivery increment. delivery-fsm sec 5 draws the clean split: Stage 1 keeps the verbatim `## RESUME STATE` echo and leaves the `artifact`/`diff`/`awaiting` context slots UNFILLED (no Q-58 dependency); Stage 2 consumes the structured Q-58 transient to fill those slots and land the `Await` sub-state. So Q-58 must be DECIDED before Stage 2, not before Stage 1. Q-59's decision already folded the session-currency signal INTO Q-58's carrier, so Q-58 is the single structured-transient carrier. Q-64 therefore composes with both: it advances Q-51's staged target and consumes Q-58's output at the Stage-2 boundary, without reopening either.

## 4. Q-65 identity and rename output

IDENTITY. On three lenses the project has crossed from SCAFFOLDING tool to structured WORKFLOW ENGINE / DRIVER: the tool owns three enforced structural invariants (W3/W4/W5), a closed-state FSM reconstruction (`next`), `render`/`validate` pinning the human view to a structured source, and a single control-constant spec the arithmetic reads; `scaffold` is one bootstrap subcommand (checkable-promptable identity read, generation-architecture identity read, delivery-fsm identity read). The skeptic's honest qualifier: there is no `src/driver/`, no transition OWNERSHIP, no write path, no scheduler; the tool ADVISES and CHECKS, so "driver/engine" in the OWNS-transitions sense is aspirational, and the accurate current identity is "a workflow-aware scaffolder + workflow validator + advisory next-projector" (yagni identity read). Both are right about different scopes: the ENFORCED + ADVISORY surface is real and already exceeds "scaffold"; the AUTHORITATIVE driver is staged and unbuilt. All four agree it is NOT a meta-harness: agents INVOKE the tool and it never runs the LLM loop, so "meta-harness" is a category error (Q-65 text; delivery-fsm identity read).

RENAME. Descriptive over metaphor is the consistent lean (the project values clarity over cleverness): descriptive candidates `agentflow`, `flowgate`, `agentctl`; metaphor candidates `cox`/`coxswain`, `maestro`/`podium`, `helm`. The timing tension is the core Q-65 disagreement: the skeptic argues the name should FOLLOW rungs 2-3 shipping (name reality, not aspiration, and avoid name-driven scope creep); the other three argue the identity has already crossed and "scaffold" imposes a standing comprehension tax against Structured-data-first (project for humans) and Prefer-the-cleaner-long-term-architecture.

## 5. Options for the human

Each option cites the 8 principles by name.

### Option A: adopt the generated-projection direction now (build rung 1 + cheap single-sourcing, defer rungs 2-4)

Build: (1) the whole-file AGENTS.md drift guard; (2) the Q-60 recommendation-in-options rule as a rendered single-source fragment; (3) the rung-1 delivery increment D1-D5 (widen `workflow.toml` to `[roles]`/`[paths]`/`[isolation].tier_order`/per-phase rule fragments; replace originated `base_reminders` with spec-rendered per-phase fragments; generalize `projected_principle_reminder` to a phase->principle-names map projected BY NAME; resolve the isolation tier; widen the isolation reminder to reviewer-spawn states per the already-decided Q-62); plus the roles / findings-naming slots that kill the existing `next.rs` duplicates. Defer rungs 2-4 (states/transitions/guards spec, `{{workflow_phases}}` generation, `record-*`, bootstrap stub) behind the sec-2(d) evidence gate. Rename: decide separately (sec 6).

- FOR: Structured-data-first and Prefer-the-cleaner-long-term-architecture drive the projection direction; Make-illegal-states-unrepresentable is served by the drift guard and by projecting principles by name (killing the numbering-bug class at its root); Ground-decisions-in-evidence and Minimal-by-default are honored by staging rungs 2-4 behind the gate; Safe-on-existing-projects holds because D1-D5 are read-only and `WorkflowSpec::builtin()`-backed, so an un-migrated project is byte-unaffected until it scaffolds the widened spec (delivery-fsm sec 7).
- AGAINST: the contested margin inside A is D1/D2's widening of `workflow.toml` to carry per-phase RULE fragments; the skeptic reads this as edging into the rung-2 knee while the workflow is still molten (Q-63 changed the protocol this week), a tension with Ground-decisions-in-evidence and Minimal-by-default.

### Option B: the skeptic's minimal path

Build ONLY the whole-file drift guard plus cheap single-sourcing of the FEW proven-duplicated slices (the Q-60 rule, principle-by-name projection replacing fragile numeric citations, and any other short/stable genuinely-duplicated slice via the proven fragment pattern). Defer ALL of the rung-1 delivery WIDENING (the `workflow.toml` rule-fragment additions, the per-phase fragment rendering) AND the rename until a multi-week workflow-stability stretch plus a measured recurring-drift signal (yagni sec 5).

- FOR: Minimal-by-default and Ground-decisions-in-evidence at their strongest; removes the one OBSERVED drift class (numbering) at its root for near-zero cost; keeps the constants-only `workflow.toml` boundary intact while the workflow is still changing; leaves prose (the cheaper-to-change substrate) governing the still-evolving logic under Prefer-the-cleaner-long-term-architecture's LONG-TERM clause.
- AGAINST: forgoes the point-of-action delivery value that Q-51/Q-54 named as the driver's core anti-drift purpose; leaves `base_reminders` as a second, drift-prone copy of guidance whose canonical form lives in the pack (delivery-fsm gap 2); under-delivers on Structured-data-first for the process itself.

### Option C: the fuller push (rung 1 + begin rung 2)

Everything in A, plus begin rung 2: widen `workflow.toml` toward states/transitions and generate the AGENTS.md workflow SECTION from the machine definition, accepting the meta-stability risk.

- FOR: maximizes Structured-data-first and one-source-of-truth (no hand-authored logic prose left to drift); realizes the Q-51 target soonest.
- AGAINST: the notes lean AGAINST this. It re-opens the Q-51 straitjacket risk (guards are machine predicates, but the load-bearing predicates here are JUDGMENTS the tool must consume, never compute), pours concrete around a workflow that changed this week (fails the meta-stability precondition), and requires the unproven prose-synthesis generator; it trades hard against Ground-decisions-in-evidence and Minimal-by-default and only weakly serves Prefer-the-cleaner-long-term-architecture because "long-term" presupposes a settled target (yagni sec 2-4).

### Recommendation

Option A, refined at the skeptic's joint. The defensible CORE of A is the set ALL FOUR lenses endorse: the whole-file drift guard (cheapest, highest-value, catches the W4-baseline class), the Q-60 rule single-source (already decided, waiting), the principle-by-name projection D3 (kills the one observed drift class at its root), the roles/findings-naming slots (kill EXISTING `next.rs` duplicates), and the isolation D4/D5 work (already decided under Q-62). Build these now. For the CONTESTED margin, D1/D2's widening of `workflow.toml` to carry per-phase RULE fragments and the `{{workflow_phases}}` phase-machine prose, attach the sec-2(d) stability gate rather than shipping them under the same breath as the proven wins; this honors the skeptic's meta-stability point (Q-63, this week) without stalling the items whose drift-removal is already evidenced. Judged against the principles: this maximizes Structured-data-first and Make-illegal-states-unrepresentable on the parts with proven drift, while Ground-decisions-in-evidence and Minimal-by-default hold the still-molten logic-generation behind the gate.

What would change the recommendation toward B: if the human weights the meta-stability signal decisively (the workflow is still being redesigned weekly), B's minimal path is the lower-regret stop and the drift guard alone captures most of the safety. What would change it toward C: a measured, review-escaping recurring drift between the workflow LOGIC prose and the transition code AFTER the cheap single-sourcing ships, plus a multi-week structural-stability stretch, plus a proven prose-from-spec generator, i.e. the sec-2(d) gate actually clearing.

### Rename recommendation

Adopt the DESCRIPTIVE-name direction and SEQUENCE the rename to land WITH the Option-A increment, not gated on rungs 2-3 but not picked in a vacuum. Reasoning: even the current ENFORCED (W3/W4/W5, `render --check`, `validate`) plus ADVISORY (`next`) surface already exceeds "scaffold," so Structured-data-first (project for humans) and Prefer-the-cleaner-long-term-architecture argue the public name should stop mis-selling a one-shot generator; but the skeptic's name-driven-scope-creep concern (Minimal-by-default) is real, so avoid "engine," which implies the unbuilt authoritative driver. Lead candidate `flowgate`: it names the workflow plus the convergence GATE the tool actually enforces (the built surface), without over-claiming authoritative driving; `agentflow` is the alternative if the human prefers a broader driver framing, `agentctl` if CLI-control convention is valued. Present the skeptic's defer-fully position (rename only once rungs 2-3 ship) honestly alongside this; the timing is genuinely the human's call, and it ties to which option above is chosen (A/C support renaming with the increment; B pairs naturally with deferring the rename).
