# Q-64 exploration: YAGNI / skeptic lens

Advisory note. Read-only explorer. This is one lens (adversarial / skeptic) among several; it argues the case AGAINST building the self-sufficiency ladder and is deliberately one-sided so the human's decision is not. I take the anti-drift goal as real and shared; the disagreement is only about which rungs earn their cost NOW versus which stay evidence-gated.

Judged throughout against the plan's numbered principles BY NAME (plan.toml `[[principle]]` 1-8): P1 Prefer-the-cleaner-long-term-architecture, P2 Minimal-by-default, P5 Make-illegal-states-unrepresentable, P6 Ground-decisions-in-evidence, P8 Structured-data-first.

## One-line thesis

The drift the project "exists to kill" bites only where TWO copies of a rule exist. The orchestrator's own irreducible floor (judgment prompts, bootstrap stub, human contract) has exactly ONE copy, so it cannot drift and generating it removes no drift. Drift-killing therefore reduces to single-sourcing a SMALL set of genuinely-duplicated slices, which is cheap, bounded, and already proven here (`control_fragment`, `render`, the isolation fragment). That is rung 1 and the tail of the driver-output-generation design, and it is worth finishing now. Rung 2 (widen `workflow.toml` to states/transitions/guards/templates and GENERATE the AGENTS.md workflow section) is the knee: it pours code around a workflow that changed THIS WEEK, re-introduces the Q-51 straitjacket risk the project explicitly deferred, and buys a hard, unproven generator to fix a drift the cheap single-sourcing already removes. Defer rungs 2-4 behind the same evidence gate Q-51's skeptic already defined and the human already accepted staging behind.

## 1. Steelman the status quo, and quantify the real drift pain

The orchestrator's anti-drift case rests on two cited incidents. Read honestly, both are small, both were caught cheaply, and neither escaped to break a workflow run.

- The driver principle-numbering bug. Concretely (driver-output-generation.design.md:26-31): the driver's reminders in `src/next.rs` cited "Principle 6" etc. as hardcoded strings into the `{{principles}}` slot, which `render_principles` renumbers by `i+1` on any selection/reorder (`src/pack.rs:185-195`), AND there are TWO disagreeing numbered lists (the AGENTS.md generic list vs the plan's `[[principle]]`). This is a genuine design smell: a hand-authored numeric citation into a generated list. But note what it actually was. It was found in a DESIGN PASS, not in production, and the fix already shipped by DE-NUMBERING the reminders (commit 77b9376 "de-number reminders"), i.e. by removing the fragile citation, not by building a generator. The related ledger hits (G10/H1: `"Principle 13"/"5"` mixing the two numberings) were REVIEW findings caught in the review loop and rated low. So the "bug" is: a citation style that was fragile, caught by the process working, and cured by deletion.
- The W4-baseline doc drift (step `reconcile-baseline-doc-drift`, order 73, plan.toml:977-988). Root cause: commit 2580476 replaced W4's derived boundary with a declared `[meta].w4_baseline` record; the AGENTS.md prose describing the old mechanism lagged one step behind the code change. Classified `low_risk`, a single increment, fixed in one pass.

Quantified: across the project's entire history the cited recurring-drift evidence is TWO incidents, both low severity, both caught by review or a design pass, neither reaching a bad workflow outcome. That is not a frequent recurring cost; it is two one-off lags of the shape "prose describing a mechanism the code changed." By P6 (Ground-decisions-in-evidence), two low-severity, caught-in-process incidents are a thin basis for "the drift the project exists to kill," and a very thin basis for the largest tooling initiative on the roadmap. The status quo (a lightly slot-generated AGENTS.md plus `validate --workflow` W3/W4/W5 catching every illegal CONTROL state post-hoc, `src/workflow.rs`) is working: it caught the receipt/waiver/round gaps it is designed to catch, and the two prose lags were caught by human review, which is exactly where prose lag SHOULD be caught cheaply.

The honest gap in the status quo is not "drift is rampant." It is "a few specific slices are duplicated between AGENTS.md and a structured source and could be single-sourced." That is a scalpel job, not a subsystem.

## 2. Cost each ladder rung; find the knee

Rungs from Q-64's ask (plan.toml:1461). Current built state: Stage 0a (`workflow.toml` constants, `pack/workflow.toml`), 0b (`control_fragment` -> `{{workflow_control}}` slot, byte-guarded), and Stage 1 (advisory `next`, `src/next.rs`) are DONE. There is no `src/driver/`, no `record-*` subcommand, and `--strict` exists only on `render`, not on `validate --workflow` (`src/main.rs`). So the tool today is: a scaffolder + a post-hoc validator + a status/next projector. Stages 2-3 and rungs 2-4 are unbuilt.

- Rung 1 DELIVERY (`next` emits the relevant slice). LARGELY SHIPPED. Marginal remaining work: fuller per-state instruction templates and the isolation-fragment-style projections from driver-output-generation (project principle name/text from `[[principle]]` instead of citing a number; already designed, Category 2). Cost: low, and it reuses proven machinery. Drift removed: the numbering-bug CLASS at its root, which is the one design smell actually observed. Verdict: JUSTIFIED NOW. This is real, cheap, and evidence-grounded (P6, P8).
- Rung 2 SOURCE (widen `workflow.toml` to states/transitions/guards/templates; generate the AGENTS.md workflow SECTION from it). THE KNEE. `workflow.toml` is today deliberately constants-only: its own header says "This file holds ONLY constants, no logic and no orderings: the transition arithmetic stays in the tool" (`pack/workflow.toml:4-5`). That boundary was drawn on purpose, honouring the Q-51 skeptic. Rung 2 erases it. Cost: a states/transitions/guards schema PLUS a generator that synthesizes correct, readable English workflow guidance from a state table. The project's two proven generators do NOT do this: `control_fragment` renders a one-line arithmetic statement from CONSTANTS, and `render` splices VERBATIM prose sidecars (`src/plan/render.rs`). Neither synthesizes control-logic prose from a spec, which the driver-output-generation design itself flags as the harder problem and rates "feasibility GENERALLY lower" for prose rules (design.md:69). Drift removed beyond rung 1: only the drift between the workflow LOGIC prose and the transition code, which is not among the two observed incidents. Verdict: DEFER, evidence-gated. Cost exceeds observed benefit; violates P2 (Minimal-by-default) and P6.
- Rung 3 ENFORCEMENT (`record-*` write path; opt-in `validate --workflow --strict` boundary gate; more invariants). Split it. The opt-in boundary gate is the least-bad enforcement (blocks only at landing, harness-agnostic, least-authority) and is cheap IF the checkable rules are already single-sourced; it is defensible once there is a measured need. The `record-*` write path is different: it reopens Q-24 (least-authority; the tool gaining a write path into durable state) and adds a fresh drift/authority surface to fix an ergonomic annoyance (hand-appending JSONL) that is not a correctness problem. Verdict: the `--strict` gate is EVIDENCE-GATED-CHEAP (build when a landing actually slips through review, not before); `record-*` is DEFER (P2, and it reopens a settled least-authority decision).
- Rung 4 BOOTSTRAP (shrink AGENTS.md to a generated stub + contract). Most speculative. Depends entirely on rung 2 existing (you can only shrink AGENTS.md to a stub once the generator can produce the rest). Verdict: DEFER; do not design against it now.

The knee is between rung 1 and rung 2. Everything up to and including the cheap single-sourcing of duplicated slices is justified; the states/transitions/guards spec + AGENTS.md-section generator is where cost overtakes observed benefit.

## 3. The cage risk (control-vs-judgment), carried from Q-51

Q-51's skeptic (mealy-workflow-driver.explorations/yagni-skeptic.md:61-79) named the straitjacket: AGENTS.md is saturated with JUDGMENT that resists deterministic encoding ("match the ceremony to the stakes," classify risky "when a defect would be costly or hard to reverse," collapse roles for a trivial change, four-plus entry modes each reusing shared machinery in judgment-dependent ways). The Q-51 DECISION honoured this by keeping the workflow logic DATA-DRIVEN but scoping the shipped spec to CONSTANTS ONLY (`pack/workflow.toml`), leaving transition arithmetic in typed Rust and the judgment in prose. That is the boundary as-built.

Q-64 rung 2 re-opens exactly this risk. "Widen to states, transitions, GUARDS, and TEMPLATES" is the straitjacket vector: a guard is a machine predicate, and the load-bearing predicates here ("is this artifact risky," "is this request trivial," "is this finding valid") are judgments the tool must consume, never compute (the orchestrator's own irreducible floor, plan.toml:1465-1467). A states/transitions/guards spec that tries to encode them either hard-codes one path and forces every judgment through an escape hatch (the spec becomes mostly escape hatches, and the "determinism" is fiction), or encodes the judgment and is confidently wrong. Either way it makes the judgment-heavy workflow HARDER to evolve: a change that is a prose edit today becomes a spec edit plus its generator re-run plus its byte-guard re-pin plus a code review. Q-64 re-introduces the Q-51 risk under new names ("guards," "templates"); the mitigation is the same one Q-51 chose, keep the spec to CHECKABLE constants and leave PROMPTABLE judgment as prose, which is the argument for stopping at rung 1.

## 4. The meta risk: tooling an evolving workflow is premature

This is the dispositive point, and it is grounded in the project's own live git history, not vibes. The workflow is NOT stable; it is being actively redesigned right now:

- The last two weeks of commits are almost entirely workflow-DESIGN churn: Q-59 (command surface), Q-60/Q-62 fold, Q-63 triager-on-findings (which CHANGED the review protocol so the triager runs only on rounds with findings, commits 935d0f2 / e34cc56, 2026-07-22), Q-64 capture, Q-65 capture (2026-07-23). The review protocol itself changed THIS WEEK.
- The Q-51 skeptic already set "WORKFLOW STABILITY: a meaningful stretch (multiple tasks / weeks) with no structural change" as a precondition for moving logic into code (yagni-skeptic.md:129). That precondition is measurably NOT met; the structure changed days ago.

Encoding a states/transitions/guards spec now pours concrete around a design still being poured. By P1 (Prefer-the-cleaner-long-term-architecture), note the word LONG-TERM presupposes a settled target; while the workflow is molten, PROSE is the cleaner substrate precisely because it is cheap to change, and a spec-plus-generator is premature by P6. The project's own evidence-first, stage-adoption discipline (the receipt and waiver PILOTS, run small and measured before generalizing) is the correct template here: pilot the cheap single-sourcing, measure whether drift actually recurs, and let evidence, not the tidiness of the ladder, pull the next rung.

## 5. Recommendation: build-now vs evidence-gated, and what NOT to build

Refine the orchestrator's direction A. "Keep AGENTS.md but demote it to a generated projection" conflates two things:

- AGENTS.md-as-duplicated-source (the slices that ALSO live in a structured source: the convergence constants, the principle citations, the isolation fragment). These SHOULD be single-sourced. This is drift-killing, and it is cheap and proven. Direction A is right FOR these.
- AGENTS.md-as-authored-floor (judgment prompts, bootstrap stub, human contract). These have NO structured duplicate, so they CANNOT drift, so generating them removes zero drift and only adds a generator. Direction A is WRONG to include these; "demote AGENTS.md to a projection" over-reaches by treating the un-duplicated floor as if it were duplicated.

BUILD NOW (justified, P6/P8, cheap and proven):
- Finish rung 1: the driver-output-generation Category-2 projections (project principle name/text from the plan's `[[principle]]`, replace the fragile numeric citations), and keep constants single-sourced via `WorkflowSpec`. This removes the ONE observed drift class (numbering) at its root.
- Single-source any OTHER slice that is genuinely duplicated AND short/stable enough to byte-guard, via the proven Stage-0b fragment pattern (Route B in the design). The isolation fragment is the model.

DO NOT BUILD NOW (defer behind an explicit evidence gate):
- Widening `workflow.toml` to states/transitions/guards/templates (rung 2). Keep the constants-only boundary the header declares.
- Generating the AGENTS.md workflow SECTION from a machine definition (rung 2). This is the unproven hard generator; do not build it to fix a drift the cheap single-sourcing already removes.
- The `record-*` write path (rung 3). It reopens Q-24 least-authority for an ergonomic gain.
- The BOOTSTRAP stub (rung 4). It depends on rung 2 existing.

THE EVIDENCE SIGNAL that would un-gate rung 2 (make it concrete and measurable, mirroring the Q-51 gate):
- Drift RATE: doc-vs-mechanism drift incidents (of the two cited shapes) recurring at a MATERIAL rate AFTER the cheap single-sourcing ships, i.e. single-sourcing the duplicated slices did NOT stop it. Today the count is two, both pre-single-sourcing, both caught cheaply. The metric already exists in shape: count such incidents in `docs/metrics/workflow.jsonl` / the ledger per month; set a threshold (e.g. more than a small N per month of drift that ESCAPED review) before building the generator.
- Workflow STRUCTURAL stability: a multi-week stretch with no structural change to the workflow logic prose. Not met today (Q-63 changed it this week).
- Generator PROVEN: the AGENTS.md-section-from-spec generation demonstrated on one small slice and shown to produce correct, readable guidance, before any workflow LOGIC moves out of prose. Until then rung 2 is two sources, not one, which is the OPPOSITE of the single-source claim it makes.

Until all three hold, rung 1 plus cheap slice single-sourcing is the right and sufficient stopping point, and the burden of proof is on rung 2 to show, with data, that single-sourcing the duplicated slices left drift on the table.

## Identity read (for Q-65)

From a skeptic's view, calling the project a WORKFLOW ENGINE / DRIVER is ASPIRATIONAL, not accurate to what is built. What exists is: a scaffolder (`pack`, `manifest`), a post-hoc VALIDATOR (`validate --workflow`, W3/W4/W5, `src/workflow.rs`), a status/next PROJECTOR (`src/next.rs`, read-only, advisory), and a plan RENDERER (`src/plan/render.rs`). There is no `src/driver/`, no transition OWNERSHIP, no write path, no scheduler. A "driver" drives control flow; this ADVISES and CHECKS. Honest current identity: a workflow-aware scaffolder with a workflow linter and an advisory next-action projector.

A rename now is PREMATURE, and worse, it is self-fulfilling scope pressure: naming the tool a "workflow engine" would justify building the deferred rungs to earn the name, dragging the project toward the very cage this note argues against (name-driven scope creep against P2). Recommendation: no rename now. Keep the scaffolder-framed name. Revisit ONLY if and when the deferred rungs (2-3) actually ship and the tool genuinely OWNS transitions and enforcement, at which point the name would describe reality rather than aspiration. If a clarifying tagline is wanted in the interim, "scaffolder + workflow validator/advisor" is accurate; "engine/driver" is not yet.
