## Repository Layout and Current Architecture

This section maps the code as built, so the plan can be resumed without prior conversation context. The code in the repository is the source of truth; this is an index into it. The completed steps carry the reasoning; this carries the shape.

Build, run, and verify (see `README.md` for full user-facing usage):

- Use `just build` / `just test` / `just clippy` / `just fmt`, or plain `cargo` inside `nix develop` (with direnv, run `direnv allow` once). The justfile wraps the Nix environment.
- Verification before each commit: `cargo clippy --all-targets -- -D warnings`, `nix fmt`, and all text ASCII-clean (no em-dashes, en-dashes, emoji, or other non-ASCII; use `->`, `>=`, `!=`).
- The binary is a plain Rust program (edition 2021, MSRV 1.88) that runs without Nix. Nix, just, and direnv are development conveniences only. Dependencies: `clap` (derive), `serde`, `toml`, `ratatui = "0.30"`, `include_dir`.

Directory layout:

- `src/` holds the binary crate: four modules described below.
- `pack/` is the built-in default pack: `pack.toml` (the manifest), `AGENTS.md`, `plan-template.md`, `principles.toml`, and `prompts/`. It is embedded into the binary at compile time with `include_dir`, and it also loads through the same manifest path as an external `--template` pack (the pack is dogfooded).
- `flake.nix`, `.envrc`, `justfile`, `rustfmt.toml` are the dev environment (fenix toolchain, treefmt formatters, recipes). `LICENSE` is Blue Oak Model License 1.0.0.

Modules and key types (all shipped and tested):

- `src/main.rs` (CLI and orchestration). `Cli` (clap derive) exposes `--output-dir`, `--force`, `--vcs <git|none>` (git by default), `--write`, `--dry-run` (conflicts with `--write`), `--principles`, `--principle-detail`, `--list-principles`, `--template <dir>`, and `--var key=value` (repeatable); there is no `-i` flag, the selector is the default on a terminal. `main` builds the active `manifest::PackSource` (the built-in pack, or a `--template` directory), parses `--var` into an overrides map, loads the active pack's principles via `pack_principles`, and resolves the selection. `use_selector` then decides the mode: the two-pane TUI opens when both stdin and stdout are terminals, there are principles to choose, and neither `--write` nor `--dry-run` was given; its Save modal is the write confirmation. Otherwise the run is non-interactive and writes only with `--write`. Either way `main` builds the asset set with `build_assets`, computes each asset's `Outcome` with `outcome_of` (read-only, for the plan preview), decides the `--vcs` action with `init_plan` (read-only; `inside_git_repo` skips when already inside a repository), prints the plan (the repository line first), and, when confirmed, runs `run_git_init` (shell out to `git init`) before writing the assets with `apply_asset`. `Outcome` is `Created`/`Refreshed`/`SkippedExisting`/`Overwritten`; `outcome_of` decides it without touching disk and `apply_asset` performs the write, so the preview never writes. `#[cfg(test)] scaffold` is a build-plus-write convenience for the drop tests. User errors print to stderr and exit 2.
- `src/pack.rs` (principle data and selection). `Principle { id, name, summary, rationale, tags, default_selected, default_order, references, related }`. `parse_principles(&str)` parses a principles TOML file. `resolve_selection(principles, spec)` implements the `--principles` token grammar (comma-separated `default` / `all` / `none` / `tag:<t>` / bare-id, de-duplicated by first occurrence; keyword and tag tokens expand in `default_order`, id lists keep their given order) and returns `SelectionError { UnknownId, UnknownTag }`. `Detail { Name, Summary, Full }` with `render_principles` renders the selected set as a numbered list. `default_principles` / `DEFAULT_PRINCIPLES_TOML` are `#[cfg(test)]` only, because production reads principles through the active pack source (see `pack_principles`).
- `src/manifest.rs` (the pack format and loader; the abstraction `optional-modules` builds on). `PackSource { Embedded(&Dir), Directory(PathBuf) }` with a public `read(rel)` and a private `manifest()`; `builtin()` returns the embedded pack. `Manifest { asset: Vec<AssetSpec>, var: Vec<VarSpec>, module: Vec<ModuleSpec> }` deserializes `pack.toml` and ignores unknown keys. `AssetSpec { source, dest, ownership, render, module: Option<String>, executable: bool }`, `VarSpec { name, default: Option<String>, module: Option<String> }`, `ModuleSpec { name, description, guidance: Option<String>, requires: Vec<String> }`, and `Ownership { Reference, Working }`. A `module = "<name>"` tag on an asset or var declares optional-module membership: untagged entries are core (always applied), tagged ones only when `--module <name>` selects that module; the `[[module]]` section is the authoritative set of known module names. `render(template, vars)` does minimal `{{key}}` substitution (unknown placeholders left as-is). `load(source, builtin_vars, overrides, selected_modules)` reads the manifest, filters out unselected modules' assets and vars, resolves the variable map with `resolve_vars` (built-in vars first, then each selected/core declared default, then `--var` overrides; the name `principles` is reserved), and returns `Vec<Asset { dest, contents, ownership }>` or `LoadError { Io, UndeclaredVar, MissingRequiredVar, ReservedVar, UnknownModule, UndeclaredModuleTag, UndeclaredModuleRequire, DuplicateModule }` (all exit 2). The reserved computed variables are `principles`, `instrument`, and `modules` (a pack may neither declare nor `--var`-set them). The `checks` module and the `agent-scaffold checks` / `scaffold --with-precommit-hook` surfaces live in `src/checks.rs` and `src/main.rs` (see the `optional-modules` step).
- `src/tui.rs` (the interactive selector; TEA-lite, functions not traits, where TEA is The Elm Architecture: a single state value, a pure `update` reducer, and a render function). The only public item is `run_selection(principles, initial_included, save_summary) -> io::Result<Option<Vec<usize>>>`, which returns the chosen principle indices in order, or `None` on abort. Internally: one `App` value; `enum Mode { Editing, Filtering, Confirming { button } }`; `enum Event { Key, Resize }`; a pure `update(&mut App, Event) -> Step` reducer (`Step` is `Continue`/`Confirm`/`Abort`); a `ui` render function; and a blocking `run` loop over `next_event()` (the single seam where a concurrent event source could later be merged). No async, no custom traits. Key bindings are documented in the `selection-ui` step.

Pack format (the contract shared by the built-in pack and `--template` packs):

- A pack is a directory containing `pack.toml` plus the source files it references. External packs are passed with `--template <dir>`.
- `pack.toml` has `[[asset]]` entries `{ source, dest, ownership = "reference"|"working", render = true|false, module?, executable? }` (one source may map to several assets; `render` and `executable` default to false; `executable = true` drops the file with its executable bit set, for example a hook script) and optional `[[var]]` entries `{ name, default?, module? }` (an absent `default` makes the variable required), plus optional `[[module]]` entries `{ name, description, guidance?, requires? }` declaring the opt-in modules (`guidance` names a partial concatenated into `{{modules}}` when the module is enabled; `requires` lists modules it auto-enables).
- Optional modules: an asset or var with a `module = "<name>"` tag is dropped or required only when that module is selected with the repeatable `--module <name>` flag; untagged entries are core (always applied), so with no module selected the output is byte-identical to the core. Every module a tag references (and every `--module`) must be declared in a `[[module]]` section, and module names must be unique; a dangling tag, an unknown `--module`, or a duplicate declaration is an error and nothing is written.
- `ownership = "reference"` assets (conventionally under `.agents/`) are always (re)written; `ownership = "working"` assets are created only if absent, unless `--force`.
- `render = true` applies `{{name}}` substitution. `{{principles}}` and `{{instrument}}` are reserved, tool-computed variables (the rendered principle selection, and the instrumentation block when `--instrument` is set); a pack may neither declare nor `--var`-set either. Other variables come from `[[var]]` defaults, overridden by `--var key=value`; setting an undeclared variable, or leaving a required one unset, is an error and nothing is written. Every rendered asset is normalised to end with exactly one trailing newline, so an unused reserved placeholder leaves no trailing blank lines and the off output is byte-identical without relying on a formatter.
- A pack that ships `principles.toml` has its principles drive selection, `--list-principles`, the interactive selector, and `{{principles}}`; a pack without one simply has no principles to select. A pack may likewise ship an `instrument.md` render fragment, which is inlined into `{{instrument}}` when `--instrument` is set; this fragment is a special source read directly (like `principles.toml`), NOT a manifest `[[asset]]`, so it is not dropped as its own file. A pack without one renders `{{instrument}}` empty.

One run's data flow: parse CLI, build the `PackSource`, read and parse the pack's `principles.toml`, `resolve_selection` (or the interactive TUI) chooses and orders principles, `render_principles` produces the `{{principles}}` value, `manifest::load` resolves the variables and reads/renders each asset, `outcome_of` computes each asset's plan outcome, and, once writing is confirmed, `apply_asset` drops each honouring ownership and `--force`.
