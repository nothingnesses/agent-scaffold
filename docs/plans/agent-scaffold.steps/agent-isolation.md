### `agent-isolation`: Capability-tiered writer isolation

Decided (`Q-18`, adopted by the human). Two parts: the isolation RULE (a workflow-doc change, always applicable) and the isolation MECHANISM (optional-module integration, heavier and deferred).

- Rule (in `pack/AGENTS.md`): run each WRITER agent in the strongest isolation the harness supports, in preference order: (1) container isolation via agent-box / agent-images if available, preferred because it isolates filesystem, environment, and a security sandbox; (2) worktree isolation if that is what the harness offers (for example claude-code native worktrees); (3) the `file-safety-rules` discipline as the fallback when the harness has no isolation. Read-only agents (reviewers reading, the triager) need no isolation (Principle 2, minimal by default: do not add machinery a read-only agent does not need). Isolation is the structural upgrade over the file-safety baseline: a killed isolated writer cannot touch the main tree, which directly addresses the two incidents in the ledger (the fmt-plus-checkout ledger clobber and the accidental kill) that are the evidence for prioritising it.
- Mechanism (optional-module work under `optional-modules`, which already lists container and worktree isolation as a behavioural module): our own build and adoption order is worktree-first, since worktree isolation is native to claude-code and cheap now, then agent-box / agent-images container integration as the heavier follow-on that adds environment and sandbox isolation. The runtime selection rule in the workflow docs is container-preferred, but for us today it resolves to worktree anyway.
