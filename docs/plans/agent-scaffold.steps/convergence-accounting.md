### `convergence-accounting`: Fix the convergence accounting (F1, F2, F7, F10)

The core design fix, addressing F1, F2, F7, and F10. The convergence rule's round taxonomy was not a partition, and the "contested" round state the separate contested-rounds cap depended on was never defined or recorded, so that cap could not be computed from the ledger; the cap-boundary wording was inconsistent (F7) and the convergence bullets omitted the total-round cap (F10).

Decision (`Q-1`, decided as approach (b)): drop the separate contested-rounds cap entirely and bound each review loop with a single total-round cap (default five). The per-round outcome is a two-way partition, `clean` or `new-valid`, with no separate round state for a persisting dispute and no cross-round finding ids; escalation fires when the total rounds on an artifact reach the cap, whatever the clean-versus-new-valid mix. Reasoning: judged against Principle 1 (prefer the cleaner long-term architecture) and Principle 5 (make illegal states unrepresentable), the single cap removes the non-partition taxonomy and the under-defined round state instead of adding per-round state and per-finding identity to track them, so there is no unrecordable state and no ambiguous 3-versus-5 boundary to get wrong. The accepted trade-off: the loop loses the "stuck on the same dispute" signal, so a loop that keeps finding new genuine issues and a loop relitigating one finding escalate on the same schedule; the total-round cap still bounds both.

The fix makes the escalation trigger deterministic and computable from the ledger, and states the same rule the same way in every document: `pack/AGENTS.md` (the Convergence and ledger sections), `pack/prompts/orchestrator.md` (the round-counting steps), the ledger round-summary schema (the `clean`/`new-valid` per-round outcome and the running total-round count), and the README diagram's cap-boundary labels (aligning "cap hit" with the prose so F7's off-by-one is gone). This step also owns the matching correction to the plan's own `core-assets` narrative; that narrative's convergence sentence and round-constant count are updated in this plan now (to a single total-round cap and two round constants) so a resuming agent is not misled, matching how F3 was corrected ahead of `workflow-doc-fixes`. Validate by reading the sources against each other and confirming a re-spawned orchestrator could recompute the escalation decision from the ledger alone.
