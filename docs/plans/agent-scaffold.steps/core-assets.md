Decisions carried from the resolved open questions:

- **Form.** A standalone Rust binary (clap CLI plus a ratatui TUI) that runs without Nix; Nix is a development-only dependency (the dev shell). Distributed via `cargo install` or prebuilt binaries, with an optional flake app for Nix users. Runs from any working directory and writes to any target via `--output-dir`, defaulting to the current directory with a namespaced layout.
- **Ownership (two-tier).** Tool-owned reference assets under `.agents/` are always refreshed; user working files (the root `AGENTS.md`, plans under `docs/plans/`) are create-if-absent, with `--force` to overwrite. An existing `AGENTS.md` is left untouched and the namespaced reference copy is refreshed for the user to merge from. Marked-block co-tenancy and a 3-way-merge `update` were considered and deferred (`later-enhancements`).
- **Template source.** A built-in default pack, with optional `--template <local-path>` for bring-your-own packs (`byo-template`); git-URL fetch is deferred to `git-url-fetch` (must not depend on Nix) and a flake-ref stays out of scope.
- **Principle data.** Structured TOML (`pack/principles.toml`). Each principle has `id`, `name`, `summary`, `rationale`, `tags` (category tags plus applicability tags `universal`/`static-types`/`fp`/`oop`), `default_selected`, `default_order`, and optional `references`/`related`. The selected set renders as a numbered list ordered by `default_order`, at `name`, `summary`, or `full` detail. The default set is about two dozen principles, applicability-tagged so it adapts to the project type (see `pack/principles.toml` for the live set and counts); the shipped prompts and guidance stay principle-agnostic and defer to the selected set.
- **Optional modules (not yet built).** A diagram prompt pack, a justfile of recipes, language starters, and container/worktree isolation. Each is defined concretely in `optional-modules`; the isolation module would integrate the separate agent-box (`https://github.com/0xferrous/agent-box`) and agent-images (`https://github.com/nothingnesses/agent-images`) projects, which are not part of this repository.

### `core-assets`: Author the minimal core assets and principle data

Wrote the canonical `AGENTS.md` guidance (with a `{{principles}}` placeholder), the plan-document template, the three core prompts, and the principle data as structured TOML (initially 48 principles, 21 default; see `pack/principles.toml` for the current set, which has grown since), parsed and tested. Content policy: the shipped prompts and guidance stay principle-agnostic and defer to the user-selected principle set in `AGENTS.md`, rather than baking in specific opinions a user may not have chosen (the open-questions gate was corrected to follow this, and the prompts instruct the agent to read `AGENTS.md` to align with the current principles). The workflow (in `AGENTS.md`) is role-separated so no agent grades its own work, and each role has its own prompt in `.agents/prompts/`: `orchestrator.md` (drives the phases, spawns the roles, routes context, runs the review loop, escalates), `planner.md` (drafts the plan; with `clarifying-questions.md` and `open-questions-gate.md`), `reviewer.md` (independent reviewers, assume-issues, given the change context via before and after commit hashes or a diff range, diverse lenses and models where available), `triager.md` (adjudicates each finding, never the agent that produced the artifact, may run a short producer-versus-reviewer debate on a contested finding), and `implementer.md`. Where the harness lacks sub-agents, one agent plays the roles in sequence but records each role's output. Implementation iterates over the plan's Roadmap steps (implement, review to convergence, mark the step complete), and the workflow stops only when every step is complete and an acceptance review (reviewers, then a triager, as in the other review phases) confirms the plan's Success Criteria. The orchestrator's convergence rule ends a single review loop after the required consecutive clean rounds (scaled to stakes, since fresh reviewers make one clean round weak evidence, and a dismissed high-severity finding is re-checked by a second triager before it counts towards a clean round); it escalates to a human, then resumes, if the artifact reaches a total-round cap (default five). Escalation is a decision request, not a stop. The two round constants (the required consecutive clean rounds and the total-round cap of five) are provisional defaults chosen to be safe rather than measured, to be calibrated against data from real use (see `workflow-calibration`). A human may also interrupt at any point with new or changed requests; the orchestrator first runs a bounded intake assessment that advises on impact and routing and gives feedback on the request (the human decides), folding a trivial request in directly and routing anything that touches scope or Success Criteria to the planner, so non-trivial human input always enters durably through the plan rather than ad hoc. Relitigation is prevented by a review ledger (each finding, its verdict, reasoning, and action, plus each round's outcome) handed to each round under a "no re-raise without new evidence" rule; the ledger is kept in a file committed to version control beside its plan so it survives the orchestrator losing context and travels across machines and sessions, deleted (deletion committed) at task close, kept out of the Open Questions section, and only durable decisions fold into the plan's steps (reopened only by evidence that beats the recorded reasoning). The plan template's Project Principles section (and the `AGENTS.md` Plan step that points at it) instructs the agent to seed a plan's principles from the `AGENTS.md` principles in order, add the project-specific ones after, consolidate any overlap into a single amended principle, and keep the list numbered so it can be referenced by number.
