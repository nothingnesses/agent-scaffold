### Workflow self-improvement cluster (structure over orchestrator disposition)

This cluster (`exploration-mode`, `ledger-parse` (skipped), `workflow-invariants`, `state-queries`) came out of a dogfooding incident on 2026-07-16: a `complete` step (`pause.md`) was committed outside the review loop on an orchestrator "match ceremony to stakes" judgement, and no check caught it. The human then asked whether the workflow adequately handles deterministic adherence (`Q-27`), token-efficient state navigation (`Q-28`), and design-space deliberation (`Q-29`). All three share one root cause: workflow state lives in the orchestrator's head rather than in checkable structure. The designs were settled by a two-independent-explorer design pass (opus + sonnet, different models) synthesised by the orchestrator, then decided by the human. Common architecture: extend the existing validator/projection family (single source of truth per region, detection not prevention, hard-fail, no binary required at workflow runtime). Build order: `exploration-mode` (complete), then `workflow-invariants` (next), then `state-queries`, each a separately reviewed step; the `ledger-parse` keystone was SKIPPED once `round-log-core` promoted the JSONL log to the single structured store, so `workflow-invariants` reads `src/plan.rs` + `src/metrics.rs` directly.

### `exploration-mode`: First-class design-space exploration (`Q-29`)

Next. Decided (`Q-29`, human, after the two-explorer design pass). The workflow handles deciding and deferring a decision but has no first-class mode for exploring a design space before options are decidable, which is what the human asked for on `Q-27`/`Q-28`. Four low-ceremony fixes, mostly documentation plus one code line:

- Vocabulary: add `exploring` to `QUEUE_EXACT_STATUSES` in `src/plan.rs` (a sub-state of `open` meaning a design pass is owed before options can be presented), and document it in `pack/plan-template.md` so the existing `plan_template_documents_every_accepted_status` drift guard extends automatically. This is the only code change; `validate --plan` today hard-rejects that phrasing in the status field (as the orchestrator hit when first queuing `Q-27`/`Q-28`/`Q-29`), forcing the nuance into body prose.
- Artifact: a design-notes convention at `docs/plans/<task>.explorations/<question>.md` (parallel to the `docs/plans/<task>.reviews/` convention), whose shape is the human-input contract written to a file before options are decidable: the question, the design space (viable options), each option's trade-offs judged against the numbered Project Principles, a recommendation with reasoning, and an explicit "what I would NOT build" (YAGNI). Prose guidance, not a machine-parsed schema (the document is human-read).
- Intake mode: name design-space exploration as a third entry mode in `pack/AGENTS.md` alongside factual-answer and decision-gate (the Socratic-input section is the natural home): the orchestrator recognises a deliberation request, records an `exploring` queue item pointing at the design-notes file, spawns the explorer(s), optionally runs a light review, then presents options via the human-input contract; the resolved answer becomes a durable Open-Questions decision, converging under the existing no-re-raise-without-new-evidence rule.
- Optional light review: reuse the reviewer/triager roles as a single pass (not the convergence loop) with a "is the design space complete, are the trade-offs honest, is the recommendation Principle-grounded" lens; note that the multi-independent-explorer-plus-synthesis setup (which produced this very decision) is the adversarial analog. Match ceremony to stakes; not mandatory.

Explicitly NOT built: a new workflow phase parallel to Plan/Implement/Accept, a machine-parsed exploration schema, or an `explore` subcommand (the artifact is a Markdown file under a named convention). First dogfood instance: the `docs/plans/agent-scaffold.explorations/workflow-process-cluster.md` note capturing this cluster's own exploration.
