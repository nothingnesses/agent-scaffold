### `findings-files`: Reviewers and triagers write findings to files

Decided (`Q-14`, adopted by the human). Reviewers and triagers write their findings to per-agent files rather than returning them for the orchestrator to transcribe. Part of the structured-record cluster with `ledger-template`, `human-review-queue`, and `state-schema`: one schema per record type (the finding schema here, the round-outcome schema in the ledger template, and the decision-queue item schema in `human-review-queue`), each defined once and referenced rather than re-invented per artifact, with `state-schema` validating and projecting over them.

Design:

- Each reviewer and triager in a round writes its findings to its own file under `docs/plans/<task>.reviews/`, so parallel writers never contend for one file (parallel-safe). The filenames follow one authoritative convention stated in `pack/AGENTS.md` (reviewer files `<step>-<role>-<disambiguator>.md`, with the orchestrator assigning each spawned reviewer a distinct disambiguator so same-role reviewers cannot collide; the triager `<step>-triage.md`; the backstop re-check triager `<step>-triage-recheck.md`), and the role prompts write to the path the orchestrator assigns rather than each inventing a divergent form (Principle 1, one source of truth; Principle 5, no representable colliding-path state).
- Other agents read those files directly. The separate triager reads the reviewers' files; the orchestrator references a finding by its file path in the ledger instead of copying its text. This removes the lossy transcription hop that dropped finding A-M5 in this very task when findings were hand-merged (Principle 1: the findings file is the single source for a finding, referenced not copied).
- The orchestrator owns a cleanup step that deletes the findings files once the round is fully resolved (or at task close). Per the commit-before-delete rule in `file-safety-rules`, the files are committed at least once and then removed as a committed deletion, so the review record survives in git history and the removal is recoverable. This resolves the earlier committed-versus-transient question toward committed.

Validate by reading the roles against each other and confirming a finding flows reviewer-file -> triager-read -> ledger-reference with no orchestrator transcription, and that the cleanup commits before deleting.
