### `no-wrap-convention`: Prose line-wrapping is not policed

Decided (`Q-22`, by the human). The human prefers no manual hard-wrapping of prose (soft-wrap); line length is never a review finding and agents must not reflow or police it. Two parts:

- Convention (workflow-doc change, always applicable): add to `pack/AGENTS.md` a short convention that prose is not hard-wrapped and line length is never a review finding, and fold "line length is never a finding" into `pack/prompts/reviewer.md` and `pack/prompts/triager.md` so reviewers and triagers do not raise or act on it.
- Formatter (tooling follow-on). Decided (human, investigation surfaced the sub-choice): the project already runs `prettier` on `*.md` through `nix fmt` (treefmt-nix, see `flake.nix`), by default at `proseWrap: preserve` (it leaves prose line breaks as authored and does not reflow). Adopt `proseWrap: never` via a repo-root `.prettierrc.json`, so the formatter actively unwraps every paragraph to a single soft-wrapped line and wrapping becomes tool-owned rather than a manual concern (the human's original wish). This is preferred over `preserve` (which needs no reflow but leaves the tool passive) because the human wants the tool, not a convention, to own wrapping; the cost, a one-time reflow diff across all markdown and slightly noisier line-based diffs afterwards (word-diff mitigates), is accepted. The evidence PoC confirmed a repo-level `.prettierrc.json { "proseWrap": "never" }` makes treefmt's prettier unwrap prose (tool-native and portable, prettier auto-discovers it even outside Nix). This supersedes the earlier "do not retroactively reflow" caution, which was aimed at wasteful manual agent reflowing; a formatter doing it once and owning it thereafter is the intended automation. Because the repo-wide reflow is a mechanical formatter run touching orchestrator-owned files (the plan and the ledger), the implementer adds the doc convention and the `.prettierrc.json`, and the orchestrator runs the repo-wide `nix fmt` reflow (incidental reformatting is the orchestrator's, per the file-safety rules) as a separate mechanical commit. The `scaffold-self` recipe also gains a trailing `nix fmt` step, because the tool's raw render is not formatter-clean under `proseWrap: never`; this `nix fmt` deliberately formats the whole tree rather than only the generated files (scoping it would duplicate the manifest's dest-path list against Principle 1), which is harmless because treefmt is idempotent and leaves the repo at a stable committed fixed point.

Validate by reading the convention and the reviewer/triager rule and confirming line length is stated once as a non-finding; and, for the formatter, that after the reflow `nix fmt` is idempotent (a second run is a no-op), `just test` / `just clippy` pass, and no structured content (tables, code blocks, the Roadmap table, rendered principles) is corrupted by the reflow.
