### `optional-modules`: Optional modules

Not started; a design-questions pass is needed before implementing (as with `git-url-fetch`). The isolation MECHANISM (container and worktree isolation) is the behavioural module here; `agent-isolation` fixes its adoption order (worktree-first, then agent-box / agent-images containers) and its runtime selection rule.

Goal: package opt-in additions as self-contained modules, each off by default, none complicating the core (Principle 2: adding a module must not change core behaviour when unused). Two kinds are expected:

- Content modules: extra assets. These are expected to live in the pack manifest via a `[[module]]` section, which `pack.toml` parsing already tolerates (unknown keys are ignored, so current packs stay valid). A module groups a set of `[[asset]]` entries (and possibly `[[var]]` entries) under a name the user opts into. When no module is selected, none of its assets are dropped, so the core output is unchanged.
- Behavioural modules: tool features rather than dropped files. These change how the tool runs, not what it writes, so they are expected to be tool flags rather than pack content.

Candidate modules (illustrative; the design pass confirms which to build and their exact contents and ownership). Each of these was named loosely in earlier planning; the concrete intent is:

- Diagram prompt pack (content): additional reusable prompts, in the same form as the built-in `.agents/prompts/` files, that guide an agent to produce and keep project diagrams current, for example architecture, sequence, and data-flow diagrams written as Mermaid (a text-based diagram syntax) or ASCII and embedded in the docs. Dropped as extra prompt assets (conventionally under `.agents/prompts/`) when opted in.
- Justfile of recipes (content): a `justfile` dropped at the project root with ready-made task recipes (build, test, lint, format, a `verify` aggregate, and workflow helpers), matching this project's own just-plus-Nix convention. Opt-in so a project that does not use `just` is not burdened; ownership is a working file (create-if-absent), since a project may already have its own `justfile`.
- Language starters (content): per-language starter skeletons, each a self-contained set of assets keyed to a language (for example a Rust starter with a flake, a `Cargo.toml`, and a `src/` layout, and equivalents for other languages), so a greenfield project gets a working toolchain alongside the agent workflow. Each starter is its own module, opted into by language.
- Container and worktree isolation (behavioural): run an agent's implementation phase in an isolated environment (a git worktree and/or a container) so its changes are sandboxed and easy to review or discard rather than applied straight to the working tree. Because this is behaviour, it is a tool flag, not dropped content, and it would integrate the separate agent-box (`https://github.com/0xferrous/agent-box`) and agent-images (`https://github.com/nothingnesses/agent-images`) projects (separate repositories, not part of this one) rather than reimplement them.

Open sub-questions to resolve in the design pass (record them in Open Questions, with approaches, trade-offs, a recommendation, and reasoning judged against the Project Principles, before implementing):

- The `[[module]]` schema: how a module names itself and references its assets/variables (for example an inline group, or asset entries tagged with a `module = "<name>"` key), and how the loader includes only opted-in modules while the unnamed core assets always load.
- The selection mechanism: a `--module <name>` flag (repeatable), a manifest default-on/off per module, and whether the interactive TUI gains a module pane or mode (`selection-ui` deliberately scoped the TUI to principles and left room to reuse its interaction code for modules).
- Behavioural-module scope: what container/worktree isolation actually does here, whether it belongs in this tool at all or stays a separate concern the scaffolded assets point to, and how it integrates agent-box/agent-images without a heavy dependency.
- Interaction with `--template`: whether external packs may define their own modules (they should, since the manifest is the shared contract), and how `--module` validates against the active pack.

Each resulting sub-step must be evidence-grounded: validate with a proof-of-concept (build plus tests plus, where relevant, a functional run); on failure fall back to the recorded next-best; if exhausted, raise the impasse. A guiding invariant for validation: with no module selected, the scaffold output must be byte-identical to today's core output.

Design-pass decisions (2026-07-15/16, human-decided; `Q-25`). The step is RESHAPED from the illustrative modules above (diagram pack, justfile, language starters) to the two concrete modules the human wants, on top of the `--module` machinery; the illustrative ones remain possible future modules but are not the priority. Build in three reviewed increments, each its own review artifact: (1) the `--module` machinery; (2) the deterministic-checks module; (3) the isolation module.

- Module schema (HYBRID): assets self-declare membership with an optional `module = "<name>"` field on `[[asset]]` (and `[[var]]`) entries (untagged = core, always dropped; tagged = dropped only when `--module <name>` is passed), PLUS a lightweight `[[module]]` section declaring each known module's `name` and `description` (metadata, and the authoritative set that `--module` validates against; an unknown `--module` errors and writes nothing). Chosen over a group section that lists a module's assets (which duplicates the asset list and drifts, Principle 1) and over tag-only (no home for a description or a validation list): membership is single-sourced on the assets (no drift), metadata and the validated module set live in the small `[[module]]` section (Principle 1, Principle 2). The manifest parser already tolerates unknown keys, so it is non-breaking. Selection is a repeatable `--module <name>` flag under the `scaffold` verb; the loader drops core assets always and a module's assets only when it is selected, so with no module selected the output is byte-identical. A TUI module pane is deferred (the selector stays principle-only).

- Increment (2), the deterministic-checks module (`Q-25`): a module that scaffolds a user-configurable, pluggable deterministic verification gate over the workflow's CODE OUTPUT, generalising the hardcoded verification convention (`clippy` / `nix fmt` / ASCII-clean). It scaffolds (a) `ast-grep` rule config as the first-party linter plus a documented way to register other check and format commands (so a language with no ast-grep grammar plugs in `ruff` / `eslint` / `clippy` / etc.), (b) a `checks-reviewer` role prompt, and (c) workflow guidance wiring it in. Execution model (human-decided): FORMATTERS auto-apply during the IMPLEMENTER's verify step (a writer action, generalising today's `nix fmt`); the CHECKS-REVIEWER, spawned as one reviewer in the work-review phase and read-only, runs the LINTERS (detect) AND the formatters in CHECK / dry-run mode (verifying the code is actually formatted without applying, so it catches an implementer that did not format), and writes its findings to a findings file per `findings-files` (`Q-14`); the TRIAGER adjudicates those deterministic findings alongside the LLM reviewers' (fix vs suppress/accept), and the implementer fixes the valid ones. Settled config layout (`Q-26` + the checks-config design pass, human-decided 2026-07-16; design record in `docs/plans/checks-module-config.explorations/checks-config-{A,B}.md`). The module owns the whole schema:
  - `.agents/checks.toml` is a create-if-absent user working file of flat `[[check]]` entries. Each entry carries `name` (a unique id), `kind` (a closed enum `lint` / `format` / `test` / `mutation`), `command` (the primary command), an optional `check` (a non-mutating secondary command whose meaning is by kind: a format dry-run, or a test compile-only run), optional `paths` globs to scope the check, and optional `budget` / `threshold` (used by `mutation`). The checks module OWNS this schema and SHIPS `test` and `mutation` as COMMENTED example rows, reserving room for the later test modules without turning them on.
  - The `{{modules}}` render slot is the FIRST increment-2 deliverable: a reserved, tool-computed builtin variable in core `AGENTS.md` mirroring `{{instrument}}` (a pack may neither declare nor `--var`-set it), placed at the TAIL, that concatenates each enabled module's guidance partial (a `guidance` partial declared per `[[module]]`); with no module enabled it renders empty so the core output stays byte-identical. This slot also carries the `requires` field on `[[module]]` that the test modules use to auto-enable their dependency (checks).
  - A new read-only `.agents/prompts/checks-reviewer.md` role is spawned as one reviewer in the work-review phase; it runs the linters (detect) and the formatters in check / dry-run mode, and writes tool-evidenced findings to `docs/plans/<task>.reviews/` for the one triager to adjudicate alongside the LLM reviewers' findings (per `findings-files`). FORMATTERS auto-apply in the IMPLEMENTER's verify step, scoped to the files it changed (generalising today's `nix fmt`).
  - The hook path is BUILT NOW (human-decided): an `agent-scaffold checks [--staged]` subcommand that reads `.agents/checks.toml` and runs the lint / format-check commands (fitting the existing `validate` / `status` CLI family), plus an opt-in `--with-precommit-hook` that scaffolds a git pre-commit hook (create-if-absent, never clobber; lint / format-check only; a mechanical backstop SECONDARY to the review-integrated path). Execution is ISOLATED (`Q-39`, human-decided 2026-07-18): the subcommand runs the checks in a SEPARATE, TEMPORARY git worktree removed when the run finishes (including on error), never the live tree, so an in-place format check (treefmt has no dry-run) cannot clobber the user's tree or unstaged work. A plain run isolates the working-tree state; `--staged` isolates the index content so the hook checks exactly what will be committed. Slice 2c into: 2c-i the `checks` subcommand with worktree-isolated execution (the runner), and 2c-ii `--staged` plus `--with-precommit-hook`.

  Rationale: reuses the role-separated + findings-file pipeline (no new machinery), keeps the orchestrator's role clean (it spawns the checks-reviewer like any reviewer rather than producing findings itself), and makes both linting and formatting deterministically-verified gates, the same "do not trust the LLM, verify with a tool" ethos as the metrics validator and the schema drift guards (Principle 6). Chosen over reusing justfile recipes (depends on `just`, couples to that module) and over ast-grep's native `sgconfig.yml` plus guidance (less pluggable for non-ast-grep tools).

- Increment (3), the isolation module (`Q-18` mechanism; content-module scope, human-decided): a CONTENT module that scaffolds POINTER SETUP integrating the external agent-box (`https://github.com/0xferrous/agent-box`) and agent-images (`https://github.com/nothingnesses/agent-images`) projects for writer-agent isolation, transactional changes, and sandboxing, rather than reimplementing isolation or taking a heavy runtime dependency (the tool stays a scaffolder; the isolation RULE already ships via `agent-isolation`). Opted into with `--module isolation`, it drops the config/scripts that wire a project to worktree/container isolation (worktree-first per `agent-isolation`). Its exact integration shape (which files, how they invoke agent-box/agent-images) is settled in its own design pass when increment (3) is reached.

Settled file shape (`Q-41`, human-decided 2026-07-18 after a two-explorer design pass; design record `docs/plans/optional-modules-inc3.explorations/isolation-module-shape-{A,B}.md`): GUIDANCE-ONLY. The `--module isolation` bundle is a single `[[module]] isolation` entry in `pack/pack.toml` (`name = "isolation"`, a `description`, `guidance = "isolation-guidance.md"`, no `requires`, orthogonal to the checks module) plus one pack-source guidance partial `pack/isolation-guidance.md` concatenated into the existing `{{modules}}` render slot when the module is enabled. It drops ZERO files into the project tree (no `[[asset]]` or `[[var]]` rows tagged `module = "isolation"`), so with no module selected the scaffold output stays byte-identical. The partial points at the `agent-isolation` Writer-isolation rule for the tier policy rather than restating it, and supplies only the concrete container-tier wiring the rule resolves to: install agent-box (`https://github.com/0xferrous/agent-box`); build/load an agent-images image (`https://github.com/nothingnesses/agent-images`) via `nix build .#<agent>` then `podman load < result` (image `localhost/agent-images/<agent>:latest`); create the GLOBAL `~/.agent-box.toml` (stressing it is a per-user file, not a project file, which is why the module scaffolds no config); run writers under the sandboxed worktree mode `ab new <repo> -s <session> --git` then `ab spawn -s <session> --git` (with `ab spawn --local` noted as the un-sandboxed mount); and a preflight container-tier detection cue (`which ab` + the configured image present). The partial is placeholder-free (plain prose + fenced blocks, like `pack/checks-guidance.md`) so a stray `{{...}}` cannot be re-substituted, ASCII-clean, not hard-wrapped. Declined (per the design pass, all in the YAGNI boundary): a dropped `.agents/isolation/README.md` project record (the one project-specific fact, image choice, is thin and a template risks an empty stub or duplicating the partial, Principle 16), a sample `.agent-box.toml` (agent-box config is a per-user home file, not a project artifact), a helper/wrapper script or justfile recipe (that is the deferred container-wrapper shim), a Nix `mkAgentImage` snippet, an `agent-scaffold isolation` subcommand or any runtime dependency, and any `requires = ["checks"]`. Verification: `scaffold --module isolation` renders the partial into `AGENTS.md`'s `{{modules}}` block, and with no module selected the scaffold is byte-identical to core (the existing byte-identity + asset-list invariants pin this).

Increment (3) STAYS pointer-scaffolding for now (`Q-33` / `Q-36`, human-decided after the two-explorer cross-harness design pass; design record in `docs/plans/cross-harness-isolation.explorations/q33-q36-{A,B}.md`). Deferred growth path, recorded here so it is not re-explored from scratch: a `.agents/spawn.toml` role->launch-command map (for example `reviewer -> open-code --model <m>`, `triager -> <other-cli>`) plus an agent-box container-wrapper shim that would let the orchestrator actually launch cross-environment writer agents. Both are DEFERRED pending a proof-of-concept and a validated multi-CLI user (Principle 6): the harness owns spawning, cross-environment agents are reachable only via a Bash shell-out launch path (never the native Agent tool), a static launch config is a staleness liability, and the orchestrator LLM can already discover available CLIs at preflight (`session-preflight`). The always-on diversity DOCS rule that this pair also produced ships now via `reviewer-diversity`, independent of this deferred integration.
