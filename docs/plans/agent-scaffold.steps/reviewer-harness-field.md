### `reviewer-harness-field`: Record the reviewer's harness in the round-log attribution (`Q-37`)

Not started. Decided (`Q-37`, human, after the `reviewer-diversity` interim metrics read, 2026-07-17). Add an OPTIONAL `harness` string to each element of the `round` record's `reviewers[]` attribution array in `docs/metrics/workflow.jsonl`, so the calibration log records WHICH harness (CLI) a reviewer ran on, not just its model. Motivation: `reviewer-diversity` generalised the diversity rule to "models OR harnesses", but the metrics schema's per-reviewer object carries only `{role, model, raw_findings, valid_findings}`, so a future cross-harness proof-of-concept could not be measured; the interim read of the 66-record log surfaced this gap. Scope: a one-field, non-breaking addition, validated as a string WHEN PRESENT (the 20 existing attributed records omit it and must still validate; the `reviewers[]` array stays optional and its four existing per-reviewer fields stay required; unknown extra fields are already permitted, so the point is to make `harness` a first-class, validated, documented field rather than an ignored unknown). Touches `src/metrics.rs` (`require_reviewers` gains the optional `harness` string check plus its doc comment), `pack/instrument.md` (document the optional `harness` field in the `reviewers` array description, so the code-vs-prose drift guard covers it), the drift-guard test (assert `harness` appears backtick-wrapped in the prose), and validator tests (a present string `harness` accepted; a non-string `harness` reported). LOW-risk (additive, off the scaffold path, tested). NOT in scope, deferred to `workflow-calibration`: a unique/marginal-valid count that would let the log answer "how much of the second reviewer's yield was unique" (needs a design pass on per-finding identity versus a derived count, so it is not bundled into this mechanical field addition).
