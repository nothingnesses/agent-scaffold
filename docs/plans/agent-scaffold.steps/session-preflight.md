Scope addition (human-decided, 2026-07-16): also document the WORKTREE LIFECYCLE and MERGE-BACK model in the same isolation-operations area of `pack/AGENTS.md`, since the isolation rule established the tiers but never said who merges an isolated writer's work back or who checks it. Decided model, review-in-worktree, merge-on-convergence: the orchestrator owns the worktree lifecycle (create the worktree, spawn the isolated implementer there, spawn the reviewers and separate triager against the worktree branch, drive fix rounds via isolated implementers in that worktree, and merge the branch into the main tree ONLY once the step converges, then remove the worktree), so main only ever carries reviewed, converged steps. The merge distinguishes authoring from integration (human-decided, 2026-07-16): a CLEAN merge (fast-forward or a conflict-free replay of the reviewed commits) authors nothing new, so the orchestrator performs it directly as integration (the same category as committing the plan and ledger), verifying only mechanical correctness (the merged diff is exactly the converged work; tests pass on main after the merge). A merge that needs AUTHORED conflict resolution counts as implementation, not integration (the resolution is content no reviewer has seen), so it is done by an isolated implementer in the worktree and reviewed, after which the orchestrator fast-forwards the now-clean branch. Correctness-according-to-the-plan is established by the review loop and the acceptance review, with `validate --workflow` (once built) as the deterministic backstop that the reviewed rounds happened. Note a merge targets main by definition, so it is intrinsically an un-isolated write; the axis that governs it is authoring-vs-integration, not isolation. This step is being built by the first worktree-isolated implementer, dogfooding the very model it documents.
