### `test-driven`: Test-driven process module (`Q-30`)

Not started; after `optional-modules` increment 2 (dependency-forced: it composes with the checks module). Decided (`Q-30`, human, after a two-independent-explorer design pass; design record in `docs/plans/test-modules.explorations/test-driven-and-mutation-{A,B}.md`). An OPTIONAL module that expands the planner's role with a per-step test spec (cases, inputs/expected outputs, properties, end-to-end flows) and inserts a test-authoring phase before implementation (tests written to fail first, reviewed and triaged, then code implemented until they pass), so an implementation is deterministically checked against the plan, the same "verify, don't trust the LLM" ethos as the metrics validator and the checks module.

Decided design:

- Dependency: the module HARD-DEPENDS on the checks module and auto-enables it via the `requires` field on `[[module]]` (the field the `{{modules}}` slot carries, per `optional-modules` increment 2).
- Roles: a SEPARATE `test-author` writer role (distinct from the implementer), so the test stays an independent spec of the plan's intent rather than a self-graded artifact (role separation, no agent grades its own work).
- Execution single-sourced: test execution is a `kind = "test"` entry in the checks-owned `.agents/checks.toml` (a run command plus a compile-only `check` command), so it composes with the deterministic-checks module rather than duplicating a runner.
- Two artifacts, two gate profiles (the phase-4 split): the TEST artifact's red profile = compiles + fails by assertion + covers the plan's test spec + non-vacuous; the CODE artifact's green profile = compiles + lints clean + all pre-authored tests pass. Red/green is a PHASE PARAMETER handed to the checks-reviewer, NOT a `checks.toml` field (the config stays declarative; the phase says which profile applies). Interface stubs are folded into the test artifact so the tests compile before the code exists. A frozen-tests git-diff tripwire flags any change to the authored tests during the green phase (so the code is made to pass the tests, not the tests weakened to pass the code).
- Risk-scaling: opt-in and matched to risk class, reusing the existing convergence risk classification (collapse the ceremony for low-risk steps), since the two-artifact split roughly doubles the reviewed-artifact count. Two new Roadmap steps (`test-driven` then `mutation`) land AFTER `optional-modules` increment 2 (dependency-forced order). Related: `mutation` (`Q-31`).
