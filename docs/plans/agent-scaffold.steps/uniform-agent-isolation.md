### `uniform-agent-isolation`: every spawned agent runs worktree-isolated; the orchestrator batch-merges parallel outputs (`Q-61`)

Encode the Q-61 decision in `pack/AGENTS.md` and the emitted prompts: every spawned agent runs in the strongest isolation the harness supports (the existing capability-tiered tier order is unchanged), the writers and the read-only reviewers, triager, and explorers alike, and the orchestrator merges each agent's output onto main. Reviewers and the triager no longer write their findings directly into main's `docs/plans/<task>.reviews/`; they write them in their own worktree, and the orchestrator merges the findings onto main, which preserves the cross-tree-readability and survival property that formerly motivated writing straight to main. When multiple agents run in parallel (a round's reviewers, or a set of explorers), the orchestrator batch-merges their worktrees onto main only after they all finish, never as each one completes; a single non-parallel agent is merged after it finishes or converges. The tool's emitted instructions state this, not only the guidance prose: `AGENTS.md` and the orchestrator, reviewer, and triager prompts all carry the uniform rule.

Why it exists: the previous design isolated only the writers and let the read-only reviewers and triager write findings directly into main, on the theory that a read-only agent has no blast radius. But even a findings file or an exploration file is a write, so a killed or misbehaving reviewer could still leave partial or stray state on main. Isolating every spawned agent keeps main pristine until the orchestrator integrates, so no spawned agent can touch main directly. The batch-merge refinement (parallel agents merged only after they all finish) maximises that isolation: no interleaved partial state from one agent reaches main while a sibling in the same round is still running. The decision (human, 2026-07-23) also resolves a standing inconsistency in how explorers were classified: the Design explorations section had called them writer agents while the worktree-lifecycle rule listed them among read-only agents needing no worktree. The uniform rule removes the contradiction: every spawned agent isolates regardless of classification, and explorers are read-only with respect to the reviewed product (they author only advisory design notes, not the plan or the code), consistent with the reviewers and the triager.

Judged against the plan's principles: P1 (prefer the cleaner long-term architecture) is served by collapsing two isolation rules (writers isolate, read-only agents do not) into one uniform rule that every spawned agent isolates, removing the carve-out that made the guidance state who-isolates twice and left the explorer classification contradictory. P5 (make illegal states unrepresentable) is the other driver: with every spawned agent isolated and the orchestrator the only writer to main, "a spawned agent writes to main directly" is no longer a reachable state, so a killed or misbehaving agent cannot corrupt main by construction rather than being guarded against after the fact.

This reconciles with the existing file-safety baseline and the worktree-lifecycle and merge-back rule rather than contradicting them. The always-on file-safety discipline still holds as the fallback when the harness offers no isolation; the tier order (container, else worktree, else that fallback) is the single source and is unchanged, this step changes only who runs under it. The authoring-versus-integration line is unchanged: a conflict-free merge of distinct findings files is orchestrator integration, and only an authored conflict resolution is implementation.
