### `user-prompts-dir`: Separate user prompts from role prompts

Decided (`Q-16`, confirmed by the human). Create a `.agents/user-prompts/` directory for human-invoked prompts (the kickoff prompt and the `compaction-prep` prompt); `.agents/prompts/` stays the role prompts (orchestrator, planner, reviewer, triager, implementer). Human prompts and role prompts are different in kind: a human copies and pastes a user prompt to drive the workflow, whereas a role prompt is handed to a spawned agent.

This REVISES `human-onboarding`: the kickoff prompt moves from an inline block in `pack/AGENTS.md` into `.agents/user-prompts/`, and the `pack/AGENTS.md` "Getting started" section becomes a thin trigger that points to the directory. The user prompts stay thin trigger-prompts so no workflow content is duplicated between them and the canonical `pack/AGENTS.md` (Principle 1). Adding pack assets also touches `pack/pack.toml` (new `[[asset]]` entries for the user-prompts directory) and its expected-asset test in `src/` (the built-in asset-list test), and the README layout description of the scaffolded tree. Validate by proof-of-concept: the new assets drop through the existing loader, the manifest asset-list test is updated and passes, and the README layout names the new directory. (The "Getting started" section that points to `.agents/user-prompts/` is validated by `human-onboarding`, which owns that section; this step only creates the directory the section points to.)
