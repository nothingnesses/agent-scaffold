### `workflow-driver`: the staged workflow DRIVER, the forward member of a projection family over the workflow (`Q-51`)

Not started; the build-start is a separate pending human decision, so this umbrella step stays `not-started` with no increments declared yet (the per-stage `[[step.increment]]` declarations land at each stage's close-out, mirroring `optional-modules` and `structured-skeleton`). Design shape DECIDED (`Q-51`, human, 2026-07-19): the human chose the FULL DRIVER as the target (over the four round-1 explorers' advisory-MVP-only recommendation), staged so no increment hardens a still-evolving workflow. Design record: `docs/plans/mealy-workflow-driver.explorations/r2-synthesis.md` (the authoritative full-driver design + staged build plan), backed by the four `r2-*.md` lens files (`r2-requirements-scope.md`, `r2-user-flows.md`, `r2-design-space-vision.md`, `r2-architecture-build-path.md`) and the round-1 `synthesis.md`. This is a MAJOR initiative, larger than `structured-skeleton`; classify each stage's risk at its own loop open (the risk class in parentheses below is the design-pass estimate).

North star (the projection-family framing). The driver is the FORWARD member of a family of folds over two durable sources: the append-only event log `docs/metrics/workflow.jsonl` and a data-driven control spec `.agents/workflow.toml`. The family: `validate --workflow` is the BACKWARD check (did the past obey the rules), `status` is the SUMMARY, `next` is the FORWARD execution (given the state, what the rules dictate next), and `render` extends the family to prose (it generates the AGENTS.md control fragments from the same spec, so process logic and process prose cannot drift). So the driver is not a new subsystem bolted on; it is the missing forward member of a family whose backward and summary members already ship, and it reuses the W3/W4/W5 reconstruction wholesale (the convergence arithmetic `src/workflow.rs` already runs to CHECK a completed history, run forward to EMIT the next move).

The data-driven-constants spec plus the Rust transition function. `.agents/workflow.toml` is a tool-owned, pack-shipped reference asset (the same for every scaffolded project; `agent-scaffold`'s own repo dogfoods its scaffolded copy) that holds ONLY the deterministic CONTROL CONSTANTS and orderings: the risk-class -> required-clean-streak map, the total-round cap, the backstop severity threshold, the phase/loop sequence, the isolation-tier preference order, the role -> prompt-path map, and the findings/ledger/exploration/report path templates. The transition FUNCTION stays Rust (the streak/cap arithmetic the W3 checker already computes, run forward). The split is deliberate: a workflow VALUE change (a constant, a threshold, an added ordering) is a data edit reviewable like a plan edit; only a genuinely new rule KIND is a code change to the interpreter. This is the answer to "keep it data-driven so the workflow evolves without pouring concrete": the parts that change often are data, the fixed interpreter is small code. Both `validate --workflow` and the driver parse one `WorkflowSpec`, which single-sources today's hardcoded `RiskClass::required_streak` (currently duplicated against AGENTS.md prose). A built-in default (`WorkflowSpec::builtin()` equal to today's constants) covers a project shipping no spec, pinned by a `builtin()`-equals-old-constants assertion so an un-migrated project validates byte-for-byte unchanged.

The control-vs-judgment boundary, enforced by types. The tool owns CONTROL (counting streaks, sequencing phases, scheduling the ready frontier, gating the backstop, resolving the isolation tier, emitting prompts); it CONSUMES agent/human JUDGMENT as typed `Input` variants (a triage verdict `{clean | new_valid}`, a recheck result, a human decision, a commit, an override) and NEVER manufactures a verdict. The typed engine (Stage 2) makes this load-bearing: `step` has no branch that produces an `outcome`, a `risk_class`, or a `human_decision`, so the type system enforces the boundary. Getting this wrong in the permissive direction hallucinates decisions; in the restrictive direction it straitjackets the agent. This is the primary correctness property.

Terminal authority (ORQ-1, decided) and the other open-requirements resolutions. The driver's END STATE is ADVISORY + validated event-RECORDING, NON-BLOCKING in-loop, with an OPT-IN boundary gate (`validate --workflow --strict` in CI or a pre-commit hook, reusing the existing checks/hook pattern) for hard enforcement at the LANDING boundary. So the loop never deadlocks if the tool is skipped, but the boundary gate catches an unrecorded gap before work lands; this also answers ORQ-2 (gate at the boundary, not per-step, keeping the loop robust, harness-agnostic, least-authority, and respecting Q-24's no-runtime-write-path-dependency stance). ORQ-3 (multi-repo) defaults to single-project (a far-stage YAGNI). ORQ-4 (spec-change ceremony): `workflow.toml` is load-bearing config, so edits go through the normal plan + review like AGENTS.md changes, no new ceremony. ORQ-5 (reminder cadence): light, each `next` surfaces only the reminders relevant to the current action, not a per-call dump that trains agents to ignore it.

Complements, does not subsume. The driver COMPLEMENTS (does not replace) the re-grounding + provenance step and the `Q-50` doc-currency guidance: they address different drift axes (the driver fights CONTROL-drift, an agent forgetting a rule that applies right here; re-grounding fights CONTEXT-drift; Q-50 fights DOC-TRUTH-drift). Q-50 is in fact a PREREQUISITE if the driver emits prompts assembled from the role-prompt files (a driver is only as correct as those files). Those cheaper items un-gate and ship on their own schedule rather than waiting on this.

Staged increments (each a reviewable increment; classify risk at loop open). The committed near-term path is 0a -> 0b -> 1 -> 2, which realises the driver's core (the data-driven spec, the advisory surface, the typed engine) while keeping the workflow evolvable as data; Stage 3+ stays behind evidence gates.

- Stage 0a (RISKY, no deps): the `.agents/workflow.toml` control-constants spec + `WorkflowSpec` + `WorkflowSpec::builtin()`; single-source the convergence constants into the W3 check (replace `RiskClass::required_streak`'s hardcoded `match` with a spec lookup, thread the spec into `w3_problems`, add the round cap as `spec.round_cap()`); add a `--workflow-spec` flag on `validate`. Behaviour-preserving, guarded by the existing W3 tests plus the `builtin()`-equals-old-constants assertion. RISKY because it touches the safety-relevant convergence bar and adds a scaffolded asset every downstream project inherits, but it changes no behaviour. VALUE independent of the driver: closes the live `required_streak`/AGENTS.md duplication (a real bug-closing win) and lays the data-driven-spec foundation the whole driver rests on. Deps: none (builds on the shipped `structured-skeleton` + `validate --workflow`). ACCEPTANCE: `validate --workflow` on the live repo is byte-for-byte unchanged with the built-in default; the `builtin()`-equals-old-constants assertion holds; a spec with an altered `required_streak` changes the enforced bar the checker applies (`round_cap` and the backstop severity ship as spec data in Stage 0a but are not W3-enforced until a later stage, so they do not alter the bar yet).
- Stage 0b (RISKY, deps 0a): generate the AGENTS.md control-constant fragment from the spec via a pack `{{workflow_control}}` render slot (the streak-per-class sentence, the cap, the backstop severity, the tier order, the path conventions), byte-compare guarded exactly as the plan's generated status vocabulary already is. This is the PACK generation path (`build_assets` filling the guidance template's slots at scaffold time), not `plan::render`; the hand-authored rationale prose stays around the slot. Proves the code-vs-prose generation closure on the smallest, highest-value content before it is widened. Deps: 0a (the spec). ACCEPTANCE: the generated fragment matches the spec constants; the byte-compare check fails on a hand edit of the generated fragment and on a stale fragment after a spec edit.
- Stage 1 (LOW_RISK, deps 0a): the advisory, read-only, stateless `agent-scaffold next` MVP, largely the already-planned `state-queries` step (`Q-28`/`Q-34`) plus a next-action reminder. For the single active review loop it forward-projects the W3 reconstruction to emit the current per-unit state, the valid transitions, and the next action as a filled instruction prompt (the role prompt path from the spec, the artifact and diff range from the ledger's `## RESUME STATE`, the findings/ledger paths from the spec templates, and the phase-keyed Principle reminders), in both JSON and human text. No typed fleet, no scheduler, no write-path. Stateless: recompute from the durable files each call (no `.fsm-state` store), so it survives a crash or compaction. LOW_RISK (read-only, reuses the shipped parsers). Deps: 0a (reads the spec); coordinate with `state-queries` to avoid duplicating the `status --resume` slice. VALUE: the point-of-action anti-drift reminder that captures most of the drift-reduction value, and the EVIDENCE SOURCE (adherence rate, drift reduction over ~10 step cycles) that gates every later stage. ACCEPTANCE: `next` emits the correct next action and filled prompt for a fixture plan + log + ledger; its forward projection of the streak/cap matches the W3 checker's backward computation on the same files; output is deterministic across machines.
- Stage 2 (RISKY, deps 1): the typed `src/driver/` FSM engine. First EXTRACT the streak arithmetic out of `round_log_consistency_problems`/`w3_problems` into a shared `reconstruct_loop(rounds, spec) -> LoopState` that BOTH the checker and the driver call (one arithmetic, two directions, made literal code reuse), so the driver and W3 provably cannot diverge. Then add `driver/fsm.rs` (a spec-parameterized `ReviewLoop` Mealy machine with a TOTAL `step` transition function, plus nested `StepMachine`/`TaskMachine`), `driver/reconstruct.rs` (the full per-unit fleet from `PlanToml` + the round log + the ledger + the spec), and `driver/emit.rs` (filled instruction prompts from the spec's role/path templates). The nesting makes "a `complete` step whose review loop never converged" unrepresentable by construction (the same invariant W3 checks post-hoc), and the total `step` makes an undefined transition unrepresentable. RISKY (the core logic; must match the checker exactly). Deps: Stage 1 (the advisory surface) and Stage 0a (the spec). ACCEPTANCE: exhaustive transition-table tests (every `(state, input)` pair has a defined image); a DIFFERENTIAL test replaying a round log forward through `step` yields the same streak `round_log_consistency_problems` computes; golden fixtures for the `next` output; every `Output` is reachable only from an `Input` carrying the corresponding judgment (the control-vs-judgment boundary tested).
- Stage 3+ (GATED on evidence): each stage stays behind its own evidence gate so no increment hardens a still-molten workflow. The `blocked_by` ready-frontier scheduler (`driver/schedule.rs`, a pure step-granularity graph function proposing the antichain of ready steps the orchestrator may fan out; GATE: real parallel multi-unit execution actually occurring, since today steps run one at a time). Generation-widening (extend 0b from the constants to the sequencing/paths/tiers fragments; GATE: 0b's constant fragment proven correct and readable). This stage also owes a pluralize-by-count fix to `control_fragment()`, which hardcodes the singular/plural nouns ("clean round", "is 5 rounds") that are grammatical only for the shipped 1/2/5 values; a future data edit (for example `cap = 1`) would render ungrammatical prose (deferred here from the Stage 0b review as finding L-1, non-blocking because 0b only ever renders `builtin()` whose values are fixed and grammatical, and any value edit goes through plan + review). The guarded `record-*` write-path (`driver/record.rs`, transition-validating append subcommands that reject an illegal transition; deliberately reopens `Q-24`'s no-write-path stance and must preserve hand-writable JSONL for a binary-less harness; GATE: measured advisory-adoption evidence from Stage 1). And the opt-in authoritative BOUNDARY gate (`validate --workflow --strict` in CI/pre-commit as the hard-enforcement landing gate per ORQ-1; GATE: a measured-low override rate showing the machine models the workflow well). The far corner (calibration closure, a `workflow-viz` read consuming the driver's JSON, multi-repo) stays out of scope until its own scale is real.

YAGNI boundary (what NOT to build). No persisted FSM-state store and no stateful `drive` daemon/REPL (recompute statelessly each call). No judgment in the spec or in `step` (verdicts, risk classes, and human decisions are only ever `Input` variants). No general/Turing-complete process DSL (the spec encodes THIS workflow's fixed constants/sequences/paths/tiers; a new rule KIND is code). No increment-level DAG scheduling (the DAG is step-granularity only). No generating the RATIONALE/role/contract/discipline prose (only the control-constant fragments are generated; the WHY stays hand-authored). No performing isolation (the tool emits the resolved tier; the orchestrator isolates). No per-turn polling during active work (a negative token trade; scope to resume, boundary, and decision points). No flipping to authoritative-blocking before the override rate is measured low. And the driver never subsumes the re-grounding step or Q-50.
