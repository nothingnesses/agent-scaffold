## Success Criteria

- One command, run from any working directory and targeting any `--output-dir`, drops the minimal core (a usable `AGENTS.md`, a planning-document template, and the core prompts) into both an empty directory and an existing repository. Writes are off unless confirmed: a run previews the plan and only writes with `--write` or the selector's Save confirmation (`--dry-run` previews explicitly), so nothing is written by accident.
- By default it creates only absent files and never modifies existing ones; `--force` is required to overwrite; a default re-run therefore reverts nothing.
- The user can choose which principles and modules to include, interactively with sane defaults or non-interactively via flags or config.
- The dropped assets are immediately usable to run one pass of the workflow (plan -> review -> implement -> review) without further setup.
- Optional modules can be added without touching or complicating the core.
- The tool can scaffold from a user-supplied template pack, not only the built-in one.
- The scaffolded workflow is internally consistent: `pack/AGENTS.md`, the prompts under `pack/prompts/`, the README diagram, and this plan state the same rules the same way, and the convergence accounting (round outcomes and the single escalation cap) is deterministic and computable from the review ledger alone.
- The self-scaffolded copies stay reliably in sync with the pack: editing a `pack/` file and running a plain `cargo build` (no manual `cargo clean`) re-embeds the change, so `scaffold-self` and the golden sync-test regenerate from the current pack, not a stale embedded one.
- The human-interface layer is present: onboarding (a "Getting started, for the human" section that points to the editable kickoff prompt in `.agents/user-prompts/` and states the human's ongoing decision duty) and a single living human-decision queue that the orchestrator updates and pushes to the human at every checkpoint.
- The scaffolded workflow carries the human-input contract (every human-input point, escalation, intake, open questions, and clarifying questions, presents options, trade-offs, a recommendation, and reasoning judged against the project's numbered Principles, scaled to stakes) and the always-separate triager rule (the triager is a separate agent, independent of both the producer and the orchestrator, for every review round).
- The scaffolded workflow carries a git-durability-and-recovery discipline: a clean tree before spawning a writer, commit-before-delete for workflow-managed files, own-files-only formatting (no repo-wide formatters or cross-owner `git checkout`), an orchestrator recovery protocol on any kill or interrupt, and destructive validation in scratch. Writer agents run under capability-tiered isolation (container if available, else worktree, else that discipline), and read-only agents need none.
- Review findings flow through per-agent files that other agents read directly and the ledger references by path, so no finding is lost to orchestrator transcription; the files are committed before the orchestrator deletes them.
- A compaction-prep checkpoint keeps the plan and ledger authoritative for resuming after context loss: before a compaction the orchestrator flushes and commits the plan, the ledger, and the Open-Questions queue and verifies the plan's resume anchor, and on resume state is reconstructed from `AGENTS.md`, the plan, and the ledger.
- Human-invoked user prompts (the kickoff prompt and the compaction-prep prompt) live in their own `.agents/user-prompts/` directory, separate from the role prompts in `.agents/prompts/`, and stay thin triggers that do not duplicate workflow content.
- The workflow offers a human-invokable review entry mode (a fourth entry alongside implement, explore, and question-driven input): a human can review the whole tree or a diff against a named plan's Success Criteria, ad-hoc constraints, or open-ended correctness/quality, and get a committed findings report with a handoff into an implement task, reusing the same reviewers, separate triager, severity scale, and high/critical dismissal recheck as the other review phases.
- Optional verification modules compose on the core without complicating it: a deterministic-checks module (a user-configurable, pluggable lint/format/test/mutation gate over the code, with a read-only checks-reviewer, a `{{modules}}` guidance slot that is empty and byte-identical when no module is enabled, and an optional pre-commit hook), a test-driven module (a separate test-author role and a test-first two-gate profile) that hard-depends on it, and a mutation module that hard-depends on that.
- The scaffolded workflow's reviewer/triager diversity rule spans different models OR harnesses where available, so a user running multiple CLIs is guided to spread reviewers across them to reduce shared blind spots.
- The plan's live skeleton is a structured single source of truth: a `<task>.plan.toml` holds the Roadmap (step status, order, dependencies), the Open-Questions queue (id, status, one-line ask, a `receipt` pointer to the JSONL decision record, typed `folded_into`/`superseded_by` cross-references; the queue's `options`/`chosen` stay in the JSONL decision receipt, not the TOML), the numbered Project Principles, the step<->increment links, and (per Q-46) the exemption waivers nested on their step and the W4 baseline in `[meta]`; the append-only `docs/metrics/workflow.jsonl` holds only genuine events (rounds, escalations, decisions, intakes, dismissals). Typed cross-references replace the parametric status strings, so a `decided` question and a `blocked_by` step are structured fields, not prose conventions, and `validate --source` checks each resolves.
- `agent-scaffold render` generates a committed-but-never-hand-edited `<task>.md` from the TOML skeleton plus opaque Markdown prose sidecars, splicing each sidecar verbatim and never parsing prose back into structure, so there is no prose round-trip and nothing to clobber: render writes only the derived `<task>.md`, and a schema violation, an unresolved cross-reference, or a missing sidecar exits non-zero and writes nothing. A do-not-edit banner heads the generated file and names its real sources, and the derived Status line and the status vocabulary are generated (not hand-maintained), so they cannot drift.
- A `render --check` guard re-renders and compares against the committed `<task>.md`, catching both a hand-edit of the generated file and a stale render after a source edit; it warns locally so a forgotten re-render never blocks an in-flight workflow step, and fails hard in CI and in the pre-commit hook so a stale plan cannot land.
- Enforcement is preserved over the migrated data as a parser swap, not a rewrite: W3, W4, W5, and the pause.md catch keep working, reading `Step`/`Question`/waiver shapes and the baseline from the TOML and the escalation evidence from the JSONL, with per-increment convergence intact, the two evidence tiers un-launderable, and W5's record-backed join to a decision-scoped escalation preserved across the two substrates.
- New `round` and `escalation` records carry a structured step/increment id, and W3 and W5 join on it in preference to the lexical `leading_slug` strip for post-migration records, so the over-strip fragility (SE-10/B6) no longer applies to newly logged increments, while pre-migration records continue to join via the leading-slug shim.
- Migrating this repo is safe and reversible: the skeleton and sidecars are built additively while the hand-authored `.md` stays the live source (`[meta].primary = "markdown"`), validated synthetic-pilot-first, shadow-rendered and fidelity-diffed against the current plan, then cut over in a single `git revert`-able commit that sets `primary = "toml"`, renders, and (per Q-46) moves and prunes the historical JSONL waiver/baseline lines; `validate` stays green at every stage, and reverting the cutover restores both the hand-authored `.md` and the pruned JSONL lines.
- The scaffolded default stays coherent: a fresh `agent-scaffold` run drops a `<task>.plan.toml` skeleton, starter prose sidecar directories, and a rendered `<task>.md` that are internally consistent and `validate`-green, with no `w4_baseline` (so every decision the fresh project makes requires a receipt).
- The shipped pack's prompts and docs describe the same TOML+render+sidecar plan flow the tool actually implements: a freshly scaffolded project's `pack/prompts/*` and `pack/user-prompts/kickoff.md` describe authoring and editing the `.plan.toml` skeleton plus prose sidecars, rendering the generated `<task>.md`, never hand-editing it, and `render --check`ing before commit, with no residual Markdown-only-plan authoring instructions or stale `docs/plans/TEMPLATE.md` references; and the shipped `README.md`, `pack/AGENTS.md`, and `pack/instrument.md` are current with the render architecture (the README documents `render`/`render --check` and `validate --source`, and `instrument.md` documents that a `type:"decision"` record requires the `task` field).
