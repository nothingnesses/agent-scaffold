# Q-33 and Q-36: cross-environment agent spawning (container isolation and cross-model/cross-harness)

Design exploration, explorer A. Question owner: the shared axis behind `Q-33` (running our own spawned writers under agent-box / agent-images container isolation) and `Q-36` (spawning reviewers/triagers/writers on models or harnesses different from the orchestrator's own). Both reduce to one question: what may a SCAFFOLDER contribute to launching cross-environment agents, given the HARNESS owns the actual spawn?

One-line position: agent-scaffold can scaffold the DATA and DOCS that tell an orchestrator how to launch a cross-environment agent (a launch table, an isolation-wrapper pointer, preflight detection, and a workflow rule), but it must never own the spawn itself, which is the harness runtime's job; Q-33 and Q-36 are two dimensions (isolation wrapper, target command) of ONE optional launch-config module plus one always-on docs rule.

## 1. The scaffolder-vs-harness boundary

Draw the line at execution. Three actors, not two:

- The SCAFFOLDER (agent-scaffold): a Rust CLI that drops files (prompts, `AGENTS.md`, plan/ledger templates, config) and then exits. It runs at scaffold time, never at workflow time. It cannot spawn, wrap, or containerize anything, because it is not present when the workflow runs.
- The HARNESS (Claude Code, or another CLI): the runtime the orchestrator lives in. It owns the spawn primitive (Claude Code's Agent tool, with its native `worktree` / `remote` isolation), the process boundary, and the tool interface the orchestrator calls. It also owns whatever shell the orchestrator can reach (Bash).
- The ORCHESTRATOR: an agent running INSIDE the harness. It is the pivot the design turns on. It can spawn sub-agents two ways: through the harness's native Agent tool (which today offers worktree and remote, not local-container, and only the harness's own model), OR by shelling out through Bash to launch any CLI on PATH (`agent-box run -- <cli>`, `open-code --model gpt-5.6`). The second path is how cross-environment agents become reachable at all, and it is driven by the orchestrator reading a config the scaffolder dropped, not by any tool feature.

What the harness owns and the tool cannot own (out of scope for a scaffolder):

- The spawn mechanism. Making the native Agent tool run its sub-agent inside an agent-box container, or launch a gpt-5.6 CLI, is a harness capability. The tool cannot reach inside Claude Code's Agent tool and change how it isolates or which model it runs. Attempting this would reimplement a harness (scope-discipline violation, Principle 2's "adding a module must not complicate the core", and a heavy runtime dependency).
- Process, filesystem, and security sandboxing at runtime. That is agent-box / agent-images' job, invoked by the runtime, not by a file-dropper.
- Guaranteeing any particular CLI or image exists on the machine. The tool scaffolds intent; the environment supplies capability.

What is legitimately scaffoldable (in scope, all just files the runtime reads):

- Workflow RULES in `AGENTS.md` and the role prompts (docs; already how `agent-isolation` and `session-preflight` ship). Harness-agnostic, always-on, no runtime dependency.
- A launch-config table: DATA mapping a role (or the container tier) to a launch command the orchestrator invokes through the harness's Bash/Agent runtime. This is the same category as `principles.toml` or a checks config: a file the workflow reads, not code the tool runs.
- Pointer setup for agent-box / agent-images: a wrapper snippet / invocation string and docs telling the orchestrator how to call them when the resolved tier is container. Pointer, not reimplementation.
- Preflight DETECTION guidance: instructions (already owned by `session-preflight`) for the orchestrator to detect which of the above are actually present and state the resolved set to the human, degrading to the next tier when absent.

The load-bearing insight: "scaffold a launch command the orchestrator invokes" stays on the scaffolder side of the line because the tool only writes the string; the orchestrator, running in the harness, executes it. Option (c)'s truth ("leave actual spawning to the harness") is therefore preserved even when we choose to scaffold config: the config is inert data until the harness runtime acts on it.

## 2. Q-33 recommendation: grow increment 3 from a pure README pointer to a thin, harness-consumable integration, still short of reimplementing isolation

Recommendation: increment 3 stays pointer-scaffolding in the sense that matters (no reimplemented isolation, no heavy runtime dependency, the tool still never spawns), but it grows past a prose-only README to drop ACTIONABLE data the orchestrator or a container-exec-capable harness can consume. Concretely, `--module isolation` drops:

- A launch-config file, `.agents/launch.toml` (shape in section 3), whose `[isolation]` section carries the container-tier wrapper: the agent-box / agent-images invocation string with placeholders (`{prompt}`, `{image}`, `{branch}`, `{worktree}`) the orchestrator fills. This is the concrete form of "container via agent-box / agent-images if wired" that the Writer isolation rule and the preflight already name in the abstract.
- A short adapter/launcher doc (or a stub script under `.agents/`) showing the orchestrator how to compose the wrapper around a writer launch: when the resolved tier per the Writer isolation rule is container, prepend the wrapper to the spawn command instead of using the native Agent tool's worktree mode.
- Workflow guidance connecting three existing pieces: the Writer isolation tier order (`AGENTS.md`), the preflight tier statement (`session-preflight`), and this config. No new rule; the config is what the existing container tier resolves TO when agent-box is present.
- Preflight detection text (extending `session-preflight`, not a new step): detect whether agent-box / agent-images and the target image are present, and state "container tier available via agent-box" or fall back to worktree.

What it does NOT do, stated honestly: it does not make Claude Code's native Agent tool containerize its sub-agents; that path only exists when the orchestrator shells out via Bash to the wrapper, or when a future harness natively supports container-exec and reads this config. On a harness that can do neither, the container row is inert and the tier resolves to worktree/file-safety, exactly as the always-on rule already promises. So the module adds reach where the environment allows and is a no-op where it does not (Principle 2), and it never becomes a runtime dependency (the wrapper is invoked by the runtime, absent-safe).

Why grow it at all rather than stay prose: a scaffolded, parameterized invocation string is strictly more useful than English instructions and stays fully within scaffolder scope (it is a file). It also unifies with Q-36 (section 4), which needs the same table. The evidence bar (Principle 6) is a small proof-of-concept: `--module isolation` drops `.agents/launch.toml`, and with the module unselected the output is byte-identical to core (the module invariant already required by `optional-modules`).

## 3. Q-36 recommendation: (a) in core plus (b) as the same optional module; (c)'s discipline kept

Recommendation: a combination, split by cost and by the harness-agnosticism principle.

(a) ships in CORE, always-on, docs-only. Strengthen the existing model-diversity rule in `pack/prompts/reviewer.md` and `AGENTS.md` from "prefer several reviewers with different lenses, and different models where available" to also cover cross-harness: "different models and, where the harness or environment provides more than one, different harnesses or CLIs (for example one reviewer on the native model, another launched on a different model via a second CLI)". This is a pure docs edit, harness-agnostic, no machinery, no dependency, and it applies on a single-CLI machine too (it just resolves to same-harness model diversity there). It belongs in core, not behind the module, because the guidance costs nothing and should be present even when the config is not. This is Q-36 option (a).

(b) ships as the SAME optional module as Q-33 (do not build a second module; see section 4). The launch-config gains a role -> launch-command mapping so an orchestrator that actually has multiple CLIs installed can launch a reviewer or triager on a different model. This is the mechanically-actionable form of (a). This is Q-36 option (b).

(c) is not chosen as the sole answer (it under-delivers: the tool can legitimately scaffold actionable data, not only prose), but its discipline is kept intact: the config is data the orchestrator reads and executes through the harness runtime; the tool still never spawns.

Config shape (`.agents/launch.toml`, dropped only under the module, ownership `working` so it is create-if-absent and never clobbered, Principle 3):

```toml
# .agents/launch.toml  (dropped only with the isolation/launch module; optional)
# Maps workflow roles and the container isolation tier to launch commands the
# orchestrator invokes THROUGH the harness runtime (Bash or a container-exec-
# capable Agent tool). The tool writes this file; it never runs these commands.
# Absent this file, the orchestrator spawns via the harness's native Agent tool
# and the core workflow is unchanged (Principle 2). Any row whose command is not
# on PATH is skipped at preflight with a stated fallback, so this file never
# REQUIRES a multi-CLI or containerized environment; it only USES what is present.

[isolation]
# Q-33: the container-tier wrapper. When the Writer isolation rule resolves the
# tier to "container" and agent-box is detected at preflight, the orchestrator
# prepends this around a writer launch instead of using native worktree mode.
# Empty or absent -> the tier falls back to worktree, then file-safety.
container_wrapper = "agent-box run --image {image} --"
image = "ghcr.io/nothingnesses/agent-images:rust"

# Q-36: role -> launch command. Optional and additive. The orchestrator MAY use a
# listed command to diversify models for that role. Placeholders {prompt},
# {branch}, {worktree}, {image} are filled by the orchestrator at spawn time.
[[agent]]
role = "reviewer"
command = "open-code --model gpt-5.6 --prompt-file {prompt}"

[[agent]]
role = "reviewer"
command = "pi --model gemini-3-pro --prompt-file {prompt}"

[[agent]]
role = "triager"
command = "open-code --model gpt-5.6 --prompt-file {prompt}"
```

How (b) composes with the isolation tiers. A cross-model agent still owes the Writer isolation rule and the file-safety baseline. Composition is by nesting: the `[[agent]]` command is the inner launch; the `[isolation].container_wrapper` is the outer wrapper the orchestrator prepends when the resolved tier is container. So a cross-model IMPLEMENTER launched via a different CLI also gets container (or, one tier down, worktree) isolation by the same composition. Important simplification: Q-36's highest-value use is diverse REVIEWERS and TRIAGERS, which are READ-ONLY and per the existing rule need no isolation at all (they read the branch under review and write findings to main's `docs/plans/<task>.reviews/`). So the common Q-36 case sidesteps wrapper composition entirely; only cross-model WRITERS (implementers) need the nesting, and they are the rarer, more advanced case.

How (b) composes with session-preflight. No new preflight step. `session-preflight` already "detects and states available isolation tiers/tools"; extend its detection to also enumerate which `[[agent]]` rows are live (which listed CLIs are on PATH, whether agent-box and the image are present) and state the resolved reviewer-diversity plan to the human, disabling unavailable rows with a stated fallback to native same-model spawning. This is where the "assumes a specific multi-CLI environment" risk is neutralized: the file is opt-in and preflight degrades each row independently, so a machine with only Claude Code silently resolves every row to the native spawn and loses nothing.

How the orchestrator invokes it. The orchestrator (in `pack/prompts/orchestrator.md`, extended by the module's guidance) reads `.agents/launch.toml`, and for a role with a live command it launches that command through Bash (filling placeholders and, for a writer, nesting the container wrapper), instead of the native Agent tool; for a role with no live command it uses the native Agent tool as today. The findings-file and worktree-lifecycle rules are unchanged: a cross-model reviewer writes to the same `docs/plans/<task>.reviews/` path; a cross-model implementer runs in the same orchestrator-owned worktree/container.

Risk, stated plainly, and its mitigation. The named risk is that the config makes the tool assume a specific multi-CLI / containerized environment. Mitigation is structural, not incidental: (1) the whole file is behind an opt-in module, off by default, byte-identical core when unused (Principle 2); (2) ownership `working` / create-if-absent, so re-scaffolding never clobbers a user's edited table (Principle 3); (3) every row is independently probed at preflight and falls back when its command is absent, so the file is a menu of opportunities, never a requirement; (4) the always-on rule (a) carries the model-diversity intent even with no config at all. The tool therefore never REQUIRES the environment; it only exploits it when present.

## 4. Unified position: one always-on rule plus one optional module carrying both dimensions

Q-33 and Q-36 are the same shape: "the orchestrator launches an agent in an environment the native Agent tool does not natively provide (a container; a different model/CLI), by invoking a scaffolded launch command through the harness runtime." They differ only in which axis of the launch command varies: Q-33 varies the isolation wrapper, Q-36 varies the target command/model. A single `.agents/launch.toml` row can carry both (a gpt-5.6 implementer inside an agent-box container), which is the proof they are one artifact, not two.

Therefore:

- ONE always-on docs rule in core (Q-36a): the strengthened model/harness-diversity guidance. Ships regardless of the module. Costs nothing, harness-agnostic.
- ONE optional module (the grown increment 3): scaffolds `.agents/launch.toml` (both the `[isolation]` container wrapper for Q-33 and the `[[agent]]` role->command rows for Q-36), the composition/invocation guidance in the orchestrator prompt, and the preflight-detection extension. Do NOT split isolation and cross-model into two modules: they share the same launch-table machinery and the same preflight-detection logic, and two modules for one artifact would duplicate and drift (Principle 1) and needlessly fragment the surface (Principle 2). Name it for the artifact, for example the "agent-launch" (or "launch") module, superseding the narrower "isolation module" label so the name matches its two dimensions.
- The tool still never spawns (option (c)'s discipline): `.agents/launch.toml` is inert data; the orchestrator, in the harness, executes it.

So the coherent answer to "does spawning cross-environment agents belong in this tool": the SPAWN does not; the DATA, DOCS, POINTER, and DETECTION that make an orchestrator able to spawn cross-environment do, as one rule plus one config module. This keeps harness-agnosticism (the rule and the config both degrade to the native path), honors Principle 2 (unused module = byte-identical core), and holds scope discipline (no reimplemented harness, no reimplemented isolation).

## 5. Sequencing relative to `optional-modules` increment 3

- The core rule-strengthening (Q-36a) is a trivial docs edit to `reviewer.md` and `AGENTS.md` and depends on nothing. Land it FIRST, independent of the module and of increment 3. It is the cheapest, most harness-agnostic slice and it makes the model-diversity intent present immediately.
- The launch-config module IS increment 3, with its scope grown from "pointer setup" to "pointer setup plus launch table (both dimensions)". It keeps its place AFTER increment 1 (the `--module` machinery, a hard prerequisite: the module system must exist to drop tagged assets) and increment 2 (the deterministic-checks module). Do not reorder it ahead of increment 1.
- Within increment 3, build the Q-33 `[isolation]` wrapper and the Q-36 `[[agent]]` rows together (they are one file and one preflight change), but validate incrementally per Principle 6: proof-of-concept that `--module <launch>` drops `.agents/launch.toml`, that core stays byte-identical when unselected, and (functional) that the orchestrator can read a row and compose a wrapper. The read-only reviewer case is the cheapest functional demo (no worktree composition needed) and should be the first validated path.
- Dependency summary: Q-36a (core docs) -> now; increment 1 (module machinery) -> unchanged prerequisite; increment 3 (this unified module) -> after increment 2, scope grown as above. `session-preflight` and `agent-isolation` are already complete and are extended in place (detection text; the container tier's concrete form), not rebuilt.

## 6. Rejected alternatives (strongest first)

Strongest rejected: a real native container integration, where the tool builds runtime machinery that wraps the harness's Agent tool so its sub-agents run inside agent-box automatically. Rejected because it makes agent-scaffold a runtime and a harness (reimplementing spawn and isolation), which is the explicit scope-discipline red line and a heavy runtime dependency on agent-box; it also cannot work harness-agnostically, since the tool cannot reach inside Claude Code's Agent tool to containerize its sub-agents (Principle 2, and the scaffolder-vs-harness boundary in section 1). The thin path in section 2 gets most of the reach with none of this.

Two separate modules (an isolation module and a distinct cross-harness module). Rejected because the two share one artifact (the launch table) and one preflight-detection code path; splitting them duplicates that machinery and invites drift (Principle 1) and fragments the module surface for no benefit (Principle 2). A single row proving it can carry both a wrapper and a model command is the direct evidence they are one thing.

Pure option (c) / prose-only pointer README for both. Rejected as the SOLE answer because it under-delivers: a scaffolder may legitimately drop actionable data (a parameterized launch table) that a harness or orchestrator consumes, which is strictly more useful than English and still fully in scope (it is just a file). Its discipline is nonetheless kept: the tool does not spawn; the config is inert until the runtime acts.

Pure option (b) with no core rule. Rejected because gating all model-diversity guidance behind the optional config would leave a single-CLI user (the common case) with no direction at all, when the guidance is free and harness-agnostic. The rule belongs in core; the config is the opportunistic upgrade on top of it.

Baking specific CLIs (open-code, pi) or a fixed image into core defaults or required fields. Rejected: it assumes a specific environment (the named risk) and dates fast. The commands live only in the opt-in, user-editable `.agents/launch.toml` as examples, every row is preflight-probed and absent-safe, and core carries only the harness-agnostic rule.
