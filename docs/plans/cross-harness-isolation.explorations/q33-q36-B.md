# Q-33 + Q-36: cross-environment agent spawning (container isolation and cross-model spawn), explorer B

Question (shared axis): agent-scaffold is a SCAFFOLDER; the HARNESS owns spawning. Q-33 asks whether our own spawned writer agents can genuinely run under agent-box / agent-images container isolation, and whether that is even a scaffolder concern. Q-36 asks whether the orchestrator can spawn agents on different models / harnesses (gpt-5.6 via open-code, gemini via pi) to cut shared blind spots, and if so what THIS tool contributes: (a) a stronger docs-only rule, (b) a scaffolded role -> launch-command config, or (c) leave it to the harness and only document. Both reduce to one question: does cross-environment agent spawning belong in this tool, and if so as what?

Short answer: the two are the SAME mechanism. Container-exec of our own agents (Q-33) and cross-model spawning (Q-36) are both reachable ONLY through one path: the orchestrator launching an external CLI as a sub-agent through the harness's Bash tool, optionally wrapped in a container. The native Agent-tool spawn path is harness-owned and gives whatever isolation tiers that harness offers (worktree / remote), but not local containers and not other models. So this tool's legitimate contribution is a single thing wearing two hats: an optional, preflight-resolved SPAWN MAP (role -> launch command + isolation tier) plus the container-wrapper glue it composes with, scaffolded as content and consumed by the orchestrator PROMPT we already ship. Plus one core docs-only rule generalization that ships regardless.

## 1. The scaffolder-vs-harness boundary, drawn crisply

The tool writes files. It never spawns, execs, or holds a runtime dependency. The actor that reads scaffolded config and runs commands is the orchestrator AGENT (an LLM following `.agents/prompts/orchestrator.md`) through the harness's own tools (Agent, Bash, git). That is the fixed point: anything we "give the orchestrator to do" is legitimate iff it is prose or config the orchestrator reads and acts on with tools the harness already exposes, and iff it degrades cleanly when those tools or targets are absent.

Harness-only (out of scope for this tool, cannot be scaffolded):

- The act of spawning a sub-agent through the harness's native Agent tool, and that path's isolation tiers (git worktree, remote). These are the harness's; we can state a preference over them but not implement or extend them.
- Whether the native Agent tool offers local-container isolation at all. The named tools (worktree / remote) do not; agent-box / agent-images are not reachable through the native Agent tool. We cannot make the native Agent tool containerize a spawn.
- Session lifecycle, streaming, permissions, and whether a Bash-launched CLI is even permitted or installed. All harness- and machine-local facts, discoverable only at runtime.

Scaffoldable (legitimate contributions):

- The isolation RULE and its tier order. Already shipped in `pack/AGENTS.md` ("Writer isolation (capability-tiered)"), harness-agnostic, resolves to the file-safety fallback when nothing better exists. This is the correct precedent and the model everything else follows.
- Prompts / docs: the orchestrator's duty to prefer diverse models for read-only roles, and to run writers under the strongest available isolation. Docs-only, always applies.
- Pointer setup for agent-box / agent-images: dropped config / scripts that make the external tools installable and invocable from this project, without vendoring or depending on them (increment 3's existing scope).
- An OPTIONAL spawn map: a config the orchestrator reads that names, per role, how to launch it (which model / CLI) and at which isolation tier. This is the one new artifact, and it is the shared vehicle for both Q-33 and Q-36.
- Preflight detection guidance: the orchestrator resolving, at session start, which of the mapped targets and tiers are actually present (CLI on PATH, container tool available, native Agent tool tiers), and stating the resolved plan to the human, exactly as the existing Preflight rule already resolves the isolation tier.

The load-bearing observation: a scaffolded launch-command config is NOT "dead config a harness never reads", because its consumer is the orchestrator PROMPT this tool authors, not a harness feature we cannot guarantee. We control the reader. What we do not control is whether the orchestrator can ACT on a given entry, and that is handled the same way the isolation rule already handles an absent tier: preflight resolves it to what is available and degrades.

## 2. Q-33 recommendation: pointer-scaffolding at the core, plus a thin optional wrapper on the launch path; container-exec is honestly reachable only off the native Agent tool

Is container-exec of our own spawned agents reachable when the harness controls spawning? Precisely:

- Through the native Agent tool: NO. Its isolation tiers are worktree / remote; it does not run the spawned agent inside an agent-box / agent-images container. Nothing this tool scaffolds changes that; it is the harness's spawn path.
- Through a Bash launch command: YES, but only because on that path the ORCHESTRATOR is doing the spawning (it shells out), so it can wrap the launched CLI in `agent-box run ...`. This works only where the harness exposes Bash and the container tool is installed on the machine.

So container isolation of our own agents is a real capability, but it lives entirely on the launch-command path, which is the same path Q-36 (b) proposes. It is not something the isolation module can bolt onto native spawning.

Recommendation for increment 3: keep pointer-scaffolding as the base, and add a thin, optional wrapper the spawn map composes with. Concretely the isolation module (`--module isolation`) drops:

- Pointer setup: a `.agents/isolation/README.md` pointing at agent-box (`https://github.com/0xferrous/agent-box`) and agent-images (`https://github.com/nothingnesses/agent-images`) with install / enable steps, and (where the project uses Nix) an optional commented flake input or devshell note, create-if-absent. This is the already-decided content scope and it stands on its own for users who wire isolation by hand.
- A portable wrapper script, `.agents/isolation/agent-box-run.sh` (a POSIX shell shim: takes a command and the writer's cwd / worktree, runs it under agent-box with the right mounts, and is a no-op passthrough if agent-box is absent). This is the glue the orchestrator invokes when a spawn map entry says `isolation = "container"`. It is plain scaffolded content, not tool runtime.

This is NOT the tool "orchestrating containers". The tool still only writes a script and a pointer doc; the orchestrator (LLM) decides when to call the script, through Bash. The module takes no build-time or run-time dependency on agent-box; with the module unselected the output is byte-identical (Principle 2). It is honest about the boundary: the module makes container isolation reachable on the launch path and says so, rather than pretending it containerizes native spawns.

Rejected for Q-33: a "real integration" where the Rust CLI itself invokes agent-box to run agents. That crosses the scaffolder line into orchestrator-runtime, adds a heavy external dependency, and duplicates a harness (Principle 1 coherence, Principle 2 minimal core, and the standing scope discipline). The design already records that the tool "stays a scaffolder"; this keeps it there.

## 3. Q-36 recommendation: (a) + (b), never pure (c); the spawn map is the concrete contribution

Choose (a) AND (b). Reject pure (c).

(a), the rule, ships in CORE (harness-agnostic, docs-only, always applies). Generalize the existing reviewer rule from "different models where available" to "different models OR harnesses where available", and extend the same instruction to the triager, the re-check triager, and explorers (all read-only, so the highest-value and lowest-risk place for model diversity, since they need no isolation). `pack/AGENTS.md` line 21 already carries 80 percent of this; the edit is small. This is the floor: even a single-Claude user loses nothing, and a multi-CLI user gets told to diversify.

(b), the spawn map, ships in the OPTIONAL isolation module as the vehicle that makes (a) actionable and makes Q-33's container path real. The rule alone tells the orchestrator to diversify but gives it no way to know WHICH command launches gpt-5.6 with this project's flags, model routing, or container mounts. The map pins that.

Config format and location (opt-in, create-if-absent, user-owned working file):

```
# .agents/spawn.toml -- OPTIONAL; scaffolded by --module isolation, create-if-absent.
# Maps a workflow ROLE to how the orchestrator launches it as a sub-agent.
# Read by the orchestrator at PREFLIGHT and resolved against what is actually
# available (CLI on PATH, container tool present, native Agent-tool tiers).
# Unresolvable entries degrade: read-only roles fall back to native spawning;
# writer roles fall back to native spawning under the harness isolation tier,
# never to an un-isolated writer. Empty / absent file == native spawning only.

[default]
# how to spawn when no role entry matches; empty means use the harness's native
# Agent tool, whose own isolation tiers then apply.

[[role]]
name = "reviewer"                 # read-only: model diversity, no isolation needed
launch = [
  { command = "open-code --model gpt-5.6", isolation = "none" },
  { command = "pi --model gemini-3-pro",   isolation = "none" },
]

[[role]]
name = "triager"                  # read-only; prefer a model distinct from the reviewers
launch = [ { command = "open-code --model gpt-5.6", isolation = "none" } ]

[[role]]
name = "implementer"             # writer: MUST resolve to worktree or container
launch = [ { command = "open-code --model gpt-5.6", isolation = "worktree" } ]
```

How the orchestrator invokes it: at preflight it reads `spawn.toml`, probes each entry (is the CLI on PATH; for `isolation = "container"` is agent-box present; for `worktree` does the harness expose worktrees), and states the resolved per-role plan to the human alongside the isolation tier it already states. During the loop, for a role with a resolved entry it spawns by running `command` through the harness Bash tool, in the writer's worktree cwd (`worktree`) or via `agent-box-run.sh` (`container`) or plainly (`none` for read-only), handing the launched CLI the role prompt from `.agents/prompts/` and only that role's context, then collecting its findings / output file exactly as for a native spawn. For parallel reviewers it picks distinct `launch` entries so the model set actually differs.

Composition with the isolation tiers and preflight (the key unification): the map's `isolation` field names the SAME tiers as the Writer isolation rule (`container` > `worktree` > `none`/file-safety). There is ONE tier vocabulary and ONE preflight resolution covering both "which target" and "which isolation". Read-only roles legitimately take a cross-model target at `isolation = "none"` (no blast radius, per the existing rule). A writer role's entry MUST resolve to `worktree` or `container`; a Bash-launched writer does not inherit the native Agent tool's worktree isolation, so the orchestrator supplies it (creates the worktree and sets the launched CLI's cwd, or wraps via the container script). If a writer's mapped target cannot be isolated on this harness, the orchestrator does NOT launch an un-isolated writer; it falls back to a native Claude spawn under the harness tier. This is make-illegal-states-unrepresentable applied to spawning (Principle 5): an un-isolated writer is not a representable outcome of the resolver.

Reject pure (c): it leaves the single most concrete, single most useful contribution unbuilt. The orchestrator is an LLM that can read prose, but the exact launch invocation (project-specific flags, model IDs, container mounts) is precisely what it will guess wrong without a pinned example, and it is also the only thing that makes Q-33's container path reachable. (c) also strands the diversity rule as an aspiration the orchestrator cannot execute. That said, the config must be empty-by-default and degrade to (a)/(c) behaviour when unpopulated, so pure-(c) IS the correct RESOLVED behaviour for a single-Claude user; we simply do not stop there for the multi-CLI user.

## 4. Unified position: one optional module (the isolation module, increment 3, scope grown) plus one core rule

Q-33 and Q-36 are one mechanism, so they get one artifact set, not two modules:

- CORE (ships always, docs-only, harness-agnostic): the diversity rule generalization (Q-36 a), edited into `pack/AGENTS.md` (reviewers line, and the explorer / triager guidance) and mirrored in `pack/prompts/reviewer.md` and `pack/prompts/triager.md`. This belongs in core because it is prose that applies with zero config and assumes nothing about external CLIs (Principle 2: it does not complicate the core, it refines an existing sentence).

- MODULE (opt-in, `--module isolation`): the pointer setup (Q-33 base), the `agent-box-run.sh` wrapper (Q-33 launch-path glue), and `spawn.toml` (Q-36 b). One config file, one preflight resolution, one tier vocabulary, so there is no duplicated logic to drift (Principle 1). This belongs in the module, not core, because it assumes external CLIs / container tooling and drops project files (Principle 2 minimal-by-default, Principle 3 safe-on-existing via create-if-absent).

So the answer to "one module, two, or a rule plus config" is: a RULE (core) PLUS a CONFIG-AND-GLUE MODULE (one module). Not two modules. A separate "spawn" module split from the "isolation" module would fracture one preflight resolution and one tier vocabulary across two opt-ins and force the orchestrator to reconcile them, which is exactly the drift Principle 1 forbids. The naming tension (reviewers' cross-model config lives under a module named "isolation" yet needs no isolation) is real but cosmetic; the binding concern is "how a sub-agent is spawned and contained", and the reviewer case is the `isolation = "none"` corner of that same resolver. If the name grates, rename increment 3's module to `spawn` and let isolation be its writer-side concern; the structure is unchanged. I keep it as one module under increment 3.

## 5. Sequencing vs optional-modules increment 3

- The core rule (Q-36 a) is decoupled and lands NOW, independent of the module roadmap. It is a few-line edit to `AGENTS.md` / `reviewer.md` / `triager.md`, needs no `--module` machinery, and carries no risk. Do it as a small docs change, not gated behind increments 1 and 2.
- The module content (Q-33 pointer + wrapper, Q-36 b spawn map) lands WITH increment 3, which is already sequenced after increment 1 (the `--module` machinery) and increment 2 (the deterministic-checks module). It needs the module machinery to exist and needs the preflight resolver design settled. Increment 3's scope grows from "pointer setup only" to "pointer setup + wrapper script + spawn map", but it stays a CONTENT module (drops files, no tool runtime dependency), consistent with its human-decided content-module framing.
- Evidence gate (Principle 6): before shipping `spawn.toml`, validate with a proof-of-concept that the orchestrator, given the config and a real second CLI on PATH, actually launches a distinct-model reviewer through Bash and collects its findings file; and that with agent-box present a `container` writer runs isolated. If a functional run is not achievable (no second CLI available to test), ship (a) the rule and the pointer setup only, and hold `spawn.toml` behind a validated multi-CLI user rather than forcing an unvalidated config (this is also the steelman below).

## 6. Adversarial failure modes and how each is answered

- Dead config no harness reads. The consumer is the scaffolded orchestrator PROMPT (an LLM this tool programs), not a harness feature. Preflight resolves every entry to available / unavailable; unavailable entries degrade (read-only to native spawn, writer to native+harness-tier). The config is never a precondition; an absent or empty `spawn.toml` yields today's native-spawn behaviour.
- Container integration that works on only one harness (breaks harness-agnosticism). The wrapper is invoked through Bash, which most harnesses expose, and it is a portable POSIX shim plus a pointer doc, not harness-specific plugin code. Container-exec is offered as one TIER that degrades like every other tier; no harness is ASSUMED to support it, and where Bash or agent-box is absent the resolver falls to worktree / file-safety. Harness-agnosticism is preserved because the artifact is portable content and the capability is discovered, not required.
- Cross-model config goes stale as CLIs change flags. `spawn.toml` is a USER-OWNED working file (create-if-absent, Principle 3 / 4), scaffolded as a COMMENTED example the user edits, not a tool-maintained registry of CLI flags. The tool commits to the SCHEMA (`role -> {command, isolation}`), never to any command string; it hardcodes no `--model gpt-5.6` promise. Staleness lives on the user's edit surface, and preflight probes (`--version` / a dry run) catch a broken command and degrade before the loop relies on it.
- The tool drifts from scaffolder to orchestrator-runtime. The Rust CLI still only WRITES files (a config template, a shell shim, prompt guidance) and adds no exec / spawn / network / agent-box dependency at build or run time. The reader-and-runner is the orchestrator agent via the harness's Bash, exactly as it already reads `AGENTS.md` and runs git and formatters. With no module selected the scaffold output is byte-identical (Principle 2). No new runtime surface is introduced.

## 7. Steelman AGAINST my own recommendation

The strongest case is to ship ONLY (a) the core rule and (Q-33) the pointer setup, and DROP `spawn.toml` and the wrapper as premature.

The orchestrator is an LLM reading prose. A well-written rule ("where you can launch other models or CLIs as sub-agents, do so for reviewer / triager / explorer diversity, and run writers under the strongest available isolation") may be sufficient on its own, because the orchestrator can discover installed CLIs itself at preflight (probe PATH for open-code / pi), read those CLIs' `--help`, and construct the launch command from its own up-to-date knowledge. In that world a scaffolded command-string config is redundant AND a liability: it is exactly the artifact my own failure-mode #3 admits will go stale, and for the majority single-Claude-harness user it is never populated, so it adds a concept to learn for zero benefit. My own preflight-probe answer to staleness (the orchestrator validates commands at runtime) partly concedes the point: if the orchestrator is smart enough to validate and degrade, it may be smart enough to construct the command without the config. Under this view the config earns its keep only when BOTH (i) the launch invocation is non-obvious and project-specific (custom mounts, model routing) such that the LLM would get it wrong unaided, AND (ii) the user actually runs multiple CLIs, which is an unproven population today. Principle 6 (ground decisions in evidence) then says: build the rule now, and defer the config until a real multi-CLI user's proof-of-concept shows the LLM cannot construct the invocation reliably, rather than scaffolding speculative config. That is a coherent, principle-backed position, and it is the honest disagreement with my recommendation.

My rebuttal (why I still recommend (a)+(b)): the config is empty-by-default and opt-in, so it costs the single-Claude user nothing (Principle 2), and it is the ONLY vehicle that makes Q-33's container path concrete (an LLM will not reliably guess this project's agent-box mounts). But I hold the disagreement open and gate the config on a functional proof-of-concept per section 5, so if the evidence says the rule alone suffices, we ship (a) + pointer setup and stop.
