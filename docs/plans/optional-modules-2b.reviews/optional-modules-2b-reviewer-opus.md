## Findings

### MEDIUM

**M1. Seeded formatter check-mode command `nix fmt -- --check` is invalid for this project's formatter and would raise a false-positive format finding every review round.** `pack/checks.toml:53` ships `check = "nix fmt -- --check"`. This project's `nix fmt` is treefmt v2.5.0 (see `flake.nix:39-61`, treefmt-nix), which has no `--check` flag: running `nix fmt -- --check` exits 1 with `Error: unknown flag: --check` (verified in the worktree). The checks-reviewer runs the format kind's `check` command and treats a non-zero exit as "the implementer left unformatted files" (`pack/prompts/checks-reviewer.md`, the `kind = "format"` bullet). So on this repo (and any treefmt-v2 project), the seeded format check ALWAYS errors and produces a spurious format finding on every work-review round, regardless of whether the tree is formatted. That undercuts one of the two active behaviours 2b ships (lint-detect + format-check-mode). The apply `command = "nix fmt"` is correct; only the check-mode flag is wrong. The correct treefmt flag is `--fail-on-change` (or `--ci`), i.e. `check = "nix fmt -- --fail-on-change"`. The wrong flag was copied verbatim from both explorations (checks-config-A line 74, checks-config-B line 69), which specified `--check` without validating it against treefmt; the implementer verified ast-grep end-to-end but not the format check-mode against the actual formatter. Not critical because checks.toml is a user-owned working file whose comment tells users to swap in their formatter's check flag, the core stays byte-identical, and the module is off by default; but it is the project's own stated convention shipped broken for the project, so a user who enables the module without editing gets a false gate. Recommend fixing the seed to `--fail-on-change` and, ideally, validating seeded commands against the repo's actual toolchain.

### LOW

**L2. Seeded lint rule is `severity: warning`, so ast-grep exits 0 on a violation; the gate holds only for the output-parsing reviewer, not for any exit-code consumer.** `pack/checks/ast-grep/rules/no-dbg-macro.yml:7`. Verified: with `severity: warning` ast-grep exits 0 when it reports the violation; with `severity: error` it exits 1. The 2b checks-reviewer parses tool OUTPUT ("turn each reported violation into a finding"), so it still catches the dbg! use, so this is not a 2b functional break. But (a) the reviewer prompt instructs reporting "its exit code" as finding evidence, which reads as exit 0 for a genuine finding (odd but harmless), and (b) any exit-code-based consumer, specifically the 2c `agent-scaffold checks` subcommand and the pre-commit hook the plan describes as "failing the commit on any violation", would see a clean pass for this seeded rule. Consider `severity: error` for the seed so the same rule gates on exit code too, or documenting that lint findings surface via output rather than exit code. Flagging for 2c coherence; 2c is out of scope for this review.

**L3. Guidance's example findings-file name does not match the convention it cites.** `pack/checks-guidance.md:9` assigns the checks-reviewer the path `<step>-checks.md` "per the findings-file naming rule", but the rule in `pack/AGENTS.md:63` is `<step>-<role>-<disambiguator>.md` for a reviewer (`<step>-triage.md` for the triager). `<step>-checks.md` omits the disambiguator. Harmless in practice (there is one checks-reviewer, and the orchestrator assigns the exact path; the prompt itself defers to "the findings-file path the orchestrator assigned you"), but the concrete example does not conform to the cited convention. Either present it as `<step>-checks-<disambiguator>.md` or note that the single deterministic reviewer needs no disambiguator.

**L4. Guidance intro still lists ASCII-clean as a generalised convention, but the seed no longer represents it.** `pack/checks-guidance.md:3` says the module generalises "the hardcoded verification convention (clippy / `nix fmt` / ASCII-clean)". After the rule swap the seed contains a `dbg!` rule (not the exploration's ASCII rule) and no regex/grep ASCII check was added; the implementer's own note acknowledges ASCII-clean enforcement would need a separate check. Describing the mechanism's PURPOSE (you plug in any check via a row) is defensible and `nix fmt` is seeded, but a reader enabling the module to get the advertised ASCII-clean generalisation finds nothing representing it. Minor; either drop ASCII-clean from the intro list or add a commented regex/grep ASCII-clean example row so the named convention has a concrete plug-in template.
