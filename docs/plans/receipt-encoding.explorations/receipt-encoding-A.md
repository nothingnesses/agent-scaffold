# Q-43 exploration: how the (chosen) Full decision receipt is ENCODED and where it lives (minimalist / core-blast-radius lens)

Explorer lens: minimalist / core-blast-radius, optimising for Project Principle 2 (minimal by default; a module or feature must not complicate the core) and Principle 3 (safe on existing projects; never change what a populated repo inherits without cause). The one question this document keeps asking of each encoding: which one adds the LEAST to the core (`src/plan.rs`, `src/metrics.rs`, `src/workflow.rs`) and to the plan format EVERY scaffolded project inherits, while still delivering the fixed Full receipt? The answer turns out to be lopsided, and the SSOT worry that reopened this pass is smaller than it looks once you see that it is closed by a CONVENTION, not by machinery.

## 1. The question, restated, and what is fixed / out of scope

Q-43 (`docs/plans/agent-scaffold.md:122`) asks ONLY how the already-chosen Full decision receipt is ENCODED and where it lives, choosing among three contenders:

- (a) a permanent `type:"decision"` JSONL record in `docs/metrics/workflow.jsonl` (matches the existing instrumentation record types; opt-in with `--instrument`; leaves the plan format untouched), cross-referenced by a new `validate --workflow` check W4 that every `decided` Open-Questions item has one.
- (b) a structured block on the `decided` Open-Questions item IN THE PLAN (single home, co-located with the decision), parsed by `src/plan.rs`, checked by W4.
- (c) a structured receipt SOURCE with a single home, plus a PROJECTION/render so `agent-scaffold` renders it into the human-readable plan/status view (single source, derive the view).

Fixed inputs this pass may NOT re-open (from the Q-42 resolution at `docs/plans/agent-scaffold.md:121` and the Q-43 item at `:122`): the Full receipt is chosen (not the null option); the receipt records the options presented + a recommendation + the human's choice, with a check that the choice is a member of the options; a W4 `validate --workflow` check asserts every `decided` item has a receipt (forward-looking; historical Q-1..Q-41 exempt); receipts do NOT live in the ledger (per-task, deleted at close); `type:"escalation"` and `type:"intake"` records stay SEPARATE (one purpose per record). The two prior receipt explorations (`docs/plans/decision-receipts.explorations/receipt-system-A.md` recommending NULL, `receipt-system-B.md` recommending FULL) settled the direction; this pass is downstream of that.

Out of scope (noted, not designed): the broader "rewrite the whole plan as a uniform structured file format" idea the Q-43 item flags. The orchestrator's own caution there (the plan is ~80% reasoning narrative; structuring a prose-heavy document fights the format and is a behaviour change for every scaffolded project) is exactly a Principle 2 / Principle 3 concern and I second it, but the receipt needs only the narrow encoding decision, so I say nothing more about the full rewrite.

A framing note that decides the whole pass: W4 is FIXED, so its cost is a CONSTANT shared by all three encodings, not a differentiator. W4 iterates `plan::parse_questions` (`src/plan.rs:274-309`), selects items whose `status` starts with `QUEUE_FOLD_PREFIX` (`src/plan.rs:87`), and asserts each has a matching receipt, forward-looking (the historical-exemption boundary is the same problem to solve wherever the receipt lives). The ONLY thing that varies across (a)/(b)/(c) is WHERE the receipt SOURCE lives and WHAT parses it. That is the axis this document measures.

## 2. The three encodings, with concrete cost accounting

### Ground truth: what exists today

- `src/plan.rs` deliberately parses only two structured REGIONS: the Roadmap pipe-table (`parse_roadmap`, `:235`) and the Open-Questions queue, from which `parse_questions` (`:274-309`) projects exactly three fields per item, `id` / `status` / `ask`, reading ONE list line per item. The module header states the boundary outright (`src/plan.rs:1-15`): "the free narrative (Step Details, motivations) holds no machine state and is not parsed." The queue vocabulary is a small closed set (`QUEUE_EXACT_STATUSES` at `:82`, `exploring` included; `QUEUE_FOLD_PREFIX` at `:87`).
- `src/metrics.rs` validates the JSONL log against a per-record-type schema: `check_record` (`:276-332`) matches on `type` and has four arms today (`round`, `escalation`, `dismissal_recheck`, `intake`), each reusing small field checkers (`require_str`, `require_enum`, `require_count`, `require_severities`). A separate best-effort projection `parse_rounds` (`:372-415`) reads only the fields the cross-reference needs. A drift-guard test (`:673-741`) asserts every accepted record type, field name, and enum spelling appears in `pack/instrument.md`.
- `src/workflow.rs` holds the cross-references. `check_workflow(plan_markdown, log_contents)` (`:77-86`) already receives BOTH artifacts; W3 (`:150-212`) is the template a W4 check follows.
- A projection/render already exists: `status` (`src/main.rs:404-431`, `run_status` at `:643-695`) parses the plan's two regions into a `Projection`, and renders it BOTH as `--json` and as a short human-readable summary. It reads the JSONL only for a record COUNT (`metrics::count_records`). It never WRITES the plan; the plan is a create-if-absent working file the human edits directly in Markdown.
- The instrumentation log is OPT-IN: `pack/instrument.md` is rendered into the scaffolded `AGENTS.md` only when `--instrument` is set (`src/main.rs:355`, the `{{instrument}}` placeholder), and the `workflow-invariants` step is explicit that the check is "meaningful when instrumentation is on" (`docs/plans/agent-scaffold.md:640`). So the JSONL log and its whole schema exist and matter ONLY for instrumented projects.

### (a) `type:"decision"` JSONL record + W4

`src/plan.rs`: ZERO change. The plan format, the queue grammar, and the no-prose boundary are all untouched. This is the load-bearing fact.

`src/metrics.rs`: one new `"decision"` arm in `check_record`'s match (`:287-330`), validating `q_id` (`require_str`), `recommendation` and `chosen` (`require_str`), and `options` (a new array-of-strings check modelled on `require_severities` at `:206-226`, minus the enum step). One genuinely new bit of logic: the member check that `chosen` is one of `options`, a cross-field constraint the existing per-field checkers do not have; it is a `contains` over the parsed `options`, roughly five lines inside the arm. Then a `parse_decisions` projection parallel to `parse_rounds` (`:372-415`) returning `(q_id, line)` pairs for W4, roughly 20-30 lines. Then two drift-guard additions: append `"decision"` to the record-type list and `q_id` / `options` / `recommendation` / `chosen` to the field list (`:681-711`), and add the new fields to `pack/instrument.md` prose. Total: a self-contained arm plus a small projection, entirely inside the module that already owns JSONL schema, following patterns already in the file. Call it moderate and localized.

`src/workflow.rs` (W4): reuses `plan::parse_questions` (already imported pattern) and the new `metrics::parse_decisions`; for each `Question` whose `status` starts with `QUEUE_FOLD_PREFIX`, require a matching `q_id`. `check_workflow` already has both inputs (`:77-86`), so NO signature change; add W4 to the `problems.extend(...)` chain. Roughly 30-40 lines plus tests, directly mirroring W3. The forward-looking historical exemption is the one non-trivial sub-problem, but it is W4-common (see section 1), not an (a)-specific cost.

Pack plan-template format change: NONE. `pack/plan-template.md` stays byte-identical. Blast radius across scaffolded projects: for a project that does not set `--instrument`, ZERO (no log, no schema, no receipt). For an instrumented project, the plan format is STILL byte-identical; only the JSONL log gains one more record type, exactly like adding `harness` did in `reviewer-harness-field` (`docs/plans/agent-scaffold.md:671-673`), which the plan itself classifies as "LOW-risk (additive, off the scaffold path)". This is the smallest possible blast radius of the three.

### (b) structured block on the `decided` OQ item IN THE PLAN + W4

`src/plan.rs`: the LARGEST core cost, and it lands squarely on the boundary the module was built to hold. `parse_questions` reads ONE line per item; a receipt block (an options list, a recommendation, a choice) is inherently multi-line, so the parser gains a notion it does not have: continuation lines belonging to a queue item. That means new field(s) on `Question` (`:35-44`), new multi-line parsing under each `decided` item, new structural validation in `queue_structure_problems` (`:201-229`), and the same member check (`chosen` in `options`) that (a) puts in `metrics.rs` moved here into the plan validator. The module header comment (`:1-15`) and the queue-item contract in the template both say the item is a one-liner with a pointer; (b) contradicts that. This is 60-100 lines of new parser plus tests, and, worse than the line count, it erodes the "two clean structured regions, prose not parsed" invariant that keeps `src/plan.rs` simple and that Principle 5 (illegal states) leans on.

`src/metrics.rs`: no change (receipt not in the log). `src/workflow.rs` (W4): arguably simpler, since the receipt and the queue item are now the same artifact, so the cross-reference could even collapse into `validate_plan`; but that saving is dwarfed by the parser cost above.

Pack plan-template format change: MANDATORY, and this is the decisive blast radius. The Open-Questions section of `pack/plan-template.md:24` defines the queue-item format ("Each item has a stable id, a one-line ask, a status ..., and a pointer to the step, ledger, or exploration that carries the detail") for EVERY scaffolded project. (b) changes that inherited CONVENTION and the parser that enforces it for ALL projects, whether or not they ever instrument, whether or not they ever record a decision receipt. The drift-guard test (`src/plan.rs:754-784`) that pins the template to the validator would also need extending. So (b) is the one option whose cost every downstream project pays unconditionally, which is exactly what Principle 3 (safe on existing projects; do not change what a populated repo inherits) and Principle 2 (do not complicate the core) exist to resist.

### (c) structured source + projection/render into the human-readable view + W4

(c) has two readings, and both cost MORE than (a), not less.

Source half: the "single home" is either the JSONL (in which case the `src/metrics.rs` cost is identical to (a)) or a NEW structured sidecar file (in which case it is a new artifact class with its own parser, strictly more than (a)). Either way the source-parsing cost is >= (a).

Render half, thin reading: extend the existing `status` `Projection` (`src/main.rs:404-431`) to read the decision records and show them in `--json` and the summary, roughly 20-40 lines reusing machinery that already exists. But this render puts the receipt in the `status` OUTPUT, not in the PLAN. The plan document a human opens still does not contain the receipt; you read it by running `agent-scaffold status`. So the thin reading is literally "(a) plus a status-rendering extension", and it does NOT deliver the "render it into the human-readable plan" goal that motivated (c). It is (a) with extra steps.

Render half, heavy reading (render the receipt INTO the plan Markdown): the tool today NEVER writes the plan; it only parses it (`run_status` reads, `scaffold` only ever create-if-absent-drops assets). Rendering derived content back into a hand-edited Markdown document is a brand-new generative-writer capability, and it is precisely the deferred `workflow-viz` step's territory (`docs/plans/agent-scaffold.md:429-431`): a derived view over an append-only event log, which "reuses the derived-projection schema", is sequenced AFTER `state-schema`, and "needs a design-questions pass before implementing". So the heavy reading of (c) forces a deferred, unbuilt, design-pass-owed capability to come first. That is the opposite of minimal.

Pack plan-template format change: none for the thin reading; for the heavy reading, the tool now co-owns a region of the plan it renders into, which is a bigger idempotency-and-clobber surface than (b) (Principle 3, Principle 4), on a file the human also edits.

The crux (c) gets wrong: (c) is sold as the most SSOT-aligned because "one source, derive the view". But the SSOT hole that reopened this pass is NOT that the human view is un-derived; it is that a JSONL receipt would be a SECOND permanent hand-authored home for the option-labels+choice alongside the QUEUE ITEM's prose (`docs/plans/agent-scaffold.md:121`, the drift A poked). A render into `status` or into the plan does not touch the queue-item prose at all; that prose is still hand-authored and still un-derived. So (c)'s render machinery does not close the hole it is invoked to close. What closes the hole is the CONVENTION that the OQ prose need NOT re-enumerate the options (section 3), and that convention is available to (a) for free.

## 3. Trade-offs judged against the numbered Project Principles

Principle 2 (minimal by default; do not complicate the core). Ranking is clean: (a) < (c) < (b) in core cost. (a) adds one JSONL arm and one projection to the module that already owns the log, and touches `src/plan.rs` not at all. (b) pushes multi-line structured parsing into `src/plan.rs` past the deliberate no-prose boundary (`src/plan.rs:1-15`) and mutates the plan-template convention. (c) is at best (a)-plus-a-render and at worst (a)-plus-a-render-plus-a-plan-writer that pulls in the deferred `workflow-viz`. Principle 2 points at (a).

Principle 3 (safe on existing projects; never change what a populated repo inherits without cause). This is the sharpest split. (a) leaves the scaffolded plan format byte-identical for everyone and confines its entire footprint to the opt-in `--instrument` log, so a project that never instruments inherits exactly nothing new (`src/main.rs:355`, `docs/plans/agent-scaffold.md:640`). (b) changes the inherited queue-item format and parser for EVERY scaffolded project unconditionally, instrumented or not, decision-recording or not (`pack/plan-template.md:24`). (c)-heavy makes the tool write into a file the human hand-edits, a create-if-absent / idempotency hazard (Principle 4). Principle 3 points hard at (a) and hard against (b).

Principle 16 (one source of truth). This is the principle (b) and (c) are argued to serve and (a) is argued to violate, so weigh it honestly. The receipt's machine payload is the option LABELS + the choice. (a) puts that payload in exactly one machine home, the JSONL record. The alleged second home is the queue-item prose that says "chose A over B". But that duplication is not forced by (a); it is dissolved by a one-line convention: the receipt is the machine home for option-labels+choice, so the OQ prose need NOT re-enumerate the options and simply says `decided -> folded into <slug>` plus a pointer, exactly as the template already prescribes (`pack/plan-template.md:24`) and exactly the queue=CONTENT / receipt=ATTESTATION split B proposed and the human accepted (`docs/plans/agent-scaffold.md:121`). With that convention, (a) has ONE machine home for the labels+choice and the queue keeps its existing job (what/why, folded into the step). The residual is that the step's WHY-narrative may still say "chose A over B" as reasoning, but that is content prose, authoritative for content, not a second machine record of the labels; the bounded, soft-resolved overlap the human already signed off on. So the SSOT cost of (a) is small and is paid down by convention, not by parser or render machinery. Meanwhile (b) manufactures its OWN Principle 16 tension: a structured block co-located with the same item whose step-detail prose also narrates the decision, and now BOTH are in the plan, closer together, with the parser forced to adjudicate. (c) does not actually reduce the duplication at all (section 2). Net: Principle 16 does not favour (b) or (c) over (a) once the convention is in place; it is roughly neutral, and (a) wins on 2 and 3 decisively.

Principle 5 (make illegal states unrepresentable) and Principle 1 (cleaner long-term architecture) are minor here but tilt to (a): keeping `src/plan.rs` as two clean parsed regions plus unparsed prose is a simpler, more defensible invariant than a queue grammar that grows a multi-line body, and a `type:"decision"` arm is coherent with the established "one validated record per event type" architecture (`src/metrics.rs:276-332`).

## 4. Recommendation

Recommend (a): the permanent `type:"decision"` JSONL record plus the W4 cross-reference, together with the CONVENTION that a `decided` Open-Questions item does not re-enumerate the options in prose (the receipt is their machine home).

Reasoning, decisive cost argument first. The blast-radius difference is not marginal, it is categorical. (a) leaves `src/plan.rs` and `pack/plan-template.md` byte-identical and confines its whole footprint to the opt-in instrumentation log, so every scaffolded project that does not set `--instrument` inherits ZERO change, and every one that does still inherits an unchanged plan format with one extra JSONL record type (the same low-risk, off-the-scaffold-path shape as the shipped `reviewer-harness-field` addition). (b) changes the queue-item format and parser that EVERY project inherits, unconditionally, and pushes structure into `src/plan.rs` across the exact boundary the module was written to hold; that is the largest core cost and the only unconditional downstream cost of the three. (c) is either (a) with a status-render bolted on that does not actually place the receipt in the plan, or (a) plus a generative plan-writer that drags in the deferred, design-pass-owed `workflow-viz`; in neither reading is it smaller than (a). The SSOT worry that reopened this pass is real but small, and it is closed by the one-line convention above, which (a) can adopt for free and which no amount of render machinery in (c) supplies. So the encoding that adds the LEAST to the core and to the inherited plan format, while delivering the fixed Full receipt and the fixed W4 check, is (a).

W4 and the historical exemption are the same in all three and should be built once against the JSONL source: iterate `plan::parse_questions`, select `QUEUE_FOLD_PREFIX` items, require a matching `q_id` decision record, forward-looking. `check_workflow` already carries both artifacts, so W4 is a strict addition alongside W3 with no signature churn.

## 5. What NOT to build (the YAGNI boundary)

- Do NOT put the receipt in the plan (reject (b)). Do not change `pack/plan-template.md:24`'s inherited queue-item convention, and do not grow `parse_questions` a multi-line item body; that pushes structure across the deliberate no-prose boundary (`src/plan.rs:1-15`) and taxes every scaffolded project, instrumented or not, for a feature most will never use.
- Do NOT build a projection that RENDERS the receipt back into the plan Markdown (reject (c)-heavy). The tool never writes the plan today; a generative plan-writer is the deferred `workflow-viz` capability (`docs/plans/agent-scaffold.md:429-431`), owes its own design pass, and is sequenced after `state-schema`. Do not force it early for a receipt.
- Do NOT add a new structured sidecar file as the receipt source. A new artifact class needs a new parser and is strictly more than reusing the JSONL log that `--instrument` already ships.
- Do NOT build a thin `status`-only render as if it closed the SSOT hole (the weak reading of (c)). If a human-readable surface for receipts is later wanted, extending the existing `status` `Projection` (`src/main.rs:404-431`) is a cheap follow-on, but it is (a) plus sugar, not a distinct encoding, and it does not reduce any duplication; keep it out of the receipt build's scope.
- Do NOT re-enumerate the options in the `decided` item's prose once the receipt exists. That re-enumeration is the ONLY thing that would make (a) a genuine second home for the labels+choice; the convention that the OQ prose points at the receipt instead is what keeps Principle 16 satisfied at zero machinery cost.
- Do NOT add a `q_id` to `type:"escalation"` or `type:"intake"` records to make them double as receipts (fixed input; one purpose per record). A `decided` item that also escalated gets its own `type:"decision"` record.
- Do NOT reach for the full-plan-structured-file-format rewrite (explicitly out of scope). The narrow receipt win does not need it, and it is a Principle 2 / Principle 3 change for every project that should be weighed entirely on its own.
