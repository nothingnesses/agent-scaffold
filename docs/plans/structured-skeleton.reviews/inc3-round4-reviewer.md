### Drift guard ties to `ALL`

`const _: () = assert!(StepStatus::ALL.len() == StepStatus::VARIANTS.len());`

`VARIANTS` is `&["not-started", "in-progress", ...]`, generated by the `enum_field!` macro from its variant list. Because the entire enum is generated by the macro, adding a new variant requires adding it to the `enum_field!` invocation; this automatically grows `VARIANTS` by one. `ALL` is a separate `[StepStatus; 7]` const defined in the `impl` block; it does NOT grow automatically. So after adding a new variant to the macro:

- `VARIANTS.len()` is 8.
- `ALL.len()` is still 7.
- `assert!(7 == 8)` fails at compile time (E0080, constant evaluation panic).

A maintainer cannot satisfy the build by only updating `label()` and the guard match arm; they must also extend `ALL`. The guard comment's claim ("a stale `ALL` that silently dropped the new status from the render engine's status distribution is a build error") is now accurate and the mechanism is strictly parallel to the `WaiverReason` guard. L3b is fully resolved.

The identical argument applies to `QuestionStatus::ALL`.

### Macro extension: backward-compatible with metrics enums

The `enum_field!` pattern change from `$($variant:ident => $text:literal),+` to `$($(#[$vmeta:meta])* $variant:ident => $text:literal),+` is a purely additive optional capture. The existing metrics enum invocations (Phase, RoundOutcome, RiskClass, WaiverUnit, WaiverReason, EvidenceTier, and the four other enums) supply no per-variant attributes; the empty token sequence satisfies `$(#[$vmeta:meta])*` and the generated code is byte-identical to before. The metrics tests (parse, validate, workflow checks) are all in the 286 passing; no metric enum behavior changed.

### `#[allow(dead_code)]` on the generated `parse`

The reasoning is sound and the choice of `#[allow]` over `#[expect]` is correct.

`parse` is actively called for at least five metrics enums (`Phase::parse`, `RoundOutcome::parse`, `RiskClass::parse`, `WaiverUnit::parse`, `WaiverReason::parse`; confirmed by grep in `src/metrics.rs`). For those instantiations the `dead_code` lint does NOT fire; if `#[expect]` were used, the "expectation unfulfilled" error would be a compile failure at those sites.

`parse` is never called for `StepStatus` or `QuestionStatus` (those values arrive typed via serde on the TOML schema; confirmed by searching the full source for `StepStatus::parse` and `QuestionStatus::parse`, both returning no matches). For those instantiations `dead_code` WOULD fire without suppression.

`#[allow(dead_code)]` in the macro body suppresses the lint for all instantiations: harmless where it does not fire (metrics enums), necessary where it does (status enums). The project preference is `#[expect]` except where a lint fires for only some instantiations - this is exactly that case (analogue of the cfg-split exception in CLAUDE.md). `#[allow]` is correct.

### `#[macro_use]` ordering and scope

`enum_field!` is the only `macro_rules!` definition in the codebase. The module order in `src/main.rs` is `checks`, `manifest`, `#[macro_use] metrics`, `pack`, `plan`. The `#[macro_use]` attribute elevates `enum_field!` to the crate root, making it available to all modules. No other module defines a macro named `enum_field!`, so there is no name collision. The macro is used in `src/plan/source.rs` (two `enum_field!` invocations for `StepStatus` and `QuestionStatus`); the crate compiles cleanly with the macro visible there, confirmed by `cargo test` and `cargo clippy`.
