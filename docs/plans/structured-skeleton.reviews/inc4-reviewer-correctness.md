## Findings

### C1 (low): W5's "round log line N" message is a wrong/misleading locator for a TOML waiver

- Where: `src/workflow.rs` `w5_problems` (the three `format!("round log line {}...", waiver.line, ...)` messages) fed by `waivers_from_toml` (`src/workflow.rs`), whose `line` is TOML document-order; `metrics::Waiver` (`src/metrics.rs:812`) has no `id` field.
- Defect: a `metrics::Waiver` projected from a `[[step.waiver]]` carries `line` = its 1-based position in TOML document order, but every W5 message prints it as `round log line {N}`. For a TOML-primary plan that phrase names a JSONL line that does not correspond to the waiver at all (there is no JSONL waiver line post-Q-46), so a human debugging a W5 violation is pointed at an unrelated round record (or a nonexistent line). The waiver's stable `id` (the human-meaningful locator that DOES exist in the TOML, and is enforced unique by `validate_source`) is dropped in `waivers_from_toml` because `metrics::Waiver` has no `id` field, so the message has nothing better to fall back on.
- Why it is not merely cosmetic-on-a-malformed-plan (correcting the prompt's framing): W5 fires on WELL-FORMED-but-mis-evidenced waivers too, not only presence-rule violations. The two Inc 4 negative tests exercise exactly this realistic path: `check_workflow_toml_w5_rejects_a_mis_tiered_waiver` (a correctly-present but mis-tiered waiver) and `check_workflow_toml_w5_rejects_a_wrong_escalation_waiver` (a correctly-tiered record-backed waiver citing an unrelated escalation). Both are legitimate non-malformed plans on which a user would see `round log line <doc-order>` and look in the wrong file.
- Impact bound (why low, not medium): the VERDICT is correct in every case (the pass/fail and the substantive text after the locator are right); only the locator prefix is wrong. And the live repo is still `primary == markdown` until Inc 5, so the TOML W5 path is not yet exercised in anger. Real-world impact today is near zero; it becomes a genuine diagnostic footgun once a plan goes TOML-primary.
- Suggested direction: either (a) make the W5 message substrate-aware (a neutral locator, e.g. `waiver \`<id>\``, which requires adding an `id: Option<String>`or a`locator`enum to`metrics::Waiver`and populating it in both`parse_waivers`and`waivers_from_toml`); or (b) at minimum change the shared wording from "round log line N" to a substrate-neutral phrase (e.g. "waiver #N") so it does not assert a JSONL provenance the TOML waiver does not have. Option (a) is the more useful fix because it restores the stable id the author actually wrote. This is the implementer's own flagged item (2); I agree it is a real defect but rate it low.

## Minor observations (not findings)

- The TOML `[[step.increment]]` id list is effectively decorative for W3: the round -> increment join keys off the ROUND record's own increment id (`round_increment_id`), not the declared TOML increment ids, so a declared increment that no round names is not cross-checked by W3. This is the documented Inc 4 design ("uses whatever join exists"), consistent with the JSONL path (which has no increment list), and `validate_source` separately checks that an increment-unit waiver names a real declared increment. Noted only so the triager knows the declared-vs-observed increment sets are not reconciled by the enforcement path.
- `step_views` deliberately does not project `blocked_by` into a `blocked on <slug>` status, so the `status` command over a TOML-primary source shows a blocked step under its base status (e.g. `in progress`) rather than as blocked. This is an explicit contract choice (W3/`status` never read the `blocked on` string) and does not affect W3/W4/W5 correctness; it is a `status`-projection fidelity difference, out of this lens's core, flagged for the fidelity reviewer if in scope there.
- No dedicated TOML test covers "a `complete` step whose rounds fall SHORT of the streak with NO covering waiver fails" in isolation (the shortfall+waiver=pass case is covered by the two accepted-at-escalation fixtures, and the no-rounds case by the pause test). The shortfall-without-waiver path is the shared `w3_problems` logic exercised by the Markdown suite, so this is a completeness gap in the TOML fixtures rather than an untested behaviour; low value to add.
