## R-1 (medium): the currency scope covers plan AUTHORING but misses stale plan STATUS-MAINTENANCE wording, which directs hand-editing the generated `.md`

Evidence:
- Scope, `docs/plans/agent-scaffold.steps/structured-skeleton.md` Inc 6 bullet, item (1): "`pack/prompts/orchestrator.md` and `pack/prompts/planner.md` (and any other `pack/prompts/*.md` referencing the plan format or `TEMPLATE.md`) move from 'author a Markdown plan from `docs/plans/TEMPLATE.md`' to the TOML+render+sidecar authoring AND editing flow ...". `pack/prompts/implementer.md` is NOT named, and its stale instruction is about status editing, not authoring.
- `pack/prompts/implementer.md` (the "Make small, reviewable changes" paragraph): "Keep the plan's status current as you go: per the plan's Documentation Protocol, set the step's Status cell in the Roadmap table, which is the single source of truth for status, and change the status there only, not in prose elsewhere." Under a TOML-primary project (`[meta].primary = "toml"`, which the Inc 6 template sets), the single source of truth for status is `[[step]].status` in the `.plan.toml`, and the Roadmap table in the generated `<task>.md` is a projection that must NEVER be hand-edited. This instruction, shipped as-is, tells a scaffolded implementer to hand-edit the generated Markdown status table, the exact anti-pattern this currency pass exists to remove; the edit would be clobbered on the next `render` and would fail `render --check`.
- `pack/prompts/orchestrator.md:31` carries the same stale claim: "the Roadmap is the single source of truth for step status".
- `pack/AGENTS.md` "Tracking progress" section (root copy AGENTS.md:57) also asserts the Roadmap status table is "the single source of truth for step status; the implementer keeps it current", but Inc 6 scope item (4) limits the `pack/AGENTS.md` edit to the phase-2 sentence only, so this second stale AGENTS.md claim is outside the enumerated scope.
- The Inc 6 step-level ACCEPTANCE only tests for "no stale `TEMPLATE.md` or Markdown-only-plan authoring references left in `pack/prompts/`". `implementer.md` has no `TEMPLATE.md` reference and its defect is a status-editing reference, not an authoring reference, so the step's own acceptance check would pass while `implementer.md` (and orchestrator.md line 31, and the AGENTS.md Tracking-progress claim) remain stale and the shipped pack remains internally incoherent.

Why it matters: the human's Q-47 request and the chosen all-inside-Inc-6 placement are explicitly about a pack that is "never internally inconsistent" and whose prompts do not still describe the old Markdown-plan flow. The single most operationally dangerous stale instruction, an implementer told to hand-edit the generated Roadmap status table, is not in the concrete enumeration and is not caught by the step-level acceptance. It is partially mitigated: the scope's catch-all hedge "(and any other `pack/prompts/*.md` referencing the plan format ...)" and the new Success Criterion, which does say "a freshly scaffolded project's `pack/prompts/*` ... describe authoring and editing the `.plan.toml` skeleton plus prose sidecars ... never hand-editing it", both reach it at the umbrella-criterion level. But an implementer building Inc 6 works from the step bullet and its acceptance, not the Success Criteria, and "referencing the plan format" is easy to read as authoring-only. Recommend naming `pack/prompts/implementer.md`, the `orchestrator.md` status-SSOT sentence, and the `pack/AGENTS.md` Tracking-progress status-SSOT claim explicitly in the Inc 6 scope (or consciously deferring the AGENTS.md Tracking-progress one alongside Q-48, since it is a canonical-guidance altitude question of the same kind), and adding an acceptance check that the shipped prompts do not instruct hand-editing the generated status table.
