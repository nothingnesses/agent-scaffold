# Reviewer (Opus) findings: waiver-model

Summary: All gates pass (206+1+3 tests, clippy clean, all three `validate` commands green on the live repo) and the migration's 15 waivers are each mechanically valid. The strict-check / best-effort-projection split is sound: every malformed-waiver shape I tried is both reported by `check_record` and dropped by `parse_waivers`, so W3 cannot be fooled by a malformed record, and the "no waiver at all" pause.md catch still fails hard. BUT the W5 integrity check has real holes that let `validate` exit 0 on an illegitimate exemption: the record-backed evidence join is not tied to the waived unit (O1, confirmed exit 0), and an increment waiver's `step` and `increment` are never cross-checked (O3, confirmed exit 0). These are the pilot-1 class of defect (a check that certifies something it should reject). O2 is a design-strength question about self-declared increment waivers; O4/O5 are lower.

---

## O1 — W5 evidence join accepts ANY decision escalation, not one related to the waived unit
- Severity: high
- File: `src/workflow.rs:363-372` (the `escalation.task == evidence && human_decision == Decision` join in `w5_problems`)
- Defect: A `record-backed` waiver passes W5 whenever SOME `type:"escalation"` record anywhere in the log has `human_decision:"decision"` and a `task` string-equal to the waiver's `evidence` pointer. W5 never checks that the cited escalation has anything to do with the step/increment the waiver exempts. So an increment that genuinely fell short of its streak can be exempted by a record-backed `accepted-at-escalation` waiver whose `evidence` names an unrelated escalation's task; W3 exempts on unit+identity, W5 is silent, and `validate --workflow` exits 0. This is exactly the "evidence join satisfied by an unrelated escalation record" path the brief flagged, and it directly falsifies the documented anti-laundering guarantee in `EvidenceTier`'s doc comment (`src/metrics.rs:277-285`: "so a cleanup cannot launder a weak self-declaration into looking as strong as an independent escalation record") and in W5's own doc ("the strong tier is always backed by a real human decision"). It is backed by a real human decision, but for a DIFFERENT unit.
- Confirmed empirically: a `complete` step `alpha` with increment `alpha-incA` at peak-1 (risky, needs 2), exempted by `{"unit":"increment","step":"alpha","increment":"alpha-incA","reason":"accepted-at-escalation","evidence_tier":"record-backed","evidence":"TOTALLY-UNRELATED"}` plus an escalation with `task:"TOTALLY-UNRELATED"` -> `workflow invariants hold`, rc=0.
- Why it matters: this is the same failure mode as pilot 1's derived-min cutoff: a check that is supposed to keep the exemption honest instead rubber-stamps it. The blast radius is bounded (a self-declared waiver would exempt the same increment anyway, see O2), but the validator is certifying a bogus strong-tier claim as valid, which is the specific thing W5 exists to prevent.
- Direction: tie the join to the waived unit. For an increment-unit waiver require `escalation.task == waiver.increment` (which is exactly how the migration's own record chooses its evidence: `evidence == increment == escalation task == optional-modules-inc2cii`); for a step-unit record-backed waiver require `leading_slug(escalation.task) == waiver.step`. Then a foreign-escalation citation is flagged.

## O3 — increment waiver's `step` and `increment` are never cross-checked; W3 ignores `step` for increment exemption
- Severity: medium
- File: `src/workflow.rs:306-309` (W3 increment exemption keys only on `waiver.unit == Increment && waiver.increment == increment`, never `waiver.step == step.slug`) and `src/workflow.rs:353-359` (W5 only checks `waiver.step` is SOME real slug, never that it matches the increment)
- Defect: For an increment-unit waiver, the `step` field is decorative. W3 exempts the shortfall on the `increment` token alone, and W5's only check on `step` is that it names some real Roadmap step. Nothing enforces `leading_slug(waiver.increment) == waiver.step`. So an increment waiver can name a real-but-wrong step and still exempt a different step's increment, passing both checks. The W5 "names a real Roadmap step" check therefore gives false assurance for increment waivers: it says nothing about whether the waiver is scoped to the increment it exempts.
- Confirmed empirically: two `complete` steps `alpha`/`beta`; `beta-incB` short (risky, peak 1); waiver `{"unit":"increment","step":"alpha","increment":"beta-incB","reason":"review-skipped","evidence_tier":"self-declared"}` -> `workflow invariants hold`, rc=0, despite the waiver naming `alpha`.
- Direction: in W3 add `&& waiver.step == step.slug` to the increment-exemption predicate, OR in W5 require `leading_slug(waiver.increment.as_deref().unwrap()) == waiver.step` for increment-unit waivers. The former is the tighter fix (a mis-scoped waiver then simply does not exempt).

## O2 — the record-backed/escalation tier is never REQUIRED to exempt an increment shortfall
- Severity: low-to-medium (likely by design; raising as a decision, not asserting a bug)
- File: `src/workflow.rs:306-309` (W3 increment exemption consults only unit+identity, never reason/tier) with `w5_problems` pairing at `src/workflow.rs:360-372`
- Observation: A bare `self-declared` increment waiver with `reason:"review-skipped"` (or `predates-logging`) exempts ANY short increment, including a fresh `risky` increment that reached only one clean round, with no escalation record at all. W3 does not look at reason/tier, and W5 accepts `review-skipped`/`self-declared` as a valid pairing. So the entire `accepted-at-escalation` / `record-backed` / escalation-join machinery is optional: an author who wants to waive a genuinely-unconverged increment just writes a self-declared one and skips all of it.
- Confirmed empirically: `alpha-incA` risky peak-1 exempted by `{"unit":"increment","step":"alpha","increment":"alpha-incA","reason":"review-skipped","evidence_tier":"self-declared"}` (no escalation) -> rc=0.
- Why I am not calling this a clear defect: the migration itself uses self-declared `predates-logging` increment waivers for `convergence-accounting`, `pack-rebuild-tracking`, and `user-prompts-dir`, so self-declared increment waivers are clearly intended, and self-declaration is the model's accepted trusted-but-visible tier (consistent with the pause.md "declared, not inferred" ethos: every exemption is still an explicit log line). The issue is that the schema doc ("an increment accepted below its required streak at an escalation") and the two-tier framing oversell the enforcement: for increments there is effectively no gate above self-declaration.
- Direction (decide): either (a) accept as-is and soften the doc so it does not imply increment shortfalls must be escalation-backed, or (b) if increment shortfalls are meant to require an escalation, restrict increment-unit waivers to `reason:"accepted-at-escalation"` in `check_record` (and then O1's join fix becomes the real gate).

## O4 — `check_record` accepts an empty `evidence` string on a record-backed waiver
- Severity: low
- File: `src/metrics.rs:518` (`require_str(obj, "evidence")?;` with no non-empty guard)
- Defect: `step` (`src/metrics.rs:475-477`) and `increment` (`src/metrics.rs:507-509`) are both checked non-empty, but `evidence` is not. A record-backed waiver with `evidence:""` passes the strict check and is projected by `parse_waivers` (which also only filters non-empty on `step`/`increment`, not `evidence`). It is not a false-pass on its own (W5's join then fails to find an escalation with `task:""`), but it is an inconsistency with the sibling fields and a latent hole: an escalation with an empty `task` (which `parse_escalations`/`check_record` do not forbid non-empty either) would let an empty-evidence waiver join. 
- Direction: mirror the `step`/`increment` treatment: `if evidence.is_empty() { return Err("field `evidence` is empty") }` in `check_record`, and add the `.filter(|s| !s.is_empty())` in `parse_waivers`.

## O5 — no check that a waived step is actually `complete`, and no duplicate-waiver handling
- Severity: low / informational
- File: `src/workflow.rs:326-374` (`w5_problems`)
- Observation: A waiver naming a real step that is `skipped`, `deferred`, `in progress`, etc. passes W5 and is inert in W3 (which only inspects `complete` steps). Multiple waivers covering the same unit are accepted with no dedup. Neither is a correctness defect (an inert or duplicate waiver grants no extra exemption), so I am not asking for a change; noting it because a mis-targeted waiver (author waived the wrong-status step) is silently accepted rather than reported, which slightly weakens the "keep waivers honest" intent.
- Direction: optional. Could report a waiver whose step is real but not `complete` as a likely mistake; low value.

## Things I checked that are NOT defects
- Strict-check vs projection split: sound. Every malformed shape (bad `unit`/`reason`/`evidence_tier`, empty `step`, `increment` present on a step-unit / absent on an increment-unit, `evidence` present on self-declared / absent on record-backed) is reported by `check_record` AND dropped by `parse_waivers`; the presence rules are duplicated in both (`src/metrics.rs:349-376` strict, `src/metrics.rs:453-467` projection), so a malformed waiver can never reach W3. Verified by the new tests and by reasoning through each match arm.
- pause.md catch intact: a `complete` step with no records and no covering step-waiver still fails (`src/workflow.rs:260-268`); an increment-unit waiver does NOT rescue a no-records step (only the `matching.is_empty()` step-waiver branch exempts that case), and a waiver for a different step does not cover it. Confirmed by tests and code.
- Risk-class inconsistency is never suppressed: the inconsistency `continue`s before the peak/waiver check (`src/workflow.rs:282-288`), so no waiver reaches it. Test `a_risk_class_inconsistency_is_not_suppressed_by_a_waiver` pins it.
- Enum matching is exact/case-sensitive (`enum_field!` `parse`, `src/metrics.rs:46-52`): `"Step"`, `" step"`, etc. return `None`, so a case/whitespace-mangled value is reported and dropped (fail-safe), never silently accepted.
- Migration is mechanically valid: all 3 self-declared increment waivers (`convergence-accounting`, `pack-rebuild-tracking`, `user-prompts-dir`) target increments that DO have a (short, peak-0) round record, so they hit the increment branch, not the no-records branch; all 11 step waivers target steps with zero round records; `optional-modules`'s other five increments each converge on their own (inc1/2a/2b peak-1 low_risk, inc2ci peak-2 risky) and only `inc2cii` (peak-1 risky) needs its record-backed waiver, whose evidence joins to the one real `decision` escalation. `validate --workflow` on the live repo exits 0.
- No dangling `trivial`/`grandfathered` references: `ROADMAP_STATUSES` retired both, the plan flipped all 15 rows to `complete`/`skipped`, TEMPLATE/plan-template updated, and the `roadmap_status_ok` test now asserts rejection. Remaining `trivial` hits are the unrelated `Classification::Trivial` intake enum and prose about "trivial confirmations", not statuses.
