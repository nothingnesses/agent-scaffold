# workflow-invariants (Q-27): trivial-completion declaration and the grandfather boundary (Explorer B)

Independent design notes for two coupled W3 sub-decisions. Scope: how a review-skipped ("trivial") completion is DECLARED so `validate --workflow` can verify it (Decision A), and how steps completed before disciplined per-round logging are exempted without opening a hole for a genuine future skip (Decision B). Recommendation is one answer per decision. All findings below are re-derived independently from `docs/metrics/workflow.jsonl`, the Roadmap in `docs/plans/agent-scaffold.md`, `src/plan.rs`, and `src/metrics.rs`.

W3 recap: for every Roadmap step marked `complete`, filter `round` records whose `task` leading-slug (before an optional `-inc<n>`) equals the step slug; if any exist their `risk_class` must be consistent and the per-artifact consecutive-clean streak must reach the class's required count (`low_risk` 1, `risky` 2). A `complete` step with NO matching records is the `pause.md` catch (marked complete without review) UNLESS it is authorized as trivial (A) or predates disciplined logging (B).

## Decision A: how a trivial completion is declared

### Options

- A1. New `trivial` Roadmap step status in `src/plan.rs` (`ROADMAP_STATUSES`), documented in `pack/plan-template.md` so the existing `plan_template_documents_every_accepted_status` drift guard extends automatically. This is exactly how `exploring` was added to `QUEUE_EXACT_STATUSES` by `exploration-mode`.
- A2. A dedicated JSONL record type, e.g. `{"type":"trivial_completion","task":...}`.
- A3. Overload a `round` record: write a synthetic round for a review that did not happen (e.g. a `clean` record at the required streak).
- A4. A side declarations file, e.g. `docs/plans/trivial.toml`, listing trivially-completed slugs.
- A5. Another plan-side marker: a status suffix like `complete (trivial)`, or a sentinel line in the Step Detail body prose.

### Evaluation

Single source of truth (Principle 16). The Roadmap pipe-table is already declared the single source of truth for step status (`src/plan.rs` module docs: "The plan is the single source of truth for status (the Roadmap pipe-table)"). A trivial completion IS a completion mode, i.e. a status. A1 keeps it in the one authoritative place. A2/A4 create a SECOND writer that must agree with the plan (a step is `complete` in the plan AND declared trivial elsewhere): drift risk, and the exact duplication `round-log-core` (Q-34) just spent an increment removing. A3 fabricates data in the round log, which is defined as the record of rounds that actually happened; it also pollutes convergence accounting and calibration (a fake `clean` inflates the clean-round statistics the log exists to measure).

Fit with existing schemas. A1 is one `&str` in a `const` plus one template line; the drift-guard machinery already exists. A2 is explicitly a `round-log-core` YAGNI non-goal ("a new log record type"), so it reverses a just-made decision. A3 breaks the `RoundOutcome` semantics W3 itself relies on (a `clean` increments the streak, a `new_valid` resets it). Note a naming collision either way: `src/metrics.rs` already has `Classification::Trivial` for the INTAKE judgement ("small interrupt, no full replan"), which is a different axis from "completed without the review loop." `metrics.rs` already documents `RiskClass` as "kept distinct from the intake `Classification`"; A1 needs the same one-line disambiguation note. A2 makes the collision worse by putting a second `trivial` in the same JSONL file the intake `trivial` lives in.

Human authorization workflow. Marking a step `trivial` in the Roadmap is the same deliberate, single-line, git-reviewable act a human already performs to set `complete` or `skipped` (the plan already carries `skipped` for `include-all-visible` and `ledger-parse`). It is the most visible surface in the repo for step state. A log line (A2/A3) is low-visibility and, under `--instrument`, machine-appended, so a declaration hides in the noise. A side file (A4) is a place a reviewer must remember to open.

Ease of checking. W3 iterates `complete` steps. Under A1, `trivial` is simply a DIFFERENT status, so W3 never selects it: no records required, no special-case code, membership test only. The plan's own framing ("a `complete` step with no matching records ... UNLESS it is a declared trivial completion") becomes cleaner: a trivial step is not a `complete` step with a caveat, it is not `complete` at all. A2/A4 force the checker to load and join a second source (more code, more failure modes).

Adversarial. Hardest to fake: A1. To fake a trivial declaration an agent must EDIT the authoritative status cell to `trivial`, which is a public, self-advertising downgrade from `complete`; that is precisely Q-27's stated resolution ("trivial changes are ... forced to be DECLARED trivial ... which turns a silent skip into a visible one"). Easiest to fake / worst: A3, a fabricated `clean` round actively simulates a review that never happened (the opposite of visibility) and is a low-visibility append; A2 is nearly as easy. Easiest to get WRONG: A3 (violates the plan's explicit "do not fabricate records" rule and corrupts metrics) and A5 (a free-text suffix or prose sentinel is un-parseable without ad-hoc string matching and invites near-miss spellings the status enum is designed to reject).

### Recommendation A: A1, a `trivial` Roadmap status in `src/plan.rs`

Add `trivial` to `ROADMAP_STATUSES`, document it in `pack/plan-template.md`, and add a one-line note (mirroring the existing `RiskClass`-vs-`Classification` note) that the plan-status `trivial` ("completed without the review loop, human-authorized") is distinct from the intake-record `Classification::Trivial`. Semantics for W3: `trivial` is its OWN status; W3 selects only `complete`, so `trivial` steps are exempt from the streak requirement by construction, not by a special case. Convention (documented, not machine-checked): the step's Step Detail records why the completion was trivial.

Principles: Principle 16 (status stays single-sourced in the Roadmap table, no second writer); Principle 5 (make illegal states unrepresentable: "completed-but-review-skipped" is a first-class enum member and a checkable state, not an out-of-band annotation the checker must infer); Principle 2 (minimal: one `const` entry plus a doc line, reusing the existing drift guard, no new file, no new record type). Trade-off accepted: the `trivial` plan-status and the `trivial` intake-classification share a word; mitigated by the disambiguation note and by their living in different namespaces (plan status vs metrics record).

## Decision B: the grandfather boundary for pre-logging complete steps

### What the data forces (see appendix for derivation)

Two facts about the real data drive this decision:

1. b2 steps are INTERLEAVED with disciplined steps in Roadmap order, so there is NO clean positional cutoff. `user-prompts-dir` (b2, never converges) sits at Roadmap row 18, AFTER `triager-independence` (15), `file-safety-rules` (16), and `agent-isolation` (17), which all converge. Logging discipline was adopted retroactively (the `round-log-core` increment-A backfill) and unevenly, not at a single line in the Roadmap. Any "enforce steps after row N" rule either still flags `user-prompts-dir` or exempts legitimately-enforceable steps.
2. Neither "exempt when no records" nor "exempt when records-exist-but-no-streak" is safe as an OPEN-ENDED rule. "Exempt when no records" leaves a permanent hole: a genuine FUTURE skip that produces no records looks identical to b1 and slips through (the exact failure mode the brief names). "Exempt when records-exist-but-no-streak" is worse: a future orchestrator that runs one review round (`new_valid`, `consecutive_clean:0`) and marks the step complete produces records that never reach the streak, so it would be auto-exempted. Both formulations make the illegal state (silent future skip) representable and undetectable.

Conclusion: the grandfather set must be a CLOSED, EXPLICIT set of slugs, not an open-ended rule keyed on record-presence. Because W3 is being introduced now, the honest closed set is "the steps already `complete` before W3 existed," which is bounded and, by convention, never grows: any NEW step must either converge or be declared `trivial` (Decision A).

### Options

- B1. Positional / named cutoff (enforce steps after "the first fully-logged task"). REJECTED by fact 1: no such clean line exists.
- B2. "Flag only if records exist AND the streak is not reached", i.e. exempt whenever there are no records or no convergence. REJECTED by fact 2: reopens the `pause.md` hole for future skips.
- B3. Per-step marker (`logged_from`, or a disposition field) set on each grandfathered step. Workable but invites per-step abuse: an open per-step field is an escape hatch an orchestrator can set on a NEW step.
- B4. Explicit closed exemption list of slugs, stored once, with a "do not grow" convention. RECOMMENDED.

Where the exemption data can live: (i) in the plan as an enumerated list (single-source with status; reuses the `src/plan.rs` section/`first_backtick` helpers, no new parser file); (ii) as a `const` in the validator (the established `ROADMAP_STATUSES` pattern, but splits step-state knowledge out of the plan); (iii) a side file (a third store, worst for Principle 16).

### Recommendation B: B4, an explicit closed exemption list stored in the plan

Enumerate the grandfathered slugs once, as a short parseable list in the plan (a `## Grandfathered steps (pre-logging)` list of backticked slugs, one reason each, read with the existing `section` + `first_backtick` helpers in `src/plan.rs`). W3 rule becomes total: a `complete` step is flagged unless it has a converged streak OR its slug is in the grandfather list; a `trivial` step is a different status and exempt by construction. The list is declared a ONE-TIME closed set with a "do not add entries; a new step must converge or be declared `trivial`" note.

Why the plan and not the code const: `src/plan.rs` already declares the plan the single source of truth for step STATUS VALUES, while the status VOCABULARY (`ROADMAP_STATUSES`) is the code const. "Which specific steps are grandfathered" is per-step state (a value), so by that same split it belongs in the plan; the grandfather RULE (the "unless in the list" logic) is the code. This keeps the human-facing reason next to the status it modifies.

Correctness. Exempts b1 by NAME (not by record-absence), so a future no-record skip whose slug is NOT listed is still flagged: the hole from fact 2 is closed. Exempts b2 by name. Covers the `round-log-core` slug mismatch (appendix) as a one-off list entry regardless of how the `-inc` regex is finally settled. Catches a genuine future skip because new steps are absent from the frozen list.

Principles: Principle 5 (make illegal states unrepresentable: "reviewed-but-pre-logging" is an explicit enumerated category, so the illegal "silent future skip" is no longer representable as "looks like b1"); Principle 16 (exemptions live in the plan beside status, one authoritative place, not inferred from the log); Principle 2 (minimal and, crucially, CLOSED, so the exemption does not accrete over time). Trade-off accepted: an enumerated historical list is a snapshot of the past baked into the check; justified because grandfathering is inherently one-time and the set is small and bounded.

## Data appendix (independently derived)

Method: parsed every `type:"round"` record from `docs/metrics/workflow.jsonl`, grouped by `task`, took the max `consecutive_clean` and the set of `risk_class` values per task; cross-referenced each Roadmap `complete` step against those tasks under the leading-slug rule.

Tasks that NEVER reach `consecutive_clean` >= 1 (max streak 0): `consolidate-plan`, `convergence-accounting`, `pack-rebuild-tracking`, `plan-maintenance`, `user-prompts-dir`, `workflow-hardening`. This is exactly the six b2 tasks the brief names. Confirmed.

But only THREE of the six are actually `complete` Roadmap steps: `convergence-accounting`, `pack-rebuild-tracking`, `user-prompts-dir`. The other three (`workflow-hardening`, `plan-maintenance`, `consolidate-plan`) have NO Roadmap row at all (verified: `grep` for a table row returns 0). They are ORPHAN log tasks. W3 iterates `complete` STEPS and filters records into them, so orphan tasks are never selected and are not a W3 concern in either direction; the brief's "six b2 tasks" conflates log tasks with steps. `metrics-fields` is likewise an orphan task (no Roadmap row; maxCC 1). `optional-modules-inc1` maps under `-inc` stripping to `optional-modules`, which is `in progress`, not `complete`, so it is not checked yet.

(b1) `complete` steps with ZERO matching records (11): `core-assets`, `file-dropper`, `idempotency-safety`, `selection-ui`, `mode-enum`, `tag-selection`, `available-filter`, `pack-manifest`, `external-packs`, `pack-owned-principles`, `init-vcs`. These are the earliest steps; they predate logging.

(b2) `complete` steps WITH records that never converge (3): `convergence-accounting` (1 record, `new_valid`, cc0, low_risk), `pack-rebuild-tracking` (1 record, cc0, low_risk), `user-prompts-dir` (1 record, cc0, low_risk). A naive "task in log -> require streak" flags these; the grandfather set must include them.

Slug-vs-step mismatches under the leading-slug rule:
- `state-schema` (complete) is logged as `state-schema-inc1/inc2/inc3`. If the `-inc<n>` strip treats `<n>` as digits, all three collapse to `state-schema`, all `low_risk` (consistent), and the collapsed set reaches `consecutive_clean:1` (inc3 and inc2 both do). PASS. No grandfathering needed for `state-schema`.
- `round-log-core` (complete) is logged as `round-log-core-incA` and `round-log-core-incB`. The suffixes are ALPHA, not numeric. If the strip is digit-only (`-inc\d+`), neither collapses to `round-log-core`, so the step matches ZERO records and is FALSE-FLAGGED as an unreviewed completion despite being fully reviewed (incA converged to cc1, incB to cc2). If the strip is broadened to alnum, the two increments DO collapse but carry DIFFERENT `risk_class` (incA `low_risk`, incB `risky`), so W3's "risk_class must be consistent" sub-rule FALSE-FLAGS it instead. Either way `round-log-core` trips W3 today. This is the single most important data finding: it means `round-log-core` needs a grandfather entry (recommendation B covers it) AND signals a W3-core defect to settle when building (see interactions).

## A <-> B interactions

1. A and B are the SAME move applied to two eras: make the exempt category EXPLICIT rather than inferred. A covers authorized FUTURE skips (a `trivial` status); B covers reviewed PAST completions that predate disciplined logging (a closed list). Together they make W3 total: a `complete` step must have a converged streak OR be grandfathered; a `trivial` step is a different status, exempt by construction; everything else is flagged. No step lands in an inferred-exempt gap.

2. They must stay SEPARATE mechanisms. The plan is explicit: grandfathered steps must NOT be mislabeled `trivial`, because they WERE reviewed (informally / pre-logging), just not logged, and a `trivial` label would both lie about history and blunt the trivial signal. So do not reuse `trivial` for b1/b2. This is why B is a list and not "just mark them trivial."

3. `round-log-core` couples B to W3-core. Listing `round-log-core` in the grandfather set is the minimal safe move now. But the underlying cause (alpha `-inc` suffixes plus per-increment risk_class differences) will recur for any future multi-increment step whose increments carry different risk classes, so it is worth fixing at the source when building W3: broaden the leading-slug strip to `-inc[A-Za-z0-9]+` AND scope the "risk_class consistent" and "streak reached" checks PER INCREMENT (each `-inc` group converges on its own terms) rather than across the collapsed step. If that fix lands, `round-log-core` no longer needs a grandfather entry; if it does not, it does. Flagging, not deciding, since it is W3-core and outside these two sub-decisions.

## Strongest argument AGAINST my own recommendations (steelman the opposition)

Against A. The round log is meant to be the single structured store of the workflow's per-step OUTCOMES (Q-34's whole thrust). A completion, even a review-skipped one, is an outcome; putting it in the plan status while rounds, escalations, intake, and dismissals all live in the JSONL SPLITS the outcome record across two stores. A reviewer or `state-queries` querying the log for "what happened to step X" will not see a trivial completion, because it is only in the plan. A dedicated `trivial_completion` log record (A2) keeps ALL outcomes in one queryable store and lets calibration see review-skips; the `round-log-core` YAGNI note rejected a new record type for ROUND data specifically, and a completion-disposition record is arguably a different, justified case. Deeper: `complete` and `trivial` are TWO orthogonal axes forced into one status string, doneness (is the step finished?) and review-path (was it reviewed, skipped, or grandfathered?). Downstream tooling that greps `complete` to count finished steps will silently miss `trivial` (and `skipped`) steps. By the make-illegal-states-unrepresentable logic I invoke FOR A1, conflating two orthogonal axes into one field is itself a modeling smell; the cleaner model is `status = complete` PLUS a separate `review_disposition in {reviewed, trivial, grandfathered}` marker. That single per-step disposition field would ALSO absorb Decision B (grandfathered is just another disposition value), unifying A and B into one mechanism instead of my split of status-enum-entry plus separate-list. My rebuttal: the plan explicitly WANTS trivial to be a visible status-level downgrade ("turn a silent skip into a visible one"), and an open per-step disposition field is exactly the escape hatch B's closed list is designed to prevent, since an orchestrator could set `disposition = grandfathered` on a brand-new step; the list's single "do not grow" chokepoint is safer than a per-step field. But the two-orthogonal-axes objection is real and the synthesizer should weigh a `complete` + `review_disposition` model against A1.

Against B. An enumerated exemption list is a maintenance smell that bakes a snapshot of history into the checker; the principled alternative is a self-maintaining CUTOFF rule. My "no clean positional line" finding only rules out a Roadmap-POSITION cutoff, not a DATE cutoff: the b2 steps all predate the logging-discipline date even though they are interleaved in Roadmap ORDER, so per-step completion TIMESTAMPS plus a single dated boundary would separate pre/post-discipline cleanly with zero enumerated names, and the exemption would be a rule ("completed before D") not a frozen list. The `round` schema already has an optional `ts` field, so the machinery is half-present. The steelman: add completion timestamps and enforce a date cutoff, which scales to any future one-time boundary without a hand-maintained list. My rebuttal: timestamps are not currently on per-step completions (only on the most recent round records), so a date cutoff is not computable from existing data today; a date cutoff STILL has a future-skip hole unless it is also anchored to explicit names (a future skip dated after D but producing no records still needs catching by name-or-streak, not by date); and grandfathering is inherently one-time, so the closed list is the minimal correct thing now and never has to grow. The synthesizer should still weigh whether the project wants per-step completion timestamps for OTHER reasons (they would also help `state-queries`), in which case a dated cutoff becomes attractive.
