## Deterministic checks

This project scaffolded the checks module (`--module checks`), a user-configurable, tool-run verification gate over the workflow's code output. It generalises the hardcoded verification convention (clippy / `nix fmt` / ASCII-clean) into a declared list of commands, and it adds three things to the workflow without editing any core role prompt.

`.agents/checks.toml` is the user-owned config: a flat list of `[[check]]` entries, each a command to run and its `kind` (`lint`, `format`, `test`, or `mutation`). It is a create-if-absent working file you edit; the module seeds it with an ast-grep linter and a formatter, and documents the field meanings in its own comments (that file is the single source for the schema, not this section). A project plugs in any other linter or formatter by adding a row.

Formatters auto-apply during the implementer's verify step. Before recording its diff, the implementer runs each `kind = "format"` check's `command` over the files it changed, honouring the existing "format only the files you changed" rule (see File safety and durability). This is a writer action that generalises today's hardcoded `nix fmt`.

The checks-reviewer runs in the work-review phase. In phase 4 the orchestrator spawns the checks-reviewer (`.agents/prompts/checks-reviewer.md`) as one more reviewer alongside the LLM reviewers, handing it the same diff range and a distinct findings-file path (`<step>-checks.md`) per the findings-file naming rule. It runs the linters (which detect) and each formatter's check-command, and writes tool-evidenced findings in the same schema the LLM reviewers use; it reports findings but never fixes them. A format check-command should be a non-mutating dry-run where the formatter supports one (for example `rustfmt --check`, `prettier --check`), so the reviewer confirms the code is formatted without changing it and catches an implementer that did not format. Some formatters have no dry-run: treefmt v2 (via `nix fmt --fail-on-change`) formats in place and then reports what changed, so that check mutates the tree and must be run where the changes are safe to discard (the isolation mechanism for that is a separate concern this module does not settle).

The triager adjudicates those deterministic findings alongside the LLM reviewers'. A checks finding is not automatically valid: the triager judges each on its evidence and severity on the one scale, the same reviewer, triager, then implementer pipeline as every other review round, and the implementer fixes the valid ones (a lint finding by changing the code, a format finding by running the apply command). This reuses the existing findings-and-triage pipeline wholesale, keeping both linting and formatting deterministically verified gates rather than trusting the LLM to have run them.

An optional pre-commit hook is a secondary, mechanical backstop. The module drops an inert hook script at `.agents/hooks/pre-commit`; it is namespaced under `.agents/` and is not installed into `.git/hooks/`, so it changes no git behaviour until you activate it, and it never clobbers an existing hook. The authoritative gate is the checks-reviewer above; the hook is a convenience on a developer machine that already has the tool installed. It delegates to `agent-scaffold checks --staged` (rather than inlining commands), so it runs the same lint and format-check commands from `checks.toml` over the staged tree and never drifts from that one source. If `agent-scaffold` is not on `PATH`, the hook prints a note and skips (exit 0) rather than blocking the commit, since the review-integrated path is the real gate; install the tool to enable the hook. It does not run `test` or `mutation` checks (too slow for a commit) unless you add them deliberately.

Two ways to activate it, in ascending convenience:

- Symlink it yourself, from the repository root (check for an existing `.git/hooks/pre-commit` first, so you do not replace one you rely on): `ln -s ../../.agents/hooks/pre-commit .git/hooks/pre-commit`. A symlink tracks later edits to the tracked hook.
- Scaffold with `--with-precommit-hook` (meaningful only alongside `--module checks`): `agent-scaffold scaffold --module checks --with-precommit-hook`. It installs a small `.git/hooks/pre-commit` that delegates to the tracked `.agents/hooks/pre-commit`, create-if-absent: it writes the hook only when none exists, and prints the manual instructions instead of clobbering an existing one.
