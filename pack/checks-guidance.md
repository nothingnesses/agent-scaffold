## Deterministic checks

This project scaffolded the checks module (`--module checks`), a user-configurable, tool-run verification gate over the workflow's code output. It generalises the hardcoded verification convention (clippy / `nix fmt` / ASCII-clean) into a declared list of commands, and it adds three things to the workflow without editing any core role prompt.

`.agents/checks.toml` is the user-owned config: a flat list of `[[check]]` entries, each a command to run and its `kind` (`lint`, `format`, `test`, or `mutation`). It is a create-if-absent working file you edit; the module seeds it with an ast-grep linter and a formatter, and documents the field meanings in its own comments (that file is the single source for the schema, not this section). A project plugs in any other linter or formatter by adding a row.

Formatters auto-apply during the implementer's verify step. Before recording its diff, the implementer runs each `kind = "format"` check's `command` over the files it changed, honouring the existing "format only the files you changed" rule (see File safety and durability). This is a writer action that generalises today's hardcoded `nix fmt`.

The checks-reviewer runs in the work-review phase. In phase 4 the orchestrator spawns the checks-reviewer (`.agents/prompts/checks-reviewer.md`) as one more reviewer alongside the LLM reviewers, handing it the same diff range and a distinct findings-file path (`<step>-checks.md`) per the findings-file naming rule. It is read-only, so it needs no isolation: it runs the linters (which detect) and the formatters in check / dry-run mode (verifying the code is formatted without applying, so it catches an implementer that did not format), and writes tool-evidenced findings in the same schema the LLM reviewers use.

The triager adjudicates those deterministic findings alongside the LLM reviewers'. A checks finding is not automatically valid: the triager judges each on its evidence and severity on the one scale, the same reviewer, triager, then implementer pipeline as every other review round, and the implementer fixes the valid ones (a lint finding by changing the code, a format finding by running the apply command). This reuses the existing findings-and-triage pipeline wholesale, keeping both linting and formatting deterministically verified gates rather than trusting the LLM to have run them.
